为什么要三次握手？ ^SvByFwXT

第一次握手（客户端 ->服务器）：客户端发送 SYN包。发送的包里设置Seq=x，SYN =1 表示同步，发送完之后，会处于Syn_Send 的状态

第二次握手（服务器 ->客户端）：服务器回复 SYN+ACK包。 发送的包里设置Seq=y，ACK= x+ 1, 表示已经收到直到Seq = x之前的包，SYN =1 表示同步，发送完之后，会处于Syn_Received 的状态

第三次握手（客户端 -> 服务器）：客户端发送ACK = y + 1， seq = x + 1 包。表示已经建立连接。变成Established的状态，服务端收到这个包之后也会变成Established状态。 ^GLUuEsTY

提问：什么是 SYN 洪泛攻击？怎么防御？ ^vBWbClHh

回答：攻击者伪造大量不存在的IP地址，向服务器发送 SYN 请求。服务器回复 SYN+ACK 后，这些伪造IP 的主机不会回应，服务器只能等超时。这些半连接会占用服务器的SYN 队列资源，队列满了正常用户就连不上了。防御手段有几种：一是调大 SYN 队列和缩短超时时间，治标不治本；二是SYN Cookie，服务器不维护半连接状态，把状态信息编码到 SYN+ACK的序列号里，收到第三次握手时再验证；三是用防火墙或代理先过滤，只有完成三次握手的连接才放进来。 ^Zewtzuh9

提问：初始序列号为什么要随机？ ^BBhJpQgq

回答：两个原因。第一是防止历史连接的包干扰新连接，假设上一个连接刚关闭，网络里还有它的残留包在飘，新连接如果用同样的四元组和相近的序列号，残留包可能被误认为是新连接的数据。随机ISN 能大幅降低这种碰撞概率。第二是安全考虑，如果ISN是固定的或者可预测的，攻击者可以伪造TCP包进行会话劫持或者TCP 重置攻击。随机ISN 让攻击者很难猜到正确的序列号。现代系统一般用时间戳加上 Hash 来生成ISN，既保证不重复又难以预测。 ^agVoBnoW

提问：FIN_WAIT_2 状态会一直等下去吗？如果服务端一直不发FIN 怎么办？ ^2z0GCUKM

回答：不会无限等。Linux 下 FIN_WAIT_2状态有个超时时间，默认是60秒，由tcp_fin_timeout 参数控制。超时后连接会被强制关闭。这种情况一般是服务端程序有 bug，比如没有正确调用 close（）。 ^Bjy4q0DW

提问：CLOSE_WAIT 状态积压很多是什么原因？ ^mnZ3lSDT

回答：典型的服务端 bug。服务端收到了客户端的 FIN 并回了 ACK，但是应用层代码没有调用 close（）关闭 socket，连接就一直卡在 CLOSE_WAIT。常见原因是代码里 socket 忘记关闭、异常处理没写好、或者死锁导致程序卡住。用 netstat 看到大量CLOSE_WAIT 就该查代码了。 ^9Ts2sxVD

提问：主动关闭方一定是客户端吗？ ^GKHvZcGz

回答：不一定。TCP 连接是对等的，谁先调用close（）谁就是主动关闭方。比如 HTTP/1.0 默认是服务端主动关闭，HTTP/1.1开启 keep-alive 后通常是客户端主动关闭。服务端检测到客户端超时、或者主动踢掉空闲连接时，也是服务端主动关闭。 ^ggbLzVI2

可靠传输 ^CuxrtVB9

TCP 靠一整套机制在不可靠的IP 网络上实现可靠传输，核心思路就是：给数据编号、收到要确认、丢了就重传、传太快要控制。 ^M2Y6asGt

1. 序列号和确认号，TCP 给每个字节都编了号，接收方收到数据后回复 ACK 告诉发送方“我收到了第 N个字节之前的所有数据”。发送方就知道哪些数据送达了、哪些可能丢了
2. 超时重传：发送方发完数据会启动一个计时器，如果超时还没收到 ACK，就认为数据丢了，直接重发。超时时间不是写死的，会根据网络往返时延动态调整。
3. 滑动窗口做流量控制：接收方在 ACK 里告诉发送方“我的缓冲区还能装多少字节”，发送方不能发超过这个量的数据，避免把接收方撑爆
4. 拥塞控制：发送方维护一个拥塞窗口，通过慢启动、拥塞避免、快速重传、快速恢复这套算法，探测网络承载能力。
5. 校验和：每个 TCP 报文都带校验和，接收方收到后重新算一遍，对不上就直接丢掉，等发送方超时重传。
6. 连接管理：三次握手确保双方都准备好了再传数据，四次挥手确保数据传完了再断开，TIME_WAIT 状态确保网络里残留的旧日报文不会干扰新连接。 ^DY958uWw

滑动窗口（流量控制窗口，通常指接收窗 rwnd），接收方在 ACK 里通告 rwnd（还能收多少字节），发送方未确认数据量 ≤ rwnd，别把接收方撑爆 ^RczKMt1a

 拥塞窗口（cwnd），发送方自己维护 cwnd，根据丢包/RTT/ACK 反馈来动态调大/调小，别把网络撑爆（链路/路由器队列有限 ^ztc3vL21

发送窗口 = min(rwnd, cwnd) ^3qldarcS

第一次超时重传后如果还是没收到 ACK，RTO会翻倍，这叫指数退避，避免网络已经拥堵了还疯狂重传火上浇油。 ^b5Cjwiwf

网络抖动大的时候 RTO 会自动变长，避免频繁误判丢包。网络稳定的时候 RTO 会缩短，丢包能更快发现。 ^nGPBXIvK

快速重传: 等超时再重传太慢了，网络好的时候 RTT 可能就几十毫秒，RTO 却要设几百毫秒。
快速重传机制是这样的：如果发送方连续收到 3个重复的 ACK，比如连续收到3个 ACK=1001，说明接收方确实收到了1001之后的某些包。
但就是没收到 1001，那大概率是1001 丢了，不用等超时直接重传。字节传1001的包 ^MPwDzVan

SACK 选择性确认：传统的累计确认有个问题：接收方只能告诉发送方"1000之前的我都收到了”，但没法说"1001 丢了但1002-2000 我都收到了"。这导致发送方不知道该重传哪些，可能把已经收到的也重传一遍。

SACK 扩展解决了这个问题，接收方可以在 ACK 里带上”我收到了1002-1500、1600-2000”这样的信息，发送方就知道只需要重传 1001 和 1501-1599。 ^KVsVdj7Y

TCP 拥塞控制的核心目的就是别把网络撑爆，发送方要根据网络状况动态调整发送速率。主要分四个阶段： ^qILhZY2b

1）慢启动：连接刚建立时，发送方不知道网络能承受多少，所以从一个 MSS 开始试探。每收到一个 ACK，cwnd 增加1个MSS（cwnd++），因为一个 RTT 内收到的 ACK 数量约等于当前 cwnd，所以效果是每个 RTT cwnd 翻倍，指数级增长，直到碰到慢启动阈值 ssthresh或者丟包。 ^zaPiJCII

2） 拥塞避免：cwnd 到了 ssthresh 之后就不能再这么激进了，改成每个 RTT 只增加一个 MSS，线性增长，小心翼翼地探测网络上限。 ^zfHWYOFa

3）快速重传：发送方连着收到3个重复 ACK，说明某个包大概率丢了，不用傻等超时，直接重传那个包。 ^3FEff92W

4）快速恢复：快速重传之后不需要从头再来，把cwnd 减半、ssthresh设为新的cwnd，然后继续线性增长，快速恢复到丢包前的传输速率。 ^gmR1Pgkx
