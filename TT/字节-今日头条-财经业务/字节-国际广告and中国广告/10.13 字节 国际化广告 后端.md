## 2. 讲一下CPU负载和CPU利用率

- **CPU 利用率**：CPU 实际在忙什么（时间占比）
	在一段时间里，CPU 有多少比例的时间在“干活”。
	常见在 top、mpstat、监控面板里看到的 %us/%sy/%id/...。
	**怎么看：**
	- **user(us)**：跑你应用代码的时间
	- **system(sy)**：跑内核代码的时间（系统调用、网络、文件系统）
	- **iowait(wa)**：CPU 在等 IO（磁盘/网络 IO），CPU 其实没算数但也没法推进
	- **idle(id)**：空闲
	- **CPU 负载（load）**：在 Linux 里，Load Average 统计的是平均有多少个任务处于：
		- **可运行（running / runnable）**：要 CPU、在排队或正在跑
		- **不可中断睡眠（uninterruptible sleep，D 状态）**：通常是卡在 IO（比如磁盘、NFS）

##  介绍一下物理内存和虚拟内存，物理内存是如何映射成虚拟内存的

[![](https://pic.code-nav.cn/mianshiya/question_picture/1783388929455529986/W5Nr877G_xuni_mianshiya.webp)](https://pic.code-nav.cn/mianshiya/question_picture/1783388929455529986/W5Nr877G_xuni_mianshiya.webp)
==物理内存（Physical Memory / RAM）是真实存在的硬件内存条空间，CPU 最终读写数据都要落到这些物理内存页上；虚拟内存（Virtual Memory）则是操作系统为**每个进程**提供的一套“看起来连续、独立、很大”的地址空间，程序里看到的指针地址（如 0x7fff…）都是虚拟地址。他们之间会存在映射==虚拟内存的作用主要有三点：**隔离与安全**（进程之间地址互不干扰，越界最多把自己搞崩，不会随便改到别的进程/内核）、**更高的内存利用率**（按需把用到的页加载进 RAM，没用的页可以回收或共享，如共享库只加载一份）、以及**容量扩展**（内存不够时把不活跃页换出到磁盘 swap，进程仍以为自己“有那么多内存”，但会变慢）。

在分页机制下，应用程序虚拟地址空间中的任意虚拟页可以被映射到物理内存中的任意物理页上，因此可以实现物理内存资源的离散分配。分页机制按照固定页大小分配物理内存，使得物理内存资源易于管理，可有效避免分段机制中外部内存碎片的问题。


## 映射过程
映射过程是：CPU 发出虚拟地址后，由 **MMU**（内存管理单元）按“页”为单位（通常 4KB）用该进程的**页表**把虚拟页号翻译成物理页号并加上页内偏移得到物理地址。MMU 把虚拟地址拆成两部分: 1. ﻿﻿高位是虚拟页. 2. ﻿﻿低位是页内偏移。然后拿虚拟页号去查页表，找到对应的物理页框号，拼上页内偏移就得到物理地址了。


![](https://oss.javaguide.cn/github/javaguide/cs-basics/operating-system/paging-virtual-address-composition.png)

为了加速翻译，会先查 **TLB**（页表缓存）。TLB 命中的话一两个时钟周期就出结果，不用再去内存查页表。现代CPU 的TLB 命中率能到99%以上，地址翻译的开销基本可以忽略。如果 TLB没命中，MMU 去内存查页表。查到的页表项会被装进 TLB，下次再访问同一个页就快了。整个过程对程序透明，程序以为自己直接访问了内存。


## 缺页
页表项里有个有效位，标记这个页当前是否在物理内存里。如果程序访问一个有效位为0的页，就触发缺页异常。

操作系统接管后先判断这次访问是否合法。

如果访问的地址压根不在进程的有效地址范围内，直接发 SIGSEGV信号干掉进程。如果地址合法但页不在内存，操作系统找一个空闲物理页框，把数据从磁盘读进来，更新页表项，然后让进程重新执行那条指令。

如果物理内存满了，操作系统得先踢掉一个页才能加载新页。Linux用的是改进版的LRU 算法，通过活跃链表和非活跃链表来跟踪页面的使用情况，优先换出长时间没访问过的页。。





## 多级页表
主要是为了省内存。

单级列表
以32位的环境为例，虚拟地址空间范围共有2^32（4G）。假设 一个页的大小是2^12（4KB），那页表项共有4G/4K=2^20个(100万个)。每个页表项为一个地址，占用4字节，（2^20\*2^2）\/（1024 \*1024）=4MB。也就是说一个程序啥都不干，页表大小就得占用4M。


为了解决这个问题，操作系统引入了 多级页表，多级页表对应多个页表，每个页表与前一个页表相关联。

32位系统一般为二级页表，64位系统一般为四级页表。

这里以二级页表为例进行介绍：二级列表分为一级页表和二级页表。一级页表共有1024个页表项，一级页表又关联二级页表，二级页表同样共有1024个页表项。二级页表中的一级页表项是一对多的关系，二级页表按需加载（只会用到很少一部分二级页表），进而节省空间占用。

假设只需要2个二级页表，那两级页表的内存占用情况为：4KB（一级页表占用）+4KB\*2（二级页表占用）=12KB。

# 什么时候会进行虚拟内存和物理内存之间的映射，什么时候解除映射。

虚拟内存到物理内存的**真正映射**通常发生在“页级别”上：进程启动时操作系统会先建立虚拟地址空间的布局，但很多页并不立刻占用物理内存；当程序**第一次访问某个虚拟页**（读/写）时触发缺页异常，内核分配/装入一个物理页并更新页表完成映射；fork 场景下还会通过 **写时复制（COW）** 在首次写入时为写入者分配新物理页并建立新映射。

总的来说，在程序启动，或者显式的分配内存(mmap，malloc)的时候，不会立刻分配内存构建虚拟内存和物理内存的映射。会在程序用到该页的时候，才会开始分配，完整映射。总得来说，按需构建。

映射的**解除**则在两类情况出现：一是显式释放，比如 munmap、free，进程退出，或内存管理器把空闲页归还给内核时会把页表项清掉；二是内存压力下的回收/换出，内核可能把不活跃页写到 swap 或丢弃文件页缓存，使页表变为“不在内存/不驻留”，物理页被回收，下次再访问会再次缺页并重新建立映射。


## malloc和mmap的区别是？

malloc 用来分配一个用户

内存分配器会根据你用 malloc 申请的大小，自己在内部决定是否使用 mmap。在 Solaris 上，如果申请超过 128KB



 **malloc** ：是用户态内存分配器的接口，它从进程的堆里切一块给你，底层可能用，brk/sbrk 扩展传统堆（heap），或 mmap 申请匿名内存（尤其是大块分配）。你得到的是一块“普通内存”，不会自动和文件挂钩
 
 **mmap**：  内核提供的“建立映射”的系统调用。直接向内核要一段 **虚拟地址区间**，并指定它映射到哪里。它能控制权限（R/W/X）、共享方式（MAP_SHARED/MAP_PRIVATE）、是否固定地址等。
/
- malloc(n)：用户态库函数，拿到一块可用堆内存，最后 free(p)
- mmap(...)：系统调用，按页建立映射（匿名或文件），最后 munmap(addr, len)
```c
void *p = malloc(nbytes);
if (!p) { /* 分配失败 */ }



int fd = open("data.bin", O_RDONLY);
if (fd < 0) { /* 打开失败 */ }

off_t offset = 0;          // 必须页对齐
size_t len = 4096 * 100;   // 需要映射的字节数

void *addr = mmap(
    NULL,
    len,
    PROT_READ,             // 只读
    MAP_PRIVATE,           // 私有映射（改动不写回文件）
    fd,
    offset
);

if (addr == MAP_FAILED) { /* 失败 */ }

close(fd); // mmap 成功后可以关 fd，映射仍然有效（常见做法）
```

#  Redis的大key

**定义**：大Key是指 Value体积过大（如String类型 > 2MB，Hash/List元素>5000个）的Key

**影响**：

1. 对大Key的操作 导致主线程的操作阻塞和资源消耗过高。
2. 若使用RDB持久化，可能内存峰值升高，导致OOM

**如何发现 Big Key：**

redis-cli--bigkeys 是 Redis 自带的扫描命令，能找出每种数据类型中最大的Key：


**解决方案**：
1. **数据拆分**：
- **Hash/List拆分**：将大Key拆分为多个小Key，例如将一个Hash Key拆分成多个Hash Key
开发层面：把大对象拆成小对象，一个 Hash 100万个 field 可以按ID 取模拆成100个 Hash；对存储的数据做压缩，比如用 Snappy、LZ4压一下再存进去；选合适的数据结构，有些用 String 存JSON 的场景换成 Hash 能省不少内存。

2. **异步删除**：
- **命令替换**：使用`UNLINK`（非阻塞）替代`DEL`，后台异步线程删除，避免影响
直接 DEL 一个大 Key 是很危险的操作。删一个 100万元素的 Hash，Redis 要一个个释放内存，可能阻塞好几秒，上服务直接就挂了。

Redis 4.0之后推荐用UNLINK 命令，它只在主线程里把 Key 从键空间摘掉，实际的内存释放交给后台线程异步处理，主线程几乎不阻塞。

如果是 4.0之前的版本，只能分批删：Hash用 HSCAN+ HDEL, Set 用SSCAN + SREM,List 用 LTRIM，每次删百个元素，分多次删完。
