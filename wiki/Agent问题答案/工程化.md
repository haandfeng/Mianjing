## Q2 — 了解过市面上有哪些 Agent

> 原题：了解过市面上有哪些智能体 agent 吗

**答题骨架**

- **通用助手**：腾讯元宝、Kimi、豆包、文心一言、Coze（扣子）—— 内部远不止 RAG，还有长短期记忆、大文档处理、token 超限处理、异常兜底等工程能力（蔚来面经原话）。

- **垂直 Agent**：客服 Agent、医疗 RAG Agent、CICD / DevOps Agent（蔚来在做的方向）。
- **趋势**：从 single-agent + tool calling → 多 Agent 协作（A2A）+ 长任务（Task / Artifact 状态机）+ 沙箱 Computer Use。

## Q5 — 介绍一些 AI 大模型

> 原题：介绍一些 AI 大模型

**答题骨架**

- **国内**：DeepSeek 671B（开源、中文强、可自部署，西门子项目首选）、千问 Qwen3（embedding 4096 维霸榜）、文心、Kimi、豆包、智谱 GLM。
- **国外**：GPT-4 / 4o / o1（OpenAI）、Claude 3.5 / 3.7 / 4 / 4.5 / 4.7（Anthropic）、Gemini 1.5/2.5（Google）、LLaMA 3 / 4（Meta）。
- **选型考量**：参数量、中英文表现、是否开源 / 可自部署、tool use 准确率、上下文窗口、成本（API vs 自部署）。

- **结构化输出**：DeepSeek 官方 / 硅基流动稳定，腾讯云 structured output 不稳定（蔚来真实踩坑）。

## Q24 — 项目中 AI 贡献的代码占比

> 原题：项目中 AI 贡献的代码占比

**答题骨架**

- **使用方式**：Copilot / Claude Code 主要做补全 + 局部重构；不是把整体 agent 模式开起来让它一把改完。
- **方法论三步法**（[[微软/Bing图像视频业务实习一面面经]]）：
  1. 让 AI 先做 plan，不直接改代码；
  2. 人工 review 方案、明确改动边界 / 风险点 / 涉及模块再授权执行；
  3. 改完每一行关键代码都要能解释。
- **代码占比**：业务样板代码 / 单测 / 文档 60-70%；架构 / 关键并发 / 性能瓶颈代码 < 20%。
- **底层不能完全交给 AI**（ARM 多线程 bug 案例）：复杂并发逻辑需要先自己读懂，再让 AI 辅助分析；自己不读懂代码，AI 修不好。

## Q28 — 长文本生成的技术方案

> 原题：长文本生成的技术方案

**答题骨架**

- **大窗口直生**：用 200K / 1M context 的模型（Claude / Gemini）+ 流式输出；适合一次性长文本，但成本贵。
- **分段生成 + Outline**：先让模型出大纲，再逐节展开，每节生成时只喂相关上下文 → 避免注意力分散。
- **Map-Reduce**：长输入分块 → 每块单独总结（map）→ 合并总结再生成（reduce）；适合长报告 / 论文综述。
- **Iterative Refine**：先写草稿，再让模型 / 评审 critic 给修改建议，循环优化。
- **流式 + 写入**：边生成边落盘 / 推前端，避免等全部生成完才返回；SSE / WebSocket。
- **token 预算**：留 ≥10% 给输出，避免 stop_reason=length 截断。
- // 部分推测：Outline / Map-Reduce 来自社区共识。

**拓展双链**：[[专栏/Agent/MCP_SSE_文档]]（流式输出）、[[Agent问题答案/Prompt工程#Q9]]

## Q29 — Agent skills

> 原题：Agent skills

**答题骨架**

- **结构**（[[专栏/Agent/Skills]]）：
  ```
  skills/<name>/
  ├── SKILL.md      # 入口说明书 + YAML frontmatter（name / description / license）
  ├── scripts/      # 可复用 Python / Shell 脚本
  ├── reference/    # 示例 / 模板 / 格式规范
  └── assets/       # 字体 / 图片 / Logo
  ```
- **触发**：模型读 description 决定要不要"装上"这个 skill；不触发就不进 prompt，省 token。
- **执行**：装上后按 SKILL.md 步骤执行，必要时调 scripts / 引用 assets。
- **vs MCP**：Skill 是本地能力包（带 prompt + 代码 + 资源）；MCP 是协议（拉外部能力）。
- **优势**：把"做某类任务的 SOP"代码化、可复用，比纯 prompt 更可控。
- **常见 skill 例子**：pptx 生成、xlsx 处理、pdf 抽取、obsidian-markdown 创建。

**拓展双链**：[[专栏/Agent/Skills]]、[[Agent问题答案/MCP与协议#Q37]]

## Q32 — 模型预热机制

> 原题：模型预热机制

**答题骨架**

- **目的**：冷启动时第一次推理特别慢（模型加载 / cuda graph 编译 / KV cache 分配），生产环境用户首条请求会感受到。
- **方案**：
  1. **启动期 warm-up**：服务启动后立即跑几条无害 prompt（"hello"），让 vLLM / TensorRT-LLM 完成 cuda graph capture 和权重加载。
  2. **健康检查打 warm**：k8s readiness probe 跑实际推理，未通过不流量。
  3. **常驻热请求**：低 QPS 时定时发 keep-alive，避免被框架自动卸载。
- **embedding 模型也要预热**：第一次 batch 嵌入比稳态慢一倍。
- **缓存预加载**：高频文档 / 高频 query 启动时预先 embed 写入 Redis / 本地。
- // 待补全：暂无直接 raw 来源（推测，但 vLLM / TensorRT 文档明确支持）。

## Q33 — NL2SQL 场景下的 SQL 安全防护

> 原题：NL2SQL 场景下的 SQL 安全防护

**答题骨架**

- **执行账户**：给 LLM 用的数据库账户**只读 + 行级 / 表级权限**，绝不允许 DELETE / UPDATE / DDL；防止 prompt injection 删库。
- **SQL 解析白名单**：用 sqlglot / pgsql-parser 解析模型输出，只放行 SELECT；遇到 INSERT / UPDATE / DELETE / DROP / GRANT / EXEC 直接拒绝。
- **资源限制**：`statement_timeout`、`max rows`（LIMIT 强制注入）、临时表大小、连接超时。
- **schema 注入 + 列级 ACL**：prompt 里只暴露当前用户能看到的表 / 列；敏感列（薪资 / 身份证）做掩码。
- **数据脱敏**：返回值过敏感词 / PII 检测，命中就替换成 `***`。
- **审计**：所有 LLM 生成的 SQL + 结果记录可追溯；接 SIEM 告警异常模式。
- // 待补全：暂无 raw 来源（推测，但与 [[蔚来/蔚来CICD组AI agent实习生一面面经-详细版]] 慢 SQL 优化部分技术栈相通）。

## Q34 — 复杂任务执行准确率提升的评估方法

> 原题：复杂任务执行准确率提升的评估方法

**答题骨架**

- **拆解评估**：把复杂任务分解为多个 step，每 step 单独打分（如检索 → 推理 → 生成），找出瓶颈环节。
- **三维度**：Correctness（结果对不对）、Process（中间步骤对不对）、Cost（耗时 / token / 调用次数）。
- **LLM Judge**：用更强模型当 judge，配合 rubric（评分标准）；用 F1 ≈ 0.8 与人工标注校准（西门子项目做法）。
- **AB 测试 / 回归集**：每次模型 / prompt / 工具改动跑回归集，对比 baseline；防止局部优化全局退化。
- **失败 case 反查**：trace 链路（LangSmith）定位失败环节，归类后批量修复。
- **黄金集 + 长尾**：黄金集保证基线不退化，长尾 case 集每周补充新真实失败样本。

**拓展双链**：[[Agent问题答案/RAG#Q20]]、[[西门子RAG]]（Q1 / Q2 评测体系）

## Q35 — AI 辅助 IDE 开发工具

> 原题：AI 辅助 IDE 开发工具

**答题骨架**

- **GitHub Copilot**：补全 / 聊天 / agent 模式；插件生态最广。
- **Cursor**：原生 AI IDE，整库索引 + 多文件编辑 + agent 模式 + Composer。
- **Claude Code / Codex CLI**：CLI 形态，强调 plan + execute + review，适合 SOP 化任务。
- **Cody (Sourcegraph)**：代码搜索 + agent，企业自部署。
- **Continue**：开源 Copilot 替代，支持自部署本地模型。
- **JetBrains AI Assistant**：JetBrains 全家桶集成。
- **使用建议**（[[微软/Bing图像视频业务实习一面面经]]）：
  - 不放权给 agent 一把改完，先 plan 再 execute。
  - 复杂场景（内核 / 多线程）AI 容易出冗余代码，要人工把关。
  - Ask 模式 / 补全模式适合大多数日常开发。

## Q43 — AI 辅助开发的实践经验

> 原题：AI 辅助开发的实践经验

**答题骨架**

- **三步法**（[[微软/Bing图像视频业务实习一面面经]]）：
  1. 让 AI 先出 plan / 设计，不直接改代码；
  2. 人工 review 改动边界 / 风险点 / 涉及模块，再授权执行；
  3. 改完每一行关键代码都要能解释，不留"看不懂的 AI 代码"。
- **场景适配**：
  - 适合 AI：单元测试、样板代码、文档、CRUD、重构 rename、配置脚本。
  - 慎用 AI：底层并发、内核驱动、数据库迁移逻辑、安全相关。
- **避坑**：
  - 上下文不全的 agent 模式经常生成"看似合理但实际错位"的代码 → 复杂场景必须先把项目读懂。
  - AI 容易引入冗余抽象 / 过度设计，code review 时要主动简化（参考本仓库 CLAUDE.md 中的"代码风格"约束）。
- **效果度量**：留意提交粒度、回归测试、代码评审通过率，而不是行数 KPI。
