## 考察重点（按出现次数排序）
1. **Redis 与 MySQL 缓存一致性** —— 价格设错的业务场景下"先改 DB 再改 Redis"才符合业务（保证下单优先），不能先删缓存阻塞下单
2. **直播抢券分片问题** —— Redis 扛不住时分片，但要解决某分片为零导致用户抢不到（一致性哈希 + 定时任务检测）
3. **Redis 热 Key 处理** —— 集群架构下复制热 Key 到多个分片（foo → foo2/foo3/foo4）+ 读写分离 + 多级缓存（Caffeine + Redis）
4. **缓存三大问题方案** —— 雪崩 TTL 加随机数 / 互斥锁 / 后台更新；击穿 互斥锁 / 永不过期 + 异步更新
5. **SQL 子查询** —— GROUP BY + HAVING；ORDER BY + LIMIT；子查询取 MAX
6. **登录闪烁问题（多系统不一致）** —— ThreadLocal 用户上下文 + Redis token；多服务部署需要主从同步

## 真题摘录（值得背的）

> **直播间用户抢券，价格设错了商户希望后台更改，怎么保证 Redis/MySQL 一致性？** 出处 [[小鹅通/我的/小鹅通二面面经（结果未知）Cider一面面经（通过）]]
> 不能"先删缓存再改 DB"——这样用户下单全部阻塞等修改，不符合业务场景。正确做法：先改 DB 再改 Redis，价格短暂错误道歉就好，保证用户能下单最重要。商户发现到修改 DB 的时间窗口本来就很短，问题不大。

> **优惠券都放 Redis 扣减，Redis 扛不住怎么办？分片后某片为零怎么办？** 出处 [[小鹅通/历史/3]]
> 分片到不同 Redis 集群；某片为零导致用户抢不到则用一致性哈希分配优惠券（避免单分片库存耗尽）+ 定时任务检测库存均衡。
