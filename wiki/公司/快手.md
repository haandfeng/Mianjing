- 方向：分两条线 —— 大数据平台（Flink 向量化加速、Rust/C++）和支付（Java 后端）

## 考察重点（按出现次数排序）
1. **MySQL 三大日志 + 两阶段提交** —— 支付线两场都问 binlog/redolog/undolog 写入顺序
2. **MVCC 与 RR 幻读** —— 支付一面、二面、挂的那场都追问可重复读还会不会幻读
3. **缓存三大问题** —— 击穿/穿透/雪崩在支付二面被打包问
4. **JVM 内存与 GC** —— 支付一面深挖 Eden/Survivor/老年代、复制 vs 标记清除
5. **秒杀方案** —— Lua 脚本 + Redis 预扣 + MQ 异步落库，几乎每场都问
6. **限流熔断** —— 滑动窗口、令牌桶、漏桶；Sentinel
7. **ConcurrentHashMap / 锁体系** —— synchronized / ReentrantLock / volatile / CAS
8. **TCP 可靠传输 + HTTPS 混合加密** —— 滑动窗口、拥塞控制、CA 证书
9. **大数据线 AI 辅助性能审查** —— C++ 值传递 vs 引用、move/copy、shared_ptr 开销

## 真题摘录（值得背的）

> **数据迁移如何保证一致性？异常如何回滚？** 出处 [[快手/大数据平台方向/快手一面面经]]
> 三阶段：迁移前快照 + 幂等 upsert；迁移中批次内事务 + checkpoint 同事务持久化；迁移后 count + hash 聚合校验。完全回滚优先用快照恢复，比覆盖重刷可靠。

> **可重复读还会不会幻读？举例。** 出处 [[快手/支付方向/快手日常实习（支付方向）面经]]
> 会。两类场景：①事务 A 快照读 → 事务 B 插入提交 → A 更新这条记录后再快照读会看到（trx_id 变成自己的）；②先快照读、再当前读（select for update）会看到 B 新插入的行。要彻底避免就在事务一开始就 select for update，让 next-key lock 兜底。

> **如何判断 AI 生成的 C++ 代码性能足够好？** 出处 [[快手/大数据平台方向/快手一面面经]]
> 两手：代码审查看常见反模式（值传递大对象、循环内 new、该 move 用 copy、该 unique_ptr 用 shared_ptr），加 Google Benchmark 实测吞吐和延迟。
