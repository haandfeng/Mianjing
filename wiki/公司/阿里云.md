# 阿里云

## 概览
- 方向：消息商业化团队（RocketMQ 开源维护 + 云上消息服务，Java）
- 难度：高 —— 整场围绕 RAG ETL 改造为 MQ 多阶段流水线、十亿级数据迁移五阶段方案、支付分布式一致性
- 一句话画像：你的西门子 RAG 和用友迁移会被极致追问到"如果换成钉钉知识库 / 电信积分场景该怎么改"，需要把"双写 + 灰度切流 + 影子读 + CDC"这一套讲到位

## 考察重点（按出现次数排序）
1. **大规模数据迁移方案设计** —— 五阶段：分块全量 + 双写 + 影子读 + 灰度切流 + 可观测性；监控比代码更重要
2. **RabbitMQ 不丢不漏全链路** —— 生产者 Confirm + mandatory；Broker Exchange/Queue/消息三层 durable；消费者手动 ACK；本地消息表兜底；死信队列
3. **CDC 增量同步 + 同用户有序** —— Kafka 用 userId 做 partition key；RabbitMQ 用 Consistent Hash Exchange 等效
4. **支付分布式一致性四层组合** —— 本地消息表（核心）+ 独立对账系统（兜底）+ TCC（多方）+ 最大努力通知（银行回调）
5. **RAG ETL 流水线扩容** —— 用 MQ 把"解析 / embedding / 落库"三阶段解耦成 competing consumers，每段独立扩容；embedding worker 攒 batch + 超时 fallback
6. **积分对账业务维度** —— 流水表净变化对比、按主键取模抽样、唯一流水 ID 幂等

## 真题摘录（值得背的）

> **如果要求十亿级数据不停机迁移怎么解？** 出处 [[阿里云/阿里云消息团队一面面经]]
> 五阶段：①全量搬迁（分块 + 禁用索引外键 + 并行 worker + checksum）；②双写（先旧后新，旧库 WAL 兜底，failed 事件进 Kafka 重试，幂等 ID）；③影子读（线上暗测新库，捕获时区/编码/排序差异）；④灰度切读（预热缓冲池，1%→100%，保持双写一周可回滚）；⑤可观测性（同步延迟 / 影子读不匹配 / 死锁 / 缓存命中率 / 业务 KPI）。"迁移本质是伪装成数据问题的监控问题。"

> **扣库存成功但发消息失败，或消费失败怎么处理？** 出处 [[阿里云/阿里云消息团队一面面经]]
> 标准答案是本地消息表：扣库存和写消息表（状态"待发送"）在同一本地事务里，再异步发到 MQ。发成功更新为"已发送"，发失败定时任务扫"待发送"重发，即使 MQ 整个挂了订单也不丢。消费多次失败 → 死信队列 → 兜底消费者写入失败记录表 + 告警，区分可重试 / 不可重试避免浪费重试次数。

> **支付怎么保证我们系统和银行的数据一致？** 出处 [[阿里云/阿里云消息团队一面面经]]
> 实时靠"本地消息表 + 唯一流水号幂等重试"，回调靠"最大努力通知"，多方资源协调用 TCC（Try 冻结 → Confirm 全部成功一起确认 → Cancel 任何失败回滚），兜底靠"每天独立对账系统"扫银行账单做差异表。原则：宁可多查一次也不能漏一笔。
