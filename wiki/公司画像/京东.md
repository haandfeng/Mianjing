# 京东

## 概览
- 方向：京东零售 后端
- 难度：偏开放性场景题（拆单库存、订单超时取消、银行对账）+ 操作系统/IO 模型基础
- 一句话画像：八股密度中等，重头是开放设计题——不要套模板，要权衡业务目标（最少拆单 / 时效 / 体验）

## 考察重点（按出现次数排序）
1. **IO 模型 + epoll** —— 五种模型阻塞/同步分类；select 两次遍历两次拷贝、poll 链表突破限制、epoll 事件驱动；Redis 单线程 + epoll
2. **支付分布式一致性** —— 本地消息表 + 对账（兜底）+ TCC + 最大努力通知；和阿里云题目一致
3. **京东业务场景题** —— 多仓拆单库存占用、订单 5/15 分钟超时取消（延迟队列 / Redis TTL / 定时任务）
4. **Java 应用并发吞吐量优化** —— 四层方法论：压测定位瓶颈 → 线程池隔离反压 → 减少共享和锁竞争（LongAdder/CHM）→ IO 资源复用降 GC
5. **Java 程序运行流程** —— javac 字节码 → JVM 类加载 → 解释 + JIT → GC 管理 → 退出
6. **缓存击穿防护**

## 真题摘录（值得背的）

> **Java 应用怎么系统提升并发吞吐量？** 出处 [[京东/京东零售一面]]
> 四层：①先用压测 + profiler 定位瓶颈（CPU / 锁竞争 / IO 等待 / GC）；②线程模型上做隔离反压（业务线程池分组 + 队列有界 + 超时熔断限流）；③并发设计减少共享和锁竞争（分段、无共享、LongAdder、CHM、避免阻塞 event loop）；④IO/资源层面复用降 RTT（连接池、批量、缓存、降低对象分配率）。最后用 QPS 和 P99 验证。

> **支付环节涉及对账/对接银行，如何保证一致性？** 出处 [[京东/京东零售一面]]
> 四层组合：本地消息表（实时主链路）→ 独立对账（每天扫银行账单兜底，三种差异：我们有银行没有 / 银行有我们没有 / 金额不一致）→ TCC（多方 Try-Confirm-Cancel 避免半成功）→ 最大努力通知（银行主动回调）。"实时靠本地消息表 + 幂等重试，兜底靠独立对账，宁可多查一次也不能漏一笔。"
