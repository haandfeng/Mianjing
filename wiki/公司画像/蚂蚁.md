## 考察重点（按出现次数排序）
1. **多源数据聚合分页** —— 三个交易系统订单合并、按时间排序、每页 10 条；不能只用全局时间戳做游标（漏单），要为每个系统记录 (lastTime, lastId) 复合游标
2. **RPC 服务发现 + Etcd** —— key 设计、watch 监听、ConcurrentHashMap 缓存
3. **一致性哈希负载均衡** —— 虚拟环 + 虚拟节点
4. **秒杀超卖 + 优惠券回滚** —— Lua 脚本判断 count>0 + Redis Set 去重；下单失败发回滚消息 INCR 库存 + SREM Set + 幂等 Key + 死信队列
5. **为什么选 Etcd** —— Go 实现性能高、Raft 强一致、watch 通知机制完善
6. **ConcurrentHashMap 锁粒度** —— 桶级别细粒度，写不同桶并发不冲突

## 真题摘录（值得背的）

> **多个交易系统订单按时间分页，怎么避免漏单？** 出处 [[蚂蚁/蚂蚁一面]]
> 关键陷阱：只记录"上一页最晚时间 T"做游标会漏单——如果下一页查询前某系统插入了 T 之后但已展示过的时间点的新订单，永远查不到。正确做法是为每个系统记录 (lastTime, lastId) 复合游标，下一页对每个系统按 `WHERE (time<lastTime) OR (time=lastTime AND id<lastId)` 查 N 条，合并排序取 10 条。每页处理约 N×系统数 条数据，性能可控。

> **优惠券领了下单失败要回滚怎么办？** 出处 [[蚂蚁/蚂蚁一面]]
> 下单失败发回滚消息到 MQ，异步消费里用 Lua 脚本原子地 INCR 库存 + SREM "已领用户" Set。每次回滚生成唯一操作 ID 做幂等 Key 防重复归还。延迟回滚靠 RocketMQ 延时消息或 Redis TTL，订单超时未支付自动释放库存。
