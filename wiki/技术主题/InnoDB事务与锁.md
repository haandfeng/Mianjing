# InnoDB事务与锁

> 表锁（MDL/意向锁/AUTO-INC）+ 行锁（记录/间隙/Next-Key）共同维护并发；读写锁配合 MVCC 做到既不阻塞读又能保证一致性。

## 答题骨架

### 1. 表级锁 4 种
- **表锁**（`LOCK TABLES`）：粗粒度，限制本线程和其他线程
- **MDL（元数据锁）**：自动加，CRUD 加 MDL 读锁，DDL 加 MDL 写锁；执行 DDL 时被长事务卡住会引发线上事故
- **意向锁**（IS / IX）：要加行级 S/X 锁前先在表级加 IS/IX，目的是**快速判断表内是否已有行被锁**，加表锁时不必遍历所有行
- **AUTO-INC 锁**：插入自增列时短暂的表级锁，5.1.22+ 改为轻量级（赋值后立即释放）；批量插入时主从一致需要 binlog 用 row 格式

### 2. 行级锁 3 种（仅作用在索引上）
- **记录锁（Record Lock）**：锁住索引上的某一行
- **间隙锁（Gap Lock）**：锁住索引上的一段开区间，防止区间内插入新行（仅 RR 启用）
- **Next-Key Lock**：记录锁 + 前面的间隙锁组合（左开右闭区间），是 InnoDB 解决幻读的主力

### 3. 锁的兼容矩阵
| | S | X | IS | IX |
|---|---|---|---|---|
| S | ✓ | ✗ | ✓ | ✗ |
| X | ✗ | ✗ | ✗ | ✗ |
| IS | ✓ | ✗ | ✓ | ✓ |
| IX | ✗ | ✗ | ✓ | ✓ |

- 表锁和行锁满足"读读共享、读写互斥、写写互斥"
- 意向锁互相不冲突（仅与表级 S/X 冲突）

### 4. `SELECT ... FOR UPDATE` 加锁分析
- RR 隔离级别下，等值查询命中**唯一索引**：仅加记录锁
- RR 等值查询**未命中**或非唯一索引：加 Next-Key Lock，可能加间隙锁
- 范围查询：从扫描的第一个键开始按 Next-Key 一直锁到下一个键
- 普通 `SELECT` 走 MVCC，不加行级锁
- 锁在事务 commit / rollback 时释放

### 5. ACID 与机制对应
| 特性 | 实现机制 |
|---|---|
| 原子性 Atomicity | undo log 回滚 |
| 一致性 Consistency | 由 ACID 其他三个共同保证 |
| 隔离性 Isolation | MVCC + 锁 |
| 持久性 Durability | redo log + WAL |

## 易错点 & 追问
- "意向锁会和行锁冲突吗？" → 不会，意向锁只与表级 S/X 冲突
- 间隙锁只在 RR 下生效，RC 默认关闭间隙锁
- 死锁排查：`SHOW ENGINE INNODB STATUS` 看 LATEST DETECTED DEADLOCK
- 普通 SELECT 不会加行锁；FOR UPDATE 是当前读，会读到最新
