## 答题骨架

### 1. B+ 树为什么适合磁盘索引
- 非叶子节点只存 key + 指针，不存数据 → 单页能容纳更多 key，树更矮，磁盘 IO 更少
- 叶子节点之间双向链表 → 范围查询、`ORDER BY`、`LIMIT` 可以顺序扫描
- 对比 Hash：只支持等值，没有范围；对比 B 树：B 树非叶子也存数据，IO 更多

### 2. 聚簇索引 vs 二级索引
- **聚簇（主键）索引**：叶子节点存完整行数据。一张表只有一个
- **二级（非聚簇）索引**：叶子节点存 索引列 + 主键值；查到主键后还要回到聚簇索引取整行 = 回表
- 没主键时 InnoDB 选第一个 NOT NULL UNIQUE 列；都没有就用隐藏的 6 字节 row_id

### 3. 联合索引最左匹配 + 索引下推
- `(A,B,C)`：能走 `A`、`A,B`、`A,B,C`；不能直接走 `B` 或 `B,C`
- 等值条件优化器会自动调换顺序 → `WHERE B=x AND A=y AND C=z` 能完全命中
- **范围查询后续列失效**：`WHERE A>1 AND B=2`，B 走不了索引（跨多个 A 后 B 无序）
- **覆盖索引**：`SELECT` 字段都在索引里 → `Using index`，不回表
- **ICP 索引下推**（5.6+）：联合索引中范围列后面的等值条件可在索引层提前过滤，少回表

### 4. 索引失效场景
| 场景 | 例子 |
|---|---|
| 跳过最左列 | `(A,B,C)` 只查 `WHERE B=x` |
| 范围后列失效 | `WHERE A>1 AND B=2` |
| 函数包裹列 | `WHERE YEAR(create_time)=2024` |
| 隐式类型转换 | `phone` 是 varchar 但 `WHERE phone=138...`（没引号） |
| 左模糊 | `LIKE '%abc'` |
| OR 连非索引列 | `WHERE A=1 OR D=2`（D 没索引） |
| `!=` / `<>` / `NOT IN` | 部分场景不走 |

### 5. Join 算法（NLJ / INLJ / BNL）
- **Index Nested-Loop Join**：被驱动表 join 列有索引 → 走索引，O(N·logM)
- **Block Nested-Loop Join**：被驱动表无索引 → 驱动表批量进 join_buffer，扫被驱动表，整表 buffer 内比对；扫描行数 = N + M，比对次数 = N×M（内存中）
- 优化：让小表（先过滤后行数更少的）做驱动表；给被驱动表 join 列加索引；调大 `join_buffer_size`

### 6. EXPLAIN 关键列
- `type`：从好到差 `const > eq_ref > ref > range > index > ALL`，看到 `ALL` 就是全表扫
- `key` / `key_len`：实际用到的索引和长度（判断联合索引用了几列）
- `rows`：估算扫描行数
- `Extra`：`Using index`（覆盖索引）`Using filesort`（额外排序，差）`Using temporary`（临时表）`Using index condition`（ICP）

### 7. 范式（1NF / 2NF / 3NF / BCNF）
- 1NF：列原子；2NF：消除部分依赖（针对组合主键）；3NF：消除传递依赖；BCNF：所有决定因子必须是候选键
- 工程上一般到 3NF，再按读写性能反范式（冗余字段）

## 易错点 & 追问
- 性别这种**低区分度**列加索引几乎没用，反而增加维护成本；高区分度（user_id、order_id）才适合
- `(A,B,C)` 三个等值条件无所谓 SQL 中的顺序——优化器会调整；但**只要中间出现范围**，后面就废了
- 覆盖索引能消除回表，但要权衡索引列宽度（叶子页能放更少节点 → 树变高）
- BNL 可怕之处是 join_buffer 装不下时会多次扫被驱动表
- `SELECT *` 容易破坏覆盖索引
