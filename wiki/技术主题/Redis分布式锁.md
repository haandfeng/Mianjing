# Redis分布式锁

> 基于 `SET key val NX EX TTL` 实现互斥；用 Lua 脚本保证"判主 + 删锁"原子；看门狗续期 + Redisson 兜底。

## 答题骨架

### 1. 加锁
```
SET unique_key  机器码:线程ID  NX  EX  TTL
```
- `NX` 保证互斥：key 不存在才成功
- `EX TTL` 防死锁：进程崩了，锁自动到期
- value 存"机器码 + 线程 ID"：用于校验锁归属，防误删

### 2. 释放锁（Lua 脚本，原子）
```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```
- 必须原子：判断和删除分开做时，可能 GC 暂停 / TTL 到期 / 别人拿到锁，再删就误删了别人的锁

### 3. 三大问题与解法
| 问题 | 描述 | 解法 |
|---|---|---|
| **锁误删** | A 业务超时，TTL 到，B 拿到锁；A 醒了删锁删的是 B 的 | value 存唯一标识，删前校验（Lua 原子） |
| **TTL 提前到** | 业务还没做完锁就释放 | **看门狗**：另起线程定时续期（如每 TTL/3） |
| **不可重入** | 同线程二次加锁失败 | Redisson 用 Hash：field=线程ID, value=重入次数；进 +1，出 -1，归零才删 |

### 4. 看门狗（Watchdog）机制（Redisson 默认）
- 加锁成功后，启动后台定时任务（默认每 10s）
- 把 TTL 重新设为初始值（默认 30s）
- 业务一直跑就一直续；进程挂了看门狗也挂，TTL 到期自动释放

### 5. RedLock（多 Redis 实例）
- 单 Redis 主从切换时丢锁：主节点写入但还没同步从节点就挂了
- RedLock：向 N（一般 5）个独立 Redis 实例都加锁，多数（≥3）成功才算加锁成功
- 业界争议：Martin Kleppmann 认为 RedLock 在 GC 暂停 / 时钟漂移下仍不可靠
- 实际工程多数场景单实例 + 看门狗够用；强一致选 ZooKeeper / etcd

### 6. 与 ZK 锁对比
| | Redis 锁 | ZooKeeper 锁 |
|---|---|---|
| 性能 | 高 | 中等（写需多数派） |
| 可靠性 | 主从切换有丢锁风险 | 临时节点 + 通知，更可靠 |
| 实现 | SETNX + Lua | 临时顺序节点 + Watch |

## 易错点 & 追问
- "为什么不能 SETNX + EXPIRE 两步加锁" → 不原子，用单条 `SET ... NX EX`
- "为什么不直接 DEL" → 可能误删别人的锁
- "Redis 集群下分布式锁安全吗" → 主从异步复制 + 主节点宕机有丢锁风险，看业务能否容忍
- 业务尽量幂等，不依赖锁的强一致性
