# Redis缓存三大问题

> 雪崩 / 击穿 / 穿透三件套，记住"是不是热点 + 是不是真存在"两个维度就能秒辨。

## 答题骨架

### 1. 三种问题区分
| 问题 | 定义 | 触发 |
|---|---|---|
| 缓存**穿透** | 查询的数据**不存在**于 DB → 缓存永远 miss | 恶意刷不存在的 ID |
| 缓存**击穿** | 单个**热点 key 失效**瞬间，海量请求打 DB | 热点 key 恰好过期 |
| 缓存**雪崩** | 大批 key 同时失效（TTL 设一样） / Redis 节点挂掉 | TTL 集中失效、Redis 宕机 |

### 2. 穿透解法
- **布隆过滤器**：DB 中不存在的 key 直接被拦截，不打 DB（有极低误判率，可能漏拦截极少量）
- **缓存空对象**：DB 查到空也写入 Redis（空值），TTL 设短，避免短期内 DB 又被插入对应数据后一直返回空
- **接口层校验**：明显非法的参数直接拒（负数 ID 等）

### 3. 击穿解法
- **互斥锁（mutex key）**：缓存失效时第一个请求 `SETNX` 拿锁去 DB，其他请求等或重试。一致性最好，有短暂等待
- **逻辑过期 + 异步重建**：value 里带"逻辑过期时间"，key 永不真过期。逻辑过期后单独线程异步刷新，其他请求继续读旧数据。可用性最好，可能短暂返回旧数据
- 永不过期的纯热点（首页、字典）+ 后台定时刷新

### 4. 雪崩解法
- **TTL 加随机抖动**（基础值 + 随机几十秒到几分钟），分散失效时间
- **多级缓存**：本地缓存（Caffeine） + Redis；本地缓存做兜底
- **Redis 集群 + 哨兵 / Cluster**，避免单节点挂导致整体雪崩
- **限流 / 降级**：失效瞬间用限流挡住，DB 有缓冲

### 5. 一致性更新策略：先更新 DB 再删缓存（标准做法）
- 不推荐"双写"：并发下时序错乱、缓存利用率低、计算浪费
- "先删缓存后更 DB"：在写写并发或 DB 慢的场景下旧值容易回写
- "先更 DB 后删缓存"：理论极端场景才会不一致（缓存正好失效 + 读慢于写），生产中概率极低
- 兜底：消息队列 / 订阅 binlog 重试删除；TTL 兜底；延迟双删（更新 DB 后睡几百 ms 再删一次）

### 6. 实际业务取舍（小鹅通改价案例）
- 商户改价时**优先保证下单可继续**而不是数据完美一致
- 实际方案：**先改 DB 再改/删 Redis**；改 Redis 时间窗口极短，比商户人工发现错误的时间小得多
- 价格错了顶多道歉退款，下单功能不能停

## 易错点 & 追问
- "先删缓存"会导致旧值回写——读了 DB 旧值，然后写回缓存
- 布隆过滤器无法删除元素（除非用 Counting BF），需要重建
- 缓存空对象的 TTL 不能太长，否则数据"真插入"后用户长期看到空
- 雪崩 != 大量请求打 DB；雪崩重点是"集中"
- 商业 prompt 问"如何避免 redis 不一致"先反问业务是允许短暂不一致还是要求强一致
