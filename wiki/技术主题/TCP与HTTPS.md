## 答题骨架

### 1. 三次握手（核心：阻止历史连接 + 同步序列号）
1. C → S：SYN, seq=x
2. S → C：SYN+ACK, seq=y, ack=x+1
3. C → S：ACK, seq=x+1, ack=y+1
- 不能两次：阻止不了历史连接（旧 SYN 重复到达，服务端无法判断）
- 不能四次：浪费一次往返
- 第三次可携带数据；前两次不行（避免 SYN Flood、连接未建立时浪费资源）

### 2. 四次挥手
1. C → S：FIN（"我没数据要发了"）
2. S → C：ACK
3. S → C：FIN（"我也发完了"）
4. C → S：ACK，进入 TIME_WAIT 2MSL
- TIME_WAIT 2MSL：保证最后一个 ACK 到达对方；让本次连接残留报文消失，避免和下一次连接混淆

### 3. TCP 可靠传输三大机制
- **序列号 + 确认号**：每字节有序号，按序拼回字节流
- **重传**：超时重传 / 快速重传（3 个重复 ACK）/ SACK
- **流量控制**：滑动窗口，接收方告知窗口大小
- **拥塞控制**：cwnd 动态调整
  - 慢启动：cwnd 翻倍
  - 拥塞避免：超过 ssthresh 后线性增长
  - 快重传 + 快恢复：3 个重复 ACK 时 cwnd / ssthresh 减半，进入快恢复
  - 超时：ssthresh = cwnd/2，cwnd 重置为 1

### 4. MTU vs MSS vs 分片 / 分段
- **MTU**（数据链路层）：以太网 1500B
- **MSS**（TCP）：1460B = MTU - IP 头 - TCP 头
- 数据链路层自己不拆，超就丢；IP 层负责分片
- TCP 分段（按 MSS）是主力；IP 分片是兜底（任何一片丢全报废，无重传）
- 现代网络通过 PMTU Discovery 让 TCP 自己控制段大小

### 5. HTTPS 加密流程
- HTTP 三大风险：窃听、篡改、冒充
- HTTPS = 混合加密：
  1. 握手用**非对称**加密协商出对称密钥
  2. 后续通信全用**对称**加密
- TLS 1.2 握手：
  1. C → S：ClientHello（支持算法列表 + 随机数 A）
  2. S → C：ServerHello（选定算法 + 随机数 B + CA 证书 + 公钥）
  3. C 校验 CA → 用公钥加密随机数 C 发给 S（pre-master secret）
  4. 双方用 A+B+C 派生对称会话密钥
  5. 互发 Finished 消息（用新密钥加密验证）
- TLS 1.3 优化到 1 RTT；0-RTT 复用会话

### 6. 数字证书防中间人
- Server 把公钥提交给 CA → CA 用自己私钥对 Server 公钥签名 → 打包成数字证书
- Server 通信时把证书一起发；C 用预置的 CA 公钥解签名 → 拿到 Server 真实公钥
- 没有证书的话 MITM 可以替换公钥，看不出来

### 7. 对称 vs 非对称加密
| | 对称 | 非对称 |
|---|---|---|
| 密钥 | 一把 | 公私钥对 |
| 性能 | 快 | 慢 |
| 痛点 | 密钥分发难 | 慢，不适合大数据 |
| 用途 | 大数据通信 | 密钥交换 + 身份认证 + 数字签名 |

### 8. 应用层粘包 / 半包三解法
- 分隔符（HTTP/1.x：`\r\n\r\n`）
- 固定长度头（HTTP/2、Dubbo）
- 固定长度消息

### 9. DNS 解析流程
- 浏览器缓存 → 系统缓存（hosts）→ 路由器缓存 → 本地 DNS 服务器
- 本地 DNS 没缓存 → **迭代查询**根 → TLD（.com）→ 权威服务器
- 客户端对本地 DNS 是**递归查询**
- DNS 默认 UDP 53 端口；超过 512 字节降级 TCP

### 10. 浏览器输入 URL 到页面渲染
1. URL 解析（协议 / 域名 / 路径）
2. DNS 解析得到 IP
3. TCP 三次握手
4. HTTPS 走 TLS 握手
5. 协议栈封装：HTTP → TCP → IP → MAC（ARP）
6. 网卡 → 交换机（看 MAC）→ 路由器（看 IP，剥换 MAC，TTL -1）
7. 服务器接收 → 解封装 → Web 服务器进程处理
8. 响应原路返回 → 浏览器渲染

### 11. JWT 结构
- `Header.Payload.Signature`，三段用 `.` 分割
- Header：算法 + 类型
- Payload：claims（user_id, exp 等）
- Signature：HMAC(Base64Url(Header) + "." + Base64Url(Payload), secret)
- 验证：服务端用相同密钥重新签名比对，无需查 DB
- 缺点：无法主动失效（要靠黑名单 / Redis 记录）

## 易错点 & 追问
- "TCP 三次握手为什么不能两次" → 防历史连接、同步双方初始序列号、避免无效连接
- 滑动窗口控的是流量，cwnd 控的是拥塞
- HTTPS 握手用 RSA 还是 ECDHE：现代用 ECDHE，提供前向安全（即使私钥泄露，过去的会话密钥也无法被推算）
- DNS 大部分时候用 UDP，TCP 仅在响应过大或 zone transfer 时
- TIME_WAIT 太多可调 `tcp_tw_reuse`
