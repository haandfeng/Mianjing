# 数据迁移与一致性

> 大表迁移要分阶段：全量同步 → 增量追平 → 切流；每批次靠数据库事务保原子，事务里同时更新数据 + checkpoint + upsert 幂等；不停机迁移用双写 + 影子读 + 灰度切流；生产对账兜底。

## 答题骨架

### 1. 三阶段：迁移前 / 中 / 后（参考 [[快手/大数据平台方向/快手一面面经]]）
**前**：
- 目标库快照备份（pg_dump / xtrabackup）
- 定 checkpoint 机制 + 幂等写入策略（upsert / `INSERT ... ON CONFLICT`）

**中**：
- 每批一个 DB 事务：同时执行 upsert + 推进 checkpoint，要么都成要么全回滚
- 传输层校验靠 gRPC / HTTP/2 内置；关键数据加 checksum / 行数对比
- 失败：事务自动 rollback，checkpoint 不推进，下次从该批次重试（配合 upsert 幂等）

**后**：
- 全量校验：count 对比 + 关键字段按主键区间 hash 聚合（MD5）

### 2. 增量同步标记的选择
- ❌ 自增 ID：只能捕获 insert，update 漏掉
- ✅ `updated_at` 时间戳：insert / update 都能捕获
- 删除：软删（`deleted_at`）配合
- 更重的方案：CDC（监听 binlog / WAL）

### 3. 不停机迁移：双写 + 灰度切流（[[阿里云/阿里云消息团队一面面经]]）
1. **全量搬迁**：分块 + 禁用索引 + 多 worker 并行 + 每块 checksum
2. **开启双写**：先写旧（主），成功再写新（副）；新库失败丢 Kafka 重试队列；幂等 ID 防重复
3. **影子读**：客户继续读旧库，后台对新库重复执行查询，比对结果，暴露时区 / 编码 / 排序差异
4. **正式切读**：新库预热缓存 → 灰度切读（1% → 5% → 20% → 100%）；继续双写一周以上备回滚
5. **下线旧库**：对账确认稳定后

### 4. 切流 / 回滚预案
- 双写阶段：关双写开关
- 灰度切读：读流量切回旧库
- 全量切换后：依赖双写期间双库一直同步，可直接切回

### 5. CDC（Change Data Capture）方案
- MySQL：监听 binlog（Canal、Maxwell、Debezium）
- PG：逻辑解码 WAL → Debezium → Kafka
- 同用户操作有序：用 user_id 作为 partition key（Kafka）/ Consistent Hash Exchange（RabbitMQ），保证 FIFO
- 消息进 Kafka topic，下游消费按序写新库

### 6. 跨系统强一致（支付 / 银行场景）
| 方案 | 思路 | 适用 |
|---|---|---|
| **本地消息表** | 业务事务里同时写消息表（待支付）→ 异步调银行 → 成功更新状态 | 多数业务首选 |
| **对账系统** | 每天 / 每小时拉银行账单逐条比对，差异写差异表 + 告警 | 兜底必选 |
| **TCC** | Try 冻结 → Confirm 真扣 → Cancel 回滚；多方协调 | 涉及余额 + 优惠券 + 积分等多资源 |
| **最大努力通知** | 银行回调 + 失败重试 | 简单但不够可靠，要配对账兜底 |

实战组合：**本地消息表 + 幂等重试 + 对账兜底**，原则"宁可多查一次也不能漏一笔"。

### 7. 积分 / 余额场景（顺序敏感）
- 不能用 UPSERT 直接覆盖余额（中间变化丢失）
- 用流水表：每笔变更都有唯一 ID 做幂等
- 对账：按时间段把流水净额加总 vs 余额变化做比对
- 顺序：用 user_id 做 partition / 一致性 Hash，同一用户的消息走同一个消费者按 FIFO 处理

### 8. 关键观测指标（贯穿全程）
- 同步 / 复制延迟（新库比旧库慢多少秒）
- 影子读不匹配数
- 新库慢查询 / 死锁
- 缓存命中率（切换后必须 95%+）
- 业务 KPI（每分钟订单数 / 收入流水）

## 易错点 & 追问
- "为什么不一开始就直接对比所有数据" → 几亿条不现实；按主键模 / 时间窗口抽样 + 业务维度（流水净额）
- 迁移完不要立即下线旧库，留作回滚
- 影子读发现的差异类型：时区 / NULL / 排序规则（utf8 vs latin1）
- 对账系统应独立部署，不和同步系统耦合
- 灰度切流需配合 Feature Flag
