# 消息队列不丢不重

> 三段链路（生产者→Broker→消费者）每段都可能丢；at-least-once + 业务幂等 = effectively-once。死信队列做兜底。

## 答题骨架

### 1. 三段链路全景
```
Producer → Broker → Consumer
  Confirm   Durable   Manual ACK
```
- **生产者→Broker**：Confirm 机制 + mandatory（路由失败回退）+ 失败重试
- **Broker 自身**：Exchange / Queue / 消息三层 durable，宕机不丢
- **Broker→Consumer**：手动 ACK + 死信队列 + 业务幂等

### 2. 生产者侧
- **Confirm 模式**：每条消息发出后 Broker 回 ACK，超时 / NACK 就重发
- **mandatory**：消息到了 Exchange 但路由不到 Queue → 退回（ReturnCallback）
- **本地消息表**（强保证）：业务事务里把订单 + 待发消息一起写库；定时任务扫"待发送"重投到 MQ

### 3. Broker 侧
- Exchange `durable=true`、Queue `durable=true`、消息 `deliveryMode=2`
- 高可靠选 Quorum Queue（多副本，类 Raft）

### 4. 消费者侧
- `autoAck=false`，业务执行成功后再 `basicAck`
- 失败：
  - 可恢复（DB 临时连不上）→ `requeue=true` 重试几次
  - 不可恢复（数据格式错）→ `requeue=false` 进死信，避免无限重试
- **业务幂等**：用业务唯一 ID（订单号）做去重，重复消费不产生重复结果

### 5. 死信队列（DLQ）原理
- 三种触发死信：
  1. 被 reject / nack 且 `requeue=false`
  2. 消息 TTL 过期
  3. 队列达到 `x-max-length`，最旧消息被挤出
- 原队列声明时配 `x-dead-letter-exchange` + `x-dead-letter-routing-key`
- 死信交换机 / 队列就是普通的 Exchange / Queue，本质就是消息一次转发

### 6. 端到端 = at-least-once + 幂等 = effectively-once
- 任何一步重试都可能产生重复
- 消费者必须幂等：唯一约束（INSERT ... ON CONFLICT DO NOTHING）、状态机校验（已支付的订单不再扣款）

### 7. 消息积压处理（[[小鹅通往年面经+我的面经/小鹅通一面面经（带答案已OC）]]）
- **生产端限流**：令牌桶 / 漏桶削峰
- **消费端扩容**：增加并发消费者，水平扩节点
- **批量消费**：一次拿一批做批处理
- **降级**：低优先级消息丢弃 / 进延迟队列；TTL 让低优先级自动过期
- **临时扩容**：堆积严重时临时上多个消费实例打捞

### 8. 三大 MQ 选型
| | RabbitMQ | RocketMQ | Kafka |
|---|---|---|---|
| 吞吐 | 万级 | 十万级 | 百万级 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 路由 | Exchange 4 种最灵活 | Topic + Tag | 仅 Topic |
| 事务消息 | 不原生 | 原生（半消息 + 回查） | 不原生 |
| 消息回溯 | 不支持 | 支持 | 支持（重置 offset） |
| 延时消息 | 插件 / TTL+DLQ | 原生 | 不支持 |
| 生态 | 通用业务 | 电商 / 金融 | 大数据 / 日志 |

### 9. RocketMQ 事务消息（半消息）
1. 生产者发"半消息" → Broker 不投递
2. 执行本地事务（扣库存）
3. 成功 → 发 commit；失败 → 发 rollback
4. Broker 长时间没收到 → 主动回查生产者状态

### 10. 业务案例：秒杀异步落库（参考 [[字节-今日头条-财经业务/字节-国际广告and中国广告/面经记录_国际商业化创业_3.9]]）
- Redis 扣库存（Lua 原子）+ 一人一单校验
- MQ 发订单消息异步落库
- 订单号做幂等
- 失败兜底：消息表 + 死信队列 + 人工介入

## 易错点 & 追问
- 死信队列 ≠ 重试队列；要专门一个消费者来处理
- 顺序消息：单队列单消费者；或 partition + 同 key 路由
- "消息丢失"和"消息重复"是两回事，分别用 ACK 和幂等处理
- Kafka 没事务消息，需要业务自己实现本地消息表
