# 缓存一致性

> 写场景下选择"先更新数据库，再删除缓存"是工程默认选择；极端不一致用消息队列重试 + 延迟双删 + TTL 兜底。

## 答题骨架

### 1. 五种方案对比
| 方案 | 并发问题 | 评价 |
|---|---|---|
| 先更新缓存，再更新 DB | 时序错乱：A 旧值覆盖 B 新值 | 不推荐 |
| 先更新 DB，再更新缓存 | 同样错乱 + 缓存利用率低（无人读也写） | 不推荐 |
| 先删缓存，再更新 DB | 读请求穿透：B 读到 DB 旧值并回写 | 概率较高 |
| **先更 DB，再删缓存（推荐）** | 仅在"缓存恰好失效 + 读慢于写"极端场景出现，概率极低 | ✅ |
| 订阅 binlog（CDC） | 异步删除缓存，最终一致 | 高一致性场景 |

### 2. "先更 DB 再删缓存"为什么是默认
- 写 DB 通常需要持锁、SQL 解析、刷盘，比读 DB + 写 Redis 慢
- 极端时序：①读请求 cache miss 读 DB 旧值 → ②写请求 update DB → ③写请求 del cache → ④读请求把旧值写入 cache
- 第三步发生在第四步之前的概率很低（写远慢于读 + cache）

### 3. 删缓存失败怎么办
- **重试**：消息队列异步重试删除（最常见兜底）
- **订阅 binlog**：MySQL canal / Debezium 监听变更事件 → 删缓存
- **TTL 兜底**：所有缓存都设 TTL，最差也只有这段时间不一致
- **延迟双删**（针对主从延迟 / 写后读 race condition）：更新 DB → 删缓存 → 睡几百 ms → 再删一次

### 4. 主从延迟引发的二次写脏
- 写主库 → 删缓存 → 主从未同步完时，读请求打到从库读到旧值 → 回写缓存
- 解决：延迟双删；或读写都强制走主库；或控制业务可容忍程度

### 5. 业务取舍：一致性 vs 可用性
- 商品价格更新 vs 下单：先保证下单可用，再更新缓存（参考 [[小鹅通/小鹅通二面面经（结果未知）Cider一面面经（通过）]]）
- 库存类强一致：必须 DB 一致，缓存只是性能加速；最终走 DB 校验
- 评论点赞这种弱一致：cache 主动延迟更新没问题

### 6. 黑马点评的事务方案
- 把"更新 DB + 删 Redis"包在 Spring `@Transactional` 内，删 Redis 失败可触发回滚
- 注意 Redis 操作不在 DB 事务里，需手动捕获异常并回滚 DB
- 比消息队列简单，适合中等并发

## 易错点 & 追问
- "更新缓存还是删除缓存？" → 删除：成本低，避免重复计算未读的数据
- 主从延迟下"延迟双删"的延迟时间难评估，过短没效果、过长影响 TPS
- 不要忘了 Redis 操作可能失败（网络抖动），生产代码必须有兜底
- 强一致需求 → 读写都走 DB，缓存退位
