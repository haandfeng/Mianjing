# 限流熔断容错

> 保护被调方靠**限流 + 降级**，保护调用方靠**超时 + 熔断 + 隔离**。两端都要做，否则下游慢就拖死上游。

## 答题骨架

### 1. 4 种限流算法
| 算法 | 思路 | 优 | 缺 |
|---|---|---|---|
| 固定窗口 | 每窗口计数 | 简单 | 临界突发：第 1s 末 100 + 第 2s 初 100 = 200 突破 |
| 滑动窗口 | 切小格滚动 | 缓解临界 | 实现稍复杂 |
| 漏桶 | 固定速率出桶 | 严格平滑 | 不允许突发，资源浪费 |
| 令牌桶 | 固定速率放令牌、消费令牌 | 允许突发 | 实际最常用 |

实战常用：**滑动窗口 + 令牌桶**

### 2. 漏桶 vs 令牌桶差别
- **漏桶控流出速率**：来多少都按桶口出
- **令牌桶控流入速率**：桶里能攒令牌，允许短暂突发
- 大多数场景令牌桶更实用（Guava `RateLimiter`、Nginx `limit_req`）

### 3. 实现技巧：不用定时器，用时间差懒计算
- 每次请求来：根据"上次操作时间 → 现在"算出这段时间应漏多少 / 应补多少令牌
- 然后再判断 / 扣减
- 漏桶和令牌桶代码极相似，只是符号反过来

### 4. 三状态熔断器
```
CLOSED ──失败率超阈值──> OPEN ──冷却时间到──> HALF_OPEN
   ▲                                            │
   └──────────试探成功 ───────────────────────────┘
   └──────────试探失败 ───────────> OPEN
```
- CLOSED：正常放行，统计失败率
- OPEN：所有请求直接拒绝，启动冷却
- HALF_OPEN：放少量探测请求，全成功回 CLOSED，失败回 OPEN

### 5. 保护被调方（B 端）
- **限流**：QPS / 并发线程 / 排队
- **降级**：压力大时关掉非核心逻辑（推荐 / 个性化排序），返回默认数据
- 工具：Sentinel / Hystrix / Resilience4j

### 6. 保护调用方（A 端）
- **超时（Timeout）**：调下游必设；不设是雪崩起点
- **熔断（Circuit Breaker）**：下游慢 → 全部拒绝，避免线程池被占满
- **隔离（Bulkhead）**：每个下游独立线程池 / 信号量，B 挂不影响 C

### 7. Redis / MQ 实现限流
| 算法 | Redis 数据结构 | 说明 |
|---|---|---|
| 固定窗口 | INCR + EXPIRE | key 带时间戳 |
| 滑动窗口 | ZSet（score 存时间戳） | ZREMRANGEBYSCORE 清旧 + ZCARD 计数 |
| 漏桶 / 令牌桶 | Hash（容量 + last_time） | Lua 脚本惰性计算 |

- 多步操作必须 Lua 脚本保证原子
- **MQ 限流**：请求入队，消费端按速率消费，天然漏桶模型；适合异步削峰，不适合实时同步

### 8. 业务实战：秒杀场景的"假漏桶 vs 真漏桶"
- 真漏桶要排队 + 定时消费，引入延迟，业务通常受不了
- 业界很多"漏桶"实现其实做的是准入控制 = 简化版令牌桶
- 真正需要严格平滑的是 Linux `tc` 流量整形，业务层基本用不上

## 易错点 & 追问
- 限流和熔断不是同一回事：限流挡进入流量，熔断挡出去调用
- "滑动窗口" 别说成"细分多个固定窗口"；它是真正按时间滑动
- 熔断半开放过的请求数要限制，避免一次性放太多
- 隔离线程池是 Bulkhead 模式核心，类比舱壁
- 限流算法都要考虑分布式：单机限流总和 ≠ 集群限流，需要中心化（Redis / Sentinel 集群）
