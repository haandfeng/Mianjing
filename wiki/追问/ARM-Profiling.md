# ARM-Profiling

## 一句话介绍
基于 perf 的 off-CPU profiling 工具，把 5.x 内核实现迁移到 6.8，能预测「优化某段代码后整体吞吐量提升多少」。

## 项目级 STAR 自陈
- **S（场景）**：思路源于 2015 年 SOSP best paper（COZ），AMD 已将类似方案应用到 LCM；2024 年作者基于 Linux 5.x 提出 off-CPU 优化方案；ARM 平台需要 6.8 内核版本。
- **T（目标）**：把工具迁移到 ARM + 6.8 内核，补足 off-CPU 分析能力（IO wait、锁等待），并增强调用栈上下文 / 预热支持。
- **A（动作）**：1）改 Linux kernel 和 perf 子系统，加入线程 sleep/wake/被调度时间戳；2）用 AI 辅助分析 5.x → 6.8 的函数和数据结构变化，自己写迁移 plan 并 review；3）补充预热参数、按代码行聚合调用栈、链接阶段符号拦截做锁竞争追踪；4）每次改完跑 MySQL TPS benchmark 验证。
- **R（结果）**：完成内核迁移与功能扩展，定位到 CRC 校验是 ARM 上的热点（x86 有专门 CRC 加速指令而 ARM 没有），证明优化 CRC 20% 执行速度可以带来整体 TPS ≈ 40% 提升。

## 高频追问 Q&A

### Q1: off-CPU profiling 的原理和背景
**出处**：[[快手一面面经]] · [[小红书产品工程师二面面经]] · [[Bing图像视频业务实习一面面经]]

**答题骨架**：
- 工具基于 perf 命令做采样，但 perf 只能看到 on-CPU 执行，采样不到线程 sleep 状态
- off-CPU：记录线程被调出 CPU、再被调回 CPU 之间经过了多久（sleep / wake / 被调度进 CPU 的时间戳）
- 工具本质是给别的路径插入 delay 做"虚拟加速"实验，估计真正优化后的全局收益
- 如果线程在 off-CPU 期间本来该被采样但没采到，要根据这段时间反推少采了多少样本，把样本补回来
- 多线程程序的 critical path 包括 IO wait 和锁等待，目前还无法对 sleep 状态线程做虚拟加速来量化优化效果

### Q2: 项目最大挑战 / 内核迁移工作
**出处**：[[快手一面面经]] · [[小红书产品工程师二面面经]] · [[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 改 Linux kernel 和 perf 子系统，深入到时间戳记录点、等待类型分类的逻辑
- 难点 1：源码理解成本高（Linux 内核源码本身复杂）
- 难点 2：迁移难度大，5.15 → 6.8 的函数、调用路径、数据结构变化要逐个分析
- 克服方法：补 CSAPP 等系统底层基础 → 重点读 Linux perf/kernel 开发文档 → AI 辅助分析旧版本改动点和原因，结合源码自己验证

---

### Q3: 工具原本有什么问题？为什么要改？
**出处**：[[小红书产品工程师二面面经]] · [[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 功能不完整：分析 MySQL 时需要先 warm up，原工具不支持
- 上下文不够：只能告诉某行代码加速会怎样，不能展示完整调用栈上下文
- 不能分析 off-CPU 问题：基于 perf 只看 on-CPU，看不到线程 sleep / 锁等待 / IO 等待

---

### Q4: 预热怎么做的
**出处**：[[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 测试数据库高频写场景下的 TPS，需要避开预热阶段对结果的污染
- 加了一个新的命令行参数，让分析采样结果的线程 sleep 指定分钟数后再启动
- 采样线程本身不受影响，只延迟启动「分析采样结果」的线程

---

### Q5: 调用栈聚合怎么做的？
**出处**：[[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 原工具有 perf 自带的 call chain，但分析某行代码时没考虑这行代码的调用栈上下文
- MySQL 这种系统，同一行底层代码在不同场景被调用（读密集 vs 写密集 vs CPU 密集）需要区分
- 实现：通过 perf call chain 拿到每条采样的 PC（指令地址）→ 结合 debug 信息映射到具体代码行 → 还原调用栈
- 以代码行为 key，聚合所有调用栈，统计每条调用路径的采样次数

---

### Q6: 在项目中发现了哪些性能瓶颈？
**出处**：[[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 主要定位到 CRC 校验是热点
- MySQL 不管 binlog/redo log/undo log，刷页落盘都走 CRC 校验
- x86 Intel 有专门 CRC 加速指令，ARM 没有，存在性能 gap
- 工具结果：CRC 函数加速 20%，整体 TPS 提升约 40%
- 我的工作主要是为团队提供佐证

---

### Q7: 锁竞争监控怎么做？面试官追问的「系统级监控」短板
**出处**：[[小红书产品工程师二面面经]]

**答题骨架**：
- 现有思路：链接阶段符号拦截，把自己的函数符号放在更前面，让程序先进入包装函数做记录和插桩，再调真实实现（函数劫持 + 轻量插桩）
- 当前是"和被测程序一起编译/打包"的方案，不是零侵入的纯系统外置工具
- 还需要在源码加宏和标记做轻量插桩
- 暴露的短板：对 inline hook / PLT-GOT / LD_PRELOAD、Linux 内核锁实现（mutex / 自旋锁 / futex / 阻塞唤醒链路）、eBPF / uprobes / kprobes / perf tracepoint 这类系统级观测手段储备不够

### Q8: AI Coding 在底层开发里的局限性
**出处**：[[快手一面面经]] · [[小红书产品工程师一面面经]] · [[Bing图像视频业务实习一面面经]]

**答题骨架**：
- 不用 agent 模式直接改代码，方法论：先让 AI 对仓库做预读建立理解 → 让它写 plan（要改哪些文件、怎么改、在哪改）→ 自己 review plan → 确认无误才让 AI 执行 → 跑 benchmark 验证
- 简单 prompt 下 agent 模式会改出不符合预期的代码（尤其内核开发场景）
- 怎么判断 AI 生成 C++ 代码性能：审查反模式（值传递 vs const 引用、循环内分配、move vs copy、shared_ptr vs unique_ptr）+ Google Benchmark 跑数据
- 实际遇到过 merge 新代码后 MySQL TPS 掉到 0 的多线程 bug，最终靠手动梳理核心线程模型 + AI 辅助定位修复，证明底层复杂问题不能只靠 AI 自动修复

---

### Q9: 怎么判断 AI 生成的 C++ 代码性能够不够好？
**出处**：[[快手一面面经]]

**答题骨架**：
- 代码审查看反模式：
  - 值传递 vs const 引用：大对象 const 引用 8 字节零拷贝
  - 循环内频繁 new/delete：vector 提到循环外用 clear() 复用
  - 该 move 没 move：copy 是逐元素拷贝，move 只是偷指针
  - 该 unique_ptr 用了 shared_ptr：shared_ptr 引用计数是原子操作
- 本质：C++ 没有 GC，每次分配/拷贝/引用计数都是实打实的开销
- Benchmark 实测：Google Benchmark 或计时脚本对比手写优化版本 vs AI 生成版本

---

### Q10: 测试用例 / Benchmark 怎么生成？
**出处**：[[小红书产品工程师一面面经]]

**答题骨架**：
- 测试与实现分离生成：让 AI 先写测试，自己 review 边界覆盖后再让它写实现，避免"自己出题自己答"
- Agent Loop 自动修复：AI 写完代码自动跑测试 → 报错 → 修复 → 再跑，循环到通过
- 交叉验证：一个 AI 写代码，另一个 AI review/写测试
- Property-based Testing：Python hypothesis 等库定义不变量，自动生成大量随机输入
- Differential Testing：和已知正确的旧实现对比，ARM 的 benchmark 脚本本质就是这个
