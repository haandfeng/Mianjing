# 黑马点评秒杀

## 一句话介绍
仿大众点评秒杀业务，Redis Lua 扣减库存（一人一单 + 不超卖）+ RabbitMQ 异步落库，平均耗时从 400ms 降到 100ms。

## 项目级 STAR 自陈
- **S（场景）**：秒杀场景对高并发与不超卖有要求，原始同步实现里扣减库存（Redis）+ 订单落库（DB）耦合在同一函数里，响应慢。
- **T（目标）**：保证不超卖、一人一单，提升响应速度。
- **A（动作）**：1）Redis Lua 脚本（原子）做库存扣减 + 一人一单判断；2）成功后把订单信息发到 RabbitMQ；3）消费者异步落库（手动 ACK + 死信队列兜底 + 失败记录表 + 本地消息表）；4）双拦截器（Token 拦截 + 鉴权拦截）+ ThreadLocal 存用户。
- **R（结果）**：耗时从 400ms 降到 100ms，主要是订单落库异步化省下的时间。

## 高频追问 Q&A

### Q1: 秒杀整体流程 / 100w 用户秒杀 100 个商品怎么扛
**出处**：[[阿里云消息团队一面面经]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[字节跳动中国商业化一面面经]] · [[小鹅通一面面经（带答案已OC）]] · [[蚂蚁一面]] · [[字节财经一面]] · [[快手日常实习面经]]

**答题骨架**：
- 假设请求都能走到 Redis（前面有前端限流/网关令牌桶、库存预热）
- Lua 脚本里：判断该 userId 是否已购买（防一人多单）→ 判断库存是否 ≤ 0（防超卖）→ DECR 库存 → SET userKey 标记已购买
- Redis 单线程事件循环 + Lua 脚本作为一个完整命令执行（脚本期间不处理其他命令），保证「检查-扣减-标记」原子性
- Lua 跑完异步发订单信息到 RabbitMQ 让消费者落库
- 整体购买走 Redis 内存性能高，写磁盘在后台进行，提高并发

### Q2: Lua 脚本为什么是原子的？
**出处**：[[Shopee DB Infra 北京平台实习面试问题总结]]

**答题骨架**：
- Redis 单线程事件循环，所有命令在同一线程顺序执行，保证单个命令原子
- Lua 脚本被视为一个完整的命令；执行期间 Redis 不处理其他任何命令，完成后才处理下一个
- 不可中断 + 顺序执行 + 无并发冲突
- 注意：Lua 不能执行太久（< 1s），不能有阻塞操作（BLPOP/BRPOP 不能用），错误不会回滚已执行的命令

### Q3: 一人一单怎么实现？乐观锁 vs 悲观锁
**出处**：[[蚂蚁一面]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[快手日常实习面经]]

**答题骨架**：
- Redis SET 数据结构记录 `seckill:user:{userId}:{goodsId}` = 1，TTL 兜底
- Lua 脚本里 EXISTS 判断 + DECR 扣减 + SET 标记，整体原子
- 乐观锁版本：判断获取到的库存是否为 0 / 获得的库存和最终下单时是否一致 + Lua 保证原子性
- 也可以用 Redis 分布式锁（SETNX + 锁 ID + 进程/线程 ID + TTL 逻辑过期 + 看门狗续期）

### Q4: 扣库存成功但发消息失败 / 消费失败怎么处理？
**出处**：[[阿里云消息团队一面面经]]

**答题骨架**：
- 三个环节保证不重不漏一定成功消费：
- 生产者：开启 Confirm 机制（消息到 Broker 持久化后回 ACK，超时重发）+ mandatory 参数（路由失败退回触发 ReturnCallback）；本地消息表（扣库存 + 写消息表在一个本地事务，定时任务扫"待发送"重发，即使 RabbitMQ 整个挂了订单也不丢）
- 队列：Exchange + Queue + 消息（deliveryMode=2）三层 durable=true；可靠性更高用 Quorum Queue
- 消费者：autoAck=false 手动 ACK，业务处理完才 basicAck；basicNack 区分 requeue=true（数据库临时连不上重试）和 requeue=false（数据格式错误进死信队列不要无限重试）
- 全链路是 at least once，靠订单 ID 唯一键（INSERT ... ON CONFLICT DO NOTHING）做幂等 → effectively once
- 多次失败：3 次重试都失败 → 死信队列 → 兜底消费者写入失败记录表 + 告警 → 人工修复后从失败记录表重新触发

### Q5: MQ 选型 RabbitMQ / Kafka / RocketMQ
**出处**：[[阿里云消息团队一面面经]] · [[小鹅通一面面经（带答案已OC）]]

**答题骨架**：
- 用哪个区别不大（QPS 不高），主要实现异步削峰
- 同一用户消息保序：Kafka 用 userId 做 partition key，partition 内天然有序；RabbitMQ 用 Consistent Hash Exchange 等效（多个 Queue + routing key hash 路由 + 每 Queue 一个消费者）
- 队列积压怎么办：生产者限流（令牌桶 / RateLimiter） + 消费者扩容（增加并发数、批量消费） + 降级（丢弃低优先级、TTL 过期）

### Q6: 用户拿到优惠券但下单失败怎么还原 / 库存回滚
**出处**：[[蚂蚁一面]]

**答题骨架**：
- 下单失败发"回滚消息"到 MQ，消息体含 userId、券 ID、操作类型、时间戳
- 库存回滚（Redis）：INCR 库存 + 1 + 从已领取用户 Set 移除该 userId，整个用 Lua 包装保证原子
- 幂等：每个回滚操作生成唯一操作 ID（订单 ID + retry ID），Redis 设幂等 Key 避免重复归还
- MQ 重试机制 + 死信队列保最终一致
- 秒杀场景加延迟回滚：RocketMQ 延时消息或 Redis TTL，订单超时未支付自动触发回滚，避免"秒杀后放弃下单"占用库存

### Q7: Token vs JWT / 双拦截器 + ThreadLocal
**出处**：[[小鹅通一面面经（带答案已OC）]] · [[银河智学]]

**答题骨架**：
- 验证码登录：手机号正则校验 → 从 Redis 取验证码校验 → 查/建用户 → 生成 UUID Token → User 转 Map 存 Redis Hash → 设 LOGIN_USER_TTL → 返回 Token
- 双拦截器：
  - 拦截器 1（拦截全部）：取请求头 Token → 基于 Token 取 Redis 用户 → 存 ThreadLocal → 刷新 Token 有效期；afterCompletion 移除 ThreadLocal 防内存泄漏
  - 拦截器 2（拦截需登录的请求）：判断 ThreadLocal 是否有用户，没有就 401 拦截
- ThreadLocal 线程安全：每个 Thread 维护自己的 ThreadLocalMap，key 是 ThreadLocal 对象，value 是变量副本

### Q8: 100w QPS Redis 怎么扛 / 如果 Redis 撑不住
**出处**：[[阿里云消息团队一面面经]] · [[Shopee DB Infra 北京平台实习面试问题总结]]

**答题骨架**：
- 分布式锁完整流程：SETNX 写入锁 ID + 进程/线程 ID + TTL 逻辑过期；解锁前判断存储的线程 ID 跟自己一致才能解；看门狗机制自动续期（每 10 秒后台判断任务是否完成，未完成续期到 30 秒）避免业务没跑完锁过期
- 缓存击穿/穿透/雪崩：
  - 雪崩：TTL 加随机数 + 多级缓存 + 热点预加载/永不过期
  - 击穿：逻辑过期（永不真正过期，过期后单线程异步更新，其他读旧数据）/ 互斥锁（SETNX 拿锁查 DB 回填）
  - 穿透：布隆过滤器 + 缓存空对象（短 TTL）

### Q9: 用户下单后看不到自己的订单怎么办
**出处**：[[小鹅通一面面经（带答案已OC）]]

**答题骨架**：
- MQ 的 ACK 机制确保每个订单都会被处理到
- Redis 先存"订单占位"：下单成功后在 Redis 记录 orderId = "处理中"；用户查询订单时先查 Redis 返回"正在处理"，等数据库落盘后再次查询就能查到订单信息
- 流程：先查 Redis pending 状态返回"处理中" → Redis 没有再查 DB 拿订单信息 → DB 也没有就是失败

### Q10: 400ms → 100ms 主要省在哪里
**出处**：[[小鹅通一面面经（带答案已OC）]] · [[阿里云消息团队一面面经]]

**答题骨架**：
- 原来同步：扣减库存（Redis）+ 订单落库（DB）在同一同步函数里
- 改造后：Redis Lua 扣库存（内存操作快）→ 把订单落库放到 RabbitMQ 异步处理 → 整体响应只算同步部分
- 削峰填谷：高并发请求转为异步处理
- 解耦：下单服务和数据库服务解耦
- 可靠性：消息持久化保证不丢
