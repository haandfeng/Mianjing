# 用友数据迁移

## 一句话介绍
帮中国电信国际升级人力系统（PostgreSQL 大版本升级、Schema 不兼容），通过 gRPC + Python 中间层做数据迁移，事务 + checkpoint 保证一致性、完整性与可回滚。

## 项目级 STAR 自陈
- **S（场景）**：旧系统和新系统都是 PostgreSQL，但表结构 Schema 不同，跨多国员工数据格式杂乱（如手机号有 +86、(86)、+86- 等），新系统会做严格校验。
- **T（目标）**：在不停机的前提下把数据迁过去，保证一致性 + 完整性 + 异常可回滚。
- **A（动作）**：1）旧系统 Java 服务暴露 gRPC（多表关联 + 枚举值转换聚合数据）→ Python 做格式清洗、批处理 → 新系统 gRPC 写入（走业务校验，避免直连脏数据）；2）批次内部数据库事务原子性，批次之间用 checkpoint + upsert 实现断点续传 + 幂等；3）三阶段切流：全量同步（记录 max updated_at 位点）→ 增量追平（WHERE updated_at > checkpoint）→ 短暂拦截写后切流；4）迁移前 pg_dump 快照备份兜底回滚；5）Redis 缓存读多写少数据 + Cache Aside；6）SQL 优化（联合索引、调 join buffer、避免 SELECT *）。
- **R（结果）**：完成迁移，遇到的主要问题是数据格式问题（手机号正则）通过迭代修正解决；事务回滚机制保证没造成脏数据。

## 高频追问 Q&A

### Q1: 数据迁移中如何保证一致性、完整性、可回滚？
**出处**：[[快手一面面经]] · [[阿里云消息团队一面面经]] · [[Shopee DB Infra 北京平台实习面试问题总结]]

**答题骨架**：
- 三阶段组织：迁移前（pg_dump 快照、checkpoint 机制、upsert 幂等）→ 迁移中 → 迁移后（count + 关键字段 hash 聚合对比）
- 一致性：每个批次在一个数据库事务里完成，要么整批 commit，要么整批 rollback，不会出现半批脏数据
- 传输层：gRPC 基于 HTTP/2 内置校验，关键数据再做 checksum / 记录数对比
- 完整性（断点续传）：每批 commit 成功后，checkpoint（最后一条 ID 或 updated_at）和数据写入放在同一个事务里持久化；失败时事务回滚，checkpoint 不推进，下次从该批次重试
- 完全回滚：迁移前已 pg_dump，整体性问题直接恢复快照，不靠覆盖式重刷（重刷本质是又一次写操作，可能引入新风险）

### Q2: 是停机迁移还是不停机？不停机怎么解？
**出处**：[[快手一面面经]] · [[阿里云消息团队一面面经]] · [[老虎国际一面面经（资讯组）]] · [[蚂蚁一面]]

**答题骨架**：
- 不停机方案分三阶段：
  - 全量同步：搬存量数据，记录同步开始位点（max updated_at）
  - 增量追平：从位点开始 `WHERE updated_at > checkpoint` 持续拉取变更，upsert 写入；监控同步延迟（当前时间 - 最新同步记录的 updated_at）降到秒级即追平
  - 流量切换：短暂拦截写请求几秒，等增量完全追平后切流，旧系统保留不下线，出问题随时切回
- 为什么用 updated_at 而非自增 ID：自增 ID 只能捕获 insert，无法感知 update；删除场景配合软删除（deleted_at）；更重的方案是 CDC 监听 WAL/binlog
- 业界主流方案（双写 + 灰度切流）：全量数据搬迁（按主键范围分块、加载期禁用索引/外键、并行 worker、每块跑 checksum）→ 双写（先旧后新、Kafka 重试队列、幂等 ID `INSERT ... ON CONFLICT DO NOTHING`）→ 影子读（线上暗测新库，捕获时区/NULL/排序规则差异）→ 灰度切读（预热缓冲池、Feature Flag 1%→5%→20%→100%，双写继续跑一周）→ 全程可观测性（同步延迟、影子读不匹配、缓存命中率、业务 KPI）
- 关键原则：幂等 / 分阶段而非大爆炸 / 先加载后建索引 / 影子读验证 / 预热再切换 / 监控 > 代码 / 全程可回滚

### Q3: 分批回滚 / checkpoint 怎么做？
**出处**：[[快手一面面经]]

**答题骨架**：
- 批次内部 — 数据库事务保证原子性，一批 500 条包在一个事务，任何一条出错整个 rollback
- 批次之间 — checkpoint 实现断点续传，checkpoint 更新和数据写入在同一个事务，保证两者原子性
- 失败时 checkpoint 不推进，下次从该批次重试，配合 upsert 幂等
```
checkpoint = load_checkpoint()
for batch in fetch_batches(start=checkpoint):
    BEGIN TRANSACTION
        upsert batch into target_db
        update checkpoint to batch.last_id
    COMMIT  # 成功推进 / 失败自动回滚 checkpoint 不变
```

### Q4: 增量同步用 updated_at 还是 CDC？
**出处**：[[快手一面面经]] · [[阿里云消息团队一面面经]] · [[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 自增 ID 只捕 insert，不能感知 update，所以不用
- updated_at 时间戳可以同时捕获 insert 和 update；删除用软删除（deleted_at）配合
- CDC（监听 WAL 逻辑解码 / binlog）是更重的方案，PG 没有 binlog 但有 WAL，可以用 Debezium 监听 logical decoding
- 人力系统数据量可控，时间戳方案足够；电商场景对积分/余额顺序敏感，需要 CDC + Kafka partition by userId 保证同一用户消息 FIFO（RabbitMQ 用 Consistent Hash Exchange 等效）

### Q5: gRPC 在迁移里的角色 / 为什么走业务接口而非直连数据库？
**出处**：[[阿里云消息团队一面面经]] · [[老虎国际一面面经（资讯组）]]

**答题骨架**：
- Python 作为中间层：从旧系统 gRPC 接口读聚合数据 → Python 做处理（手机号正则、类型转换、批处理）→ 通过新系统 gRPC 接口写入
- 旧系统数据不是简单 SELECT 能拿到的，需要 Java 业务逻辑做多表关联和枚举值转换
- 新系统通过自己的 gRPC 服务做字段合法性校验和关联关系检查后入库，避免绕过业务层导致脏数据
- 遇到的主要问题：数据格式（手机号格式不统一）+ 新系统校验失败，正则迭代修正

### Q6: 电商场景积分/点券增量同步对账怎么做？
**出处**：[[阿里云消息团队一面面经]]

**答题骨架**：
- 同步系统和对账系统分离，对账从业务维度校验
- 抽样数据对账：按主键取模或时间范围抽样，定时从新旧库各捞一批比对，不一致写差异表 + 告警
- 业务维度对账：用流水表，把某用户某段时间所有流水净变化额加总，和余额实际变化对比，能发现"流水写了但余额没变"或"顺序错乱中间状态异常"
- 积分幂等：不能用 UPSERT 覆盖余额，要用流水表记录每笔变更，以流水唯一 ID 做幂等
- 对账节奏：持续跑几天到一周差异率稳定为零才进入下一阶段切流；切完流后继续对账确认稳定才下线旧库
- 金融场景常专门搭对账系统，独立于同步系统

### Q7: 支付/对接银行的数据一致性
**出处**：[[阿里云消息团队一面面经]]

**答题骨架**：
- 分层组合：
  - 本地消息表（核心）：扣余额 + 记一条待支付消息在同一本地事务，再异步调银行；唯一支付流水号保证幂等
  - 独立对账系统（兜底）：每天/每小时拉银行账单和系统流水比对，三类差异（我们有银行没有 / 银行有我们没有 / 金额不一致），小额自动补偿，大额告警
  - TCC（多方资源协调）：Try 冻结 → Confirm → Cancel，避免半成功
  - 最大努力通知：银行回调 + 重试（1s/5s/30s/5min）
- 一句话：实时靠本地消息表 + 幂等重试，回调靠最大努力通知，多方靠 TCC，兜底靠对账。核心原则"宁可多查也不能漏一笔"

### Q8: Redis 缓存了什么？怎么保证一致性？
**出处**：[[快手一面面经]] · [[老虎国际一面面经（资讯组）]] · [[Shopee DB Infra 北京平台实习面试问题总结]]

**答题骨架**：
- 缓存读多写少、变更频率低的数据：组织架构与部门信息（部门层级、负责人）/ 岗位字典 / 招聘流程配置（面试轮次、审批链）/ 用户权限与角色信息
- 秋招高峰期读 QPS 很高，每次都打 PG 容易瓶颈，加缓存后 90% 请求直接从 Redis 返回
- 一致性策略：Cache Aside —— 写入时删除缓存，下次读取 miss 后从 DB 加载并回填，配合 TTL 兜底
- 双删策略（备选）：先删 DB 数据，再删 Redis 数据；PG 写入比 Redis 长，这个顺序大概率保证一致性
- 人力系统没有热点数据，简单策略足够，没遇到一致性问题

### Q9: 慢查询怎么优化？
**出处**：[[老虎国际一面面经（资讯组）]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[银河智学]] · [[Shopee DB Infra 北京平台实习面试问题总结]]

**答题骨架**：
- 三种识别方式：log_min_duration_statement 慢查询日志 / pg_stat_statements 聚合统计（avg_time 高 vs total_time 高）/ pg_stat_activity 实时监控
- 拿到慢 SQL 后 EXPLAIN (ANALYZE, BUFFERS) 看执行计划：最耗时节点、估算 vs 实际行数差距大说明统计不准、Sort Method: external merge → work_mem 不够
- 优化手段：
  - 索引：WHERE 条件高频列建组合索引（区分度强/等值条件优先放前面）；JOIN 列两侧都有索引；ORDER BY/GROUP BY 字段可以包含进去
  - SQL 写法：避免 SELECT *、避免索引列上做函数（用表达式索引）、拆复杂 SQL
  - 表结构：分区表归档历史数据
  - 配置：work_mem、shared_buffers 设物理内存 25%、autovacuum
- 实战案例：迁移后比对 SQL（全表 JOIN + SELECT *）→ 加联合索引 + 调大 join buffer + 走覆盖索引；HR 报表 N+1（1+50 次查询）→ 改写成一条 JOIN + 聚合 SQL，子表加联合索引（position_id, stage）

### Q10: JOIN 优化具体怎么做的？
**出处**：[[老虎国际一面面经（资讯组）]] · [[银河智学]]

**答题骨架**：
- 数据量小的表作驱动表：两个表按各自条件过滤后，参与 join 字段总数据量小的作驱动表
- 优化索引：针对 WHERE 建联合索引（注意顺序和范围，防止失效）；给被驱动表 join 字段加索引；覆盖索引避免回表
- 能用被驱动表索引就用，不能用就只能 Block Nested-Loop Join，应避免
- 调大 join_buffer_size：BNL 时 buffer 越大被驱动表磁盘扫描轮次越少

### Q11: 连接池报错 / 连接断开怎么处理？
**出处**：[[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 连接池耗尽（拿不到连接）：慢查询占着不释放 / 异常分支没 finally 关闭 / 连接池配置太小；先排查根因再调最大连接数和等待超时，并发大引入 PgBouncer 做连接复用
- 连接断开（拿到死连接）：网络抖动 / 数据库重启 / 防火墙超时；开启空闲检测（定期 SELECT 1 探活）+ 设置最大空闲时间 + 应用层重试
