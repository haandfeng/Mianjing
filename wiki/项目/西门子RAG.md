# 西门子RAG

## 一句话介绍
西门子 CT 机操作文档智能问答系统（学城），重构 ETL + 多路混合检索 + 重排，准确率 76% → 92%，嵌入吞吐 ×2。

## 项目级 STAR 自陈
- **S（场景）**：进组时已有基于 LangChain Chat 的基础 RAG（Elasticsearch + 单链路检索 → 喂给 LLM 回答），上游开源框架停止维护，效果不达标，无评测体系。
- **T（目标）**：搭评测流水线 → 重构 ETL/embedding 流水线 → 提升回答准确率到可上线水平。
- **A（动作）**：1）基于 LangSmith 搭测试流水线（200+ 中英文标准问答），跑出基线 76%；2）重写 ETL（XML→Markdown 表格嵌套处理、增量更新、多模型适配），换 BGE-M3 → 千问3，迁移 ES→Milvus 支持 4096 维；3）批量 + 异步 embedding 提升吞吐 ×2；4）Query Refine（多语言改写）+ 三路混合检索（千问3 稠密 + BGE-M3 稀疏 + BM25）+ Reranker（BGE-M3 / RRF）；5）按机型元数据 filter 隔离不同设备文档。
- **R（结果）**：准确率 76% → 92%，相关度/忠实度均 99%+，已上线学城，月访问几千次。

## 高频追问 Q&A

### Q1: RAG 整体优化链路怎么讲？76% → 92% 是怎么提升的？
**出处**：[[小红书产品工程师一面面经]] · [[小红书产品工程师二面面经]] · [[Bing图像视频业务实习一面面经]] · [[老虎国际一面面经（资讯组）]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[蔚来/蔚来一面RAG深挖|xhs]] · [[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- 评测发现问题（76%）→ 重构 ETL 流水线（80%+）→ Query Refine（解决中英文术语 gap）→ 多路混合检索（解决稀疏关键词稀释）→ Reranker → 92%
- 不是从零搭 RAG，是接手已有系统做评测体系搭建 + ETL pipeline 重构 + retrieval quality 优化
- 92% 是内部评测集（200+ 中英文标准问答）上的 correctness 分数；同时用 F1（约 0.8）评估「LLM judge 与人工评审一致性」

---

### Q2: 评测系统怎么做？怎么确保 LLM judge 可靠？
**出处**：[[小红书产品工程师一面面经]] · [[小红书产品工程师二面面经]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[Bing图像视频业务实习一面面经]] · [[蔚来/蔚来一面RAG深挖|xhs]]

**答题骨架**：
- 测试中心提供 200+ 问题（中文 180+、英文 90+）+ 标准答案
- 三个维度：correctness（与标准答案一致）、relevance（与问题相关）、faithfulness（忠于检索文档，无幻觉）
- LLM judge 用 DeepSeek V3（V3.1 不带思考）；每轮人工 double check 验证 judge 质量
- F1 ≈ 0.8 验证人工与 LLM 评判的一致性
- LangSmith 存储测试问答 + trace 整条链路（每节点入参/返回/耗时），失败 case 反查具体哪一段链路的问题
- 数据安全顾虑：LangSmith 走外网有出境风险，正式上线考虑字节扣子等内部部署方案

---

### Q3: Embedding 模型怎么选的？为什么从 BGE-M3 换千问3？
**出处**：[[小红书产品工程师一面面经]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[蔚来/蔚来一面RAG深挖|xhs]] · [[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- 定位瓶颈：跑完测试看具体哪些问题答不出来 → 发现核心问题是相关文档检索不到
- 典型 case：问题多两个字 embedding 偏移严重，BGE-M3 稠密向量在长文段里识别不到一两个关键词，多语言术语对齐能力差
- 选型依据：HuggingFace embedding 榜单，千问3 在 8B/32B 规模霸榜前三，国产、可本地部署
- 千问3 dense 4096 维 vs BGE-M3 dense 1024 维；ES 最大 1024 维放不下 → 迁移到 Milvus
- 最终上线：千问3（稠密语义） + BGE-M3（稀疏向量）双模型并用

---

### Q4: 为什么换向量库？ES vs Milvus
**出处**：[[Shopee DB Infra 北京平台实习面试问题总结]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[Bing图像视频业务实习一面面经]] · [[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- ES 最大 1024 维，千问3 是 4096 维放不下
- ES 不支持稀疏向量
- Milvus 是专业向量数据库，稠密 + 稀疏 + BM25 都能在同一 table 存
- Milvus 增量更新支持更好
- 索引选择：稠密用 HNSW（基线对比 FLAT 全量扫描），延迟降低 80%

---

### Q5: 混合检索怎么设计？为什么不只用稠密向量？
**出处**：[[小红书产品工程师一面面经]] · [[小红书产品工程师二面面经]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[老虎国际一面面经（资讯组）]] · [[Bing图像视频业务实习一面面经]] · [[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- 出发点：问题里只有两三个专业术语，在长文档里出现稀疏，纯稠密会被整篇语义稀释
- 三路：千问3 稠密向量（语义） + BGE-M3 稀疏向量（既有语义又有关键词权重） + BM25（纯关键词，IDF + 文档长度归一化）
- 都走 Milvus 一套表，按机型元数据 filter
- 没用更重的多向量方案：算力和工程成本高，混合检索性价比够

---

### Q6: Reranker 怎么选？大模型 rerank vs RRF
**出处**：[[小红书产品工程师一面面经]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- 测过两套：千问3 reranker（10 文档加 2 秒）、BGE-M3 reranker（30 文档 2 秒）、RRF 算法（毫秒级）
- 最终上线 BGE-M3 reranker：本地部署，30 取 10 容错率高
- RRF 缺点：把"只在某一路出现且排名靠后但实际是答案"的文档排到 10 名开外，准确率下降
- 大模型 rerank 准确率最好但耗时长，leader 不满意，所以也保留 RRF 选项

---

### Q7: Embedding 效率提升 2 倍怎么做的？
**出处**：[[快手一面面经]] · [[阿里云消息团队一面面经]] · [[小红书产品工程师二面面经]]

**答题骨架**：
- 原来串行：一个文档发服务器 → 等返回 → 下一个，同步阻塞
- 改为 batch + 异步：批量发文档做 embedding，异步拿结果分批落库
- 利用 Ollama 等部署框架自带的异步 API
- 增量更新：每篇文档唯一 ID，相同 ID 覆盖，支持单文档删除/更新，避免全量重建
- 文档量大时一次全量要 1-2 小时，全量主要发生在新机型/新数据表初始化，日常多为政策更新增量

---

### Q8: 持续高频涌入文档（钉钉知识库场景），怎么近实时不丢不漏？
**出处**：[[阿里云消息团队一面面经]]

**答题骨架**：
- 用消息队列把 ETL 各阶段解耦：Queue1（待解析）→ 解析 workers → Queue2（待嵌入）→ Embedding workers → Queue3（待落库）→ 落库 workers → Milvus
- Queue1 只放 `{doc_id, action, ts}`，worker 自己拉文档、转 Markdown、切片
- Queue2 用 prefetch_count 攒批 32 条 + 超时 200ms，批量推理
- 不丢：生产者 confirm + queue/message durable + 消费者手动 ACK + doc_id+chunk_index 幂等 + 死信队列兜底
- 近实时：监控每个 Queue 积压量，目标端到端秒级
- 入口瓶颈不仅 embedding 服务器，S3 拉文档 + 转 Markdown 同样需要扩容

### Q9: 文档分块（chunk）策略
**出处**：[[蔚来/蔚来一面RAG深挖|xhs]] · [[PingCAP 平凯星辰/面试问题总结]] · [[Shopee DB Infra 北京平台实习面试问题总结]]

**答题骨架**：
- 公司已切好（每个 chunk 约 500 token，按章节或章节小点切），没二次切分
- 常见策略：固定长度、语义分块（按句子/段落语义边界）、滑动窗口（10-20% 重叠）、层次化分块（按文档结构）、递归分块、按标记符切分
- 技术文档推荐：层次化 + 滑动窗口；代码库按函数/类切；通用场景递归分块
- 文档 XML 转 Markdown 时重点处理表格嵌套合并：Markdown 不支持表格合并，对合并的格用同样信息复制每行每格让 LLM 能正确理解

### Q10: 多语言（中英文术语 gap）怎么解决？
**出处**：[[小红书产品工程师一面面经]] · [[小红书产品工程师二面面经]] · [[Bing图像视频业务实习一面面经]] · [[老虎国际一面面经（资讯组）]]

**答题骨架**：
- 文档主体中文，用户习惯混用英文专业术语，embedding 模型对术语翻译/对齐能力有限，数据量也不够做专项微调
- 工程方案：让小模型对问题做改写，生成 5 个左右中英文不同表达版本，一起送去检索
- 代价 ≈ 1 秒延迟，但解决专业术语歧义性价比高

---

### Q11: 为什么用 RAG 而不是封装成工具让大模型选择性调用？
**出处**：[[蔚来CICD组AI agent实习生一面面经-详细版]]

**答题骨架**：
- 我们的 RAG 用了工具调用，但检索知识库这一步是写死必调的
- 选择性调用风险：模型可能"应该调但没调"产生幻觉
- 医疗行业准确性 > 效率，宁愿延迟高一点也要保证答案来源可追溯
- 实际效果：不相关问题模型会回答"无法从已有知识里获取相关回答"，可接受
- 温度设 0：医疗行业对准确性、幻觉、法律风险要求极高

---

### Q12: 自部署模型 vs 闭源 API？模型选型对比
**出处**：[[阿里云消息团队一面面经]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[Shopee DB Infra 北京平台实习面试问题总结]] · [[蔚来/蔚来一面RAG深挖|xhs]]

**答题骨架**：
- 数据是医院数据，公司规定开源模型自己部署
- 测过 LLaMA 70B、ChatGPT OSS 20B/120B、千问 36B、DeepSeek 671B、Gemini
- LLaMA / Gemini 中文表现差，无法理解中文专业术语
- 最终用 DeepSeek 671B：参数量最大，中英文都好，符合开源要求
- API 提供商对比：DeepSeek 官方/硅基流动稳定 + Structured Output 好；腾讯云不稳定

### Q13: 长上下文/多轮对话怎么处理？
**出处**：[[Bing图像视频业务实习一面面经]] · [[蔚来CICD组AI agent实习生一面面经-详细版]]

**答题骨架**：
- 简单 summary 压缩：达到 10 轮对话做一次 summary，作为新 prompt 上下文
- 单轮约 5000 token，10 轮约 5 万 token
- 我们当时优化目标主要是单轮问答质量，不是多轮会话能力
- 复杂 Agent 系统涉及长期记忆/短期记忆/大文档处理/TOKEN 超限/异常处理，是后续要补的

### Q15: 如果做外包付费版，怎么进一步优化？
**出处**：[[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- 架构层：Agent Loop + 自适应 Query Rewriting（检索→评估→改写→再检索的迭代循环）
- 检索层：Embedding 领域微调 + 优化 chunk 策略（语义/滑动窗口/层次化）+ GPU 加速多向量
- 生成层：上下文压缩 + LongContextReorder + CoT/ToT
- 动态 RAG：高质量问答入库做长期记忆，相似问题命中历史答案
- Graph RAG：实体关系建图，多跳推理

### Q16: 和 LangChain / LangGraph 的关系
**出处**：[[Bing图像视频业务实习一面面经]] · [[蔚来CICD组AI agent实习生一面面经-详细版]] · [[PingCAP 平凯星辰/面试问题总结]]

**答题骨架**：
- LangChain/LangGraph 帮做 workflow 状态管理和编排，底层数据库/SDK 调用很多时候自己直接调（封装文档不完善）
- LangChain：组件库 + 简单链式调用，适合快速搭建
- LangGraph：图编排 + 状态管理，支持长流程、可恢复、可回溯、human-in-the-loop
- 复杂生产系统通常自己控制底层实现
