## 一、项目架构

### 简历原文（来自 [[wiki/我的简历/黑马点评项目]]）

> 基于 **SpringBoot, SpringCloud** 等构建核心模块（商品、用户、订单等），通过 **Nacos** 实现服务注册与配置热加载。并使用 **Sentinel** 对服务进行请求限流、线程隔离、服务降级，保护整体服务可用性。
>
> 使用 **Redis** 实现登录功能，**ThreadLocal** 配合网关校验 token 并刷新，完成用户认证，通过拦截器清理 ThreadLocal。
>
> 基于 **Redis 和 Lua 脚本**保障用户"一人一单"，解决库存超卖问题；使用**乐观锁**提升并发写入性能。
>
> 使用 **RabbitMQ + Lua** 实现异步秒杀下单，**平均耗时由 400ms 降至 100ms**。基于 **RabbitMQ 死信队列**实现订单超时关闭。

### 整体架构图（口语版）

```
┌─────────┐    ┌─────────┐    ┌──────────┐
│  前端   │ →  │  网关   │ →  │ 用户服务 │ ──────┐
└─────────┘    └─────────┘    └──────────┘       │
                    │                            │
                    ├─→ 商品服务 ─→ Redis 缓存   │
                    │     ↓                      │
                    │   MySQL 主从               │
                    │                            │
                    └─→ 订单服务（秒杀核心）     │
                         │                       │
                         ├─→ Redis Lua 脚本      │
                         │   （扣库存 + 一人一单）│
                         │                       │
                         └─→ RabbitMQ ─→ 异步落库
                                          + 死信队列订单超时
```

技术栈：SpringBoot + SpringCloud + Nacos + Sentinel + Redis + Lua + RabbitMQ。

## 二、登录认证（Token + ThreadLocal + 双拦截器）

> 出处：[[小鹅通/小鹅通一面面经（带答案已OC）]]、[[其他/银河智学]]

### 验证码登录的处理逻辑

来自 [[专栏/后端/黑马点评常问的内容和黑马点评优化]]：

1. 校验手机号（正则）。
2. 把手机号作为 key，从 Redis 取验证码。
3. 校验验证码是否一致。
4. 一致则从数据库查询用户。
5. 用户不存在 → 自动注册新用户。
6. 用户存在 → 随机生成 UUID 作为 Token。
7. 把用户信息（脱敏后）以 Hash 形式存进 Redis，key = `LOGIN_USER:{token}`，TTL 30 分钟。
8. 返回 Token 给前端。

### 双拦截器逻辑

```
拦截器 1（拦截全部请求）：
    1. 取请求头中的 Token
    2. 用 Token 在 Redis 查用户
    3. 存在 → 把用户存到 ThreadLocal
    4. 刷新 Token 有效期
    5. afterCompletion 移除 ThreadLocal（防内存泄漏）

拦截器 2（只拦截需要登录的接口）：
    1. 判断 ThreadLocal 是否有用户
    2. 没有 → 返回 401
    3. 有 → 放行
```

为什么要两个拦截器？
- 拦截器 1 必须**全局都跑**，这样所有请求都能刷新 Token、把用户信息塞进 ThreadLocal。
- 拦截器 2 只在受保护接口上工作 → 把"是否需要登录"和"是否有 Token"的逻辑解耦。

### 为什么用 ThreadLocal

- Web 应用一个请求一个线程，把用户信息存 ThreadLocal 里 → 同一线程的 Service / Controller 都能取到，免去逐层透传。
- 不是每次都查 Redis → 减少 IO。
- ThreadLocalMap 的 key 是弱引用，**长生命周期线程未 remove 会导致 value 泄漏**，所以必须在 afterCompletion 里清。

### 拦截器三个核心方法

| 方法 | 时机 |
|---|---|
| `preHandle(request, response, handler)` | Controller 之前；返回 `false` 直接中断 |
| `postHandle(request, response, handler, modelAndView)` | Controller 之后、视图渲染前 |
| `afterCompletion(request, response, handler, ex)` | 整个请求完成（**包括异常**），用于清理 ThreadLocal |

### Token vs JWT

- 项目里用的是**不透明 Token + Redis 存储**：服务端可以随时让 Token 失效（删 key 即可）。
- JWT 自包含、无状态、不查后端 → 高扩展性，但**注销难**。
- 黑马点评选 Token + Redis 是因为登录态需要随时刷新 / 强制下线。

## 三、一人一单（Lua + 乐观锁）

> 出处：[[蚂蚁/蚂蚁一面]]、[[虾皮/Shopee DB Infra 北京平台实习面试问题总结]]、[[快手/支付方向/快手日常实习面经]]

### 问题

秒杀下单接口高并发时容易出现两个问题：
1. **超卖**：库存只剩 1，多个请求同时进来都判定 ≥ 0 → 都成功扣减 → 卖出去多个。
2. **一人多单**：同一用户连点两次都成功。

### Lua 脚本扣减库存

```lua
-- KEYS[1] = stock:{voucherId}
-- KEYS[2] = order:{voucherId}（用户集合）
-- ARGV[1] = userId

if (tonumber(redis.call('get', KEYS[1])) <= 0) then
    return 1   -- 库存不足
end
if (redis.call('sismember', KEYS[2], ARGV[1]) == 1) then
    return 2   -- 已下过单
end
redis.call('incrby', KEYS[1], -1)
redis.call('sadd', KEYS[2], ARGV[1])
return 0       -- 成功
```

### 为什么 Lua 是原子的

- Redis **单线程事件循环**，所有命令在同一线程顺序执行，单命令天然原子。
- Lua 脚本被视为一个完整命令；执行期间 Redis **不处理任何其他命令**，完成后才处理下一个。
- 不可中断 + 顺序执行 + 无并发冲突。

注意点：
- Lua 不能跑太久（< 1s），不能有阻塞操作（BLPOP / BRPOP 不能用）。
- 错误**不会回滚**已执行的命令（不是数据库事务）。

### 乐观锁补偿

数据库层落库时再做一次乐观锁：

```sql
UPDATE seckill_voucher
SET    stock = stock - 1
WHERE  voucher_id = #{id}
  AND  stock > 0;            -- 不需要 stock = #{原值}，直接判断 > 0 即可
```

- `WHERE stock > 0` 就是隐式的乐观锁条件，避免 ABA 问题。
- 失败的事务返回 0，前端给"已售完"。

### 也可以用 Redis 分布式锁

来自 [[阿里云/阿里云消息团队一面面经]]：

```
SET lock:order:userId:goodsId  threadId  NX  EX 30
```

- `SETNX` 写入锁 ID + 进程/线程 ID + TTL 逻辑过期。
- 解锁前判断存储的线程 ID 跟自己一致才能解（不是别人的锁）。
- 看门狗：每 10 秒后台判断任务是否完成，未完成续期到 30 秒，避免业务没跑完锁过期。

## 四、库存超卖（乐观锁）

详细方案见 [[Redis]] 缓存一节，本章只列要点：

1. **Redis 预扣 + Lua 原子检查**（前文）。
2. **数据库 UPDATE 乐观锁**（前文）：`WHERE stock > 0`。
3. **库存预热**：上架时把库存写入 Redis，避免冷启动击穿。
4. **库存回滚**（订单超时未付款）：通过死信队列触发，Lua 脚本 `INCR stock + SREM userSet`。

## 五、异步秒杀（RabbitMQ + 死信队列）

> 出处：[[阿里云/阿里云消息团队一面面经]]、[[小鹅通/小鹅通一面面经（带答案已OC）]]

### 改造前 vs 改造后

```
[改造前]                         [改造后]
请求 → 扣库存(Redis)              请求 → Lua 扣库存(Redis)
     → 落库(MySQL)同步        →        → 发消息(RabbitMQ) 立即返回
     → 返回响应（~ 400ms）            消费者异步落库（用户感知 ~ 100ms）
```

主要省下的是 MySQL 同步落库的 IO 时间，以及连接池等待。

### 不丢不重三个环节

| 环节 | 措施 |
|---|---|
| 生产者 | Confirm 机制（消息到 Broker 持久化后回 ACK，超时重发）、`mandatory` 参数（路由失败 ReturnCallback）、本地消息表（扣库存 + 写消息表在一个本地事务，定时扫"待发送"重发） |
| Broker / 队列 | Exchange + Queue + 消息（`deliveryMode = 2`）三层 `durable = true`；可靠性更高用 Quorum Queue（Raft 共识） |
| 消费者 | `autoAck = false` 手动 ACK；`basicNack` 区分 `requeue=true`（DB 临时连不上，重试）和 `requeue=false`（数据格式错误，进死信队列） |

### 至少一次 + 幂等 = 等效一次

- 全链路是 **at-least-once** 投递。
- 消费者用订单 ID 唯一键 + `INSERT ... ON CONFLICT DO NOTHING` 做幂等 → 等效 **effectively once**。

### 死信队列三种触发条件（DLX）

1. 消息被拒绝（`basicReject` / `basicNack` 且 `requeue=false`）。
2. 消息 TTL 过期（`x-message-ttl`）。
3. 队列长度超限。

### 订单超时关闭就是用 TTL + DLX 实现

```
订单创建 → 发消息到 order.queue（设 TTL = 30 分钟）
        → TTL 到期未消费 → 转入 dlx.queue
        → 死信消费者读取，回滚库存 + 关闭订单
```

### 队列积压怎么办

来自 [[阿里云/阿里云消息团队一面面经]]、[[小鹅通/小鹅通一面面经（带答案已OC）]]：

- 生产者限流（令牌桶 / RateLimiter）。
- 消费者扩容（增加并发数、批量消费）。
- 降级（丢弃低优先级、TTL 过期）。
- 同一用户消息保序：Kafka 用 userId 做 partition key；RabbitMQ 用 Consistent Hash Exchange（多个 Queue + routing key hash 路由 + 每 Queue 一个消费者）。

## 六、Sentinel 限流 / 熔断 / 降级

来自简历："使用 Sentinel 对服务进行请求限流、线程隔离、服务降级"。

### Sentinel 的核心概念

- **Resource**：一段被保护的代码或接口。
- **Rule**：流控、熔断、热点参数、系统保护、授权。
- **Slot Chain**：每次进入 resource 时跑过的责任链：NodeSelector → ClusterBuilder → StatisticSlot → SystemSlot → AuthoritySlot → FlowSlot → DegradeSlot。

### 三种保护策略

| 类型 | 触发 | 行为 |
|---|---|---|
| **流控（Flow）** | QPS / 线程数超阈值 | 直接拒绝、Warm Up 启动、匀速排队 |
| **熔断（Degrade）** | 异常比例 / 异常数 / 平均 RT 超阈值 | 在时间窗口内**快速失败**，半开探活 |
| **系统保护（System）** | 整机 Load / CPU / RT 超阈值 | 自适应保护 |

### 在黑马点评中怎么用

- 秒杀接口加 QPS 流控（按商品 ID 做热点参数限流）。
- 商品详情查询失败时降级为返回静态默认值。
- 第三方接口（短信验证码）熔断 → 告警 + 降级到备份通道。

### 与 Hystrix 的区别

- Sentinel：Java + Dashboard 可视化，规则可推送热更新；轻量，覆盖**流控 / 熔断 / 系统保护 / 热点参数**。
- Hystrix：已停止维护；线程池隔离开销大。

## 七、100w QPS Redis 分库 / 集群

> 出处：[[阿里云/阿里云消息团队一面面经]]、[[虾皮/Shopee DB Infra 北京平台实习面试问题总结]]

### 单机 Redis 撑不住 100w QPS 怎么办

#### 1. 缓存击穿 / 穿透 / 雪崩

- **雪崩**：大批 key 同时过期 → DB 被打爆。
  - TTL 加随机数（基础 TTL + 随机抖动）。
  - 多级缓存（本地 Caffeine + Redis）。
  - 热点 key 永不过期 + 后台异步刷新。
- **击穿**：单个热 key 过期，瞬间大量请求穿到 DB。
  - **逻辑过期**：永不真正过期，过期后**单线程**异步更新，其他读旧数据。
  - **互斥锁**：`SETNX` 拿锁查 DB 回填。
- **穿透**：查询根本不存在的数据，DB 反复查。
  - 布隆过滤器（前置过滤）。
  - 缓存空对象（短 TTL）。

#### 2. 分片 / 集群

- **客户端分片**（Jedis Sharding 已淘汰）。
- **代理分片**（Twemproxy / Codis）：客户端无感，单点风险。
- **Redis Cluster**：16384 个 slot 分片到节点；客户端缓存 slot map；MOVED / ASK 重定向。

#### 3. 多级缓存

- L1：本地 Caffeine（堆内、ms 级）。
- L2：Redis 集群。
- L3：数据库 + 物化视图。

#### 4. 异步预热

- 上架时把热点商品数据预加载到 Redis。
- 热度感知：实时计算 TopN 商品，提前预热到本地缓存。

### 分布式锁完整流程

```
SET lock:seckill:{voucherId} {threadId} NX EX 30

  → 拿到锁 → 业务逻辑（带看门狗续期）→ Lua 释放锁
  → 没拿到锁 → 短轮询重试 / 直接降级失败
```

释放锁的 Lua（防止误删）：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

## 八、监控告警

### 链路监控

- **SkyWalking / Zipkin**：分布式 trace，定位慢请求。
- **Prometheus + Grafana**：指标监控（QPS、延迟、错误率、JVM、Redis、MySQL）。
- **ELK / Loki**：日志采集与告警。

### 关键监控点

| 模块 | 指标 |
|---|---|
| 网关 | QPS / P99 / 4xx / 5xx |
| 业务服务 | JVM（堆、线程、GC）、慢接口、异常率 |
| Redis | 内存、命中率、慢查询、连接数 |
| MySQL | 连接数、慢日志、主从延迟 |
| RabbitMQ | 队列深度、消费速率、未确认数 |

### 告警分级

- P0（电话）：服务宕机、库存超卖、订单丢失。
- P1（IM）：CPU / 内存 / 错误率超阈值。
- P2（邮件）：慢查询、积压。

## 九、用户视角的三个边界场景（[[小鹅通/小鹅通一面面经（带答案已OC）]]）

### Q：用户下单后看不到自己的订单怎么办？

- MQ 的 ACK 机制确保每个订单都会被处理。
- Redis 先存"订单占位"：下单成功后写 `orderId = "处理中"`。
- 用户查询：先查 Redis 返回"正在处理"；后台落库后再次查询能查到订单详情。
- 流程：先查 Redis pending → Redis 没有再查 DB → DB 也没有就是失败。

### Q：用户拿到优惠券但下单失败 / 库存怎么回滚？

- 下单失败发"回滚消息"到 MQ，消息体含 userId、券 ID、操作类型、时间戳。
- 库存回滚（Redis）：`INCR stock + SREM userSet`，整个用 Lua 包装保证原子。
- 幂等：每个回滚操作生成唯一操作 ID（订单 ID + retry ID），Redis 设幂等 key 避免重复归还。
- MQ 重试 + 死信队列保最终一致。

### Q：扣库存成功但发消息失败 / 消费失败怎么处理？

见上文「不丢不重三个环节」+「至少一次 + 幂等 = 等效一次」。

## 十、口语化总结

> 黑马点评是个仿大众点评的项目，重点是秒杀场景。
>
> **登录**用手机号验证码 + Redis 存 Token + ThreadLocal 透传用户，两个拦截器，一个全局做 Token 校验、一个保护私有接口。
>
> **秒杀**核心是 Redis Lua 脚本：单线程原子地做"判库存 → 判一人一单 → 扣库存 → 标记用户"，扣完发消息到 RabbitMQ 异步落库，把响应从 400ms 降到 100ms。
>
> **不超卖 / 一人一单**靠 Lua 原子 + DB 乐观锁双重保险；**异步可靠性**靠生产者 confirm + queue durable + 消费者手动 ACK + 本地消息表 + 死信队列；**订单超时关闭**用 TTL + DLX 实现。
>
> **保护层**用 Sentinel 做流控、熔断、降级；缓存击穿/穿透/雪崩用 TTL 抖动、互斥锁 / 逻辑过期、布隆过滤器 / 空对象。
>
> 需要扛 100w QPS 时，靠 Redis Cluster 分片 + 本地多级缓存 + 热点预热。
