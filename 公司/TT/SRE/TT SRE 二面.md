# 输入一个URL，到界面显示，这中间发生了什么，越深入越好

## 准备和大包

解析URL(http/https, 域名)-> 根据域名获取IP地址（查缓存，浏览器，系统，路由器，DNS）-> 通过TCP发起连接-> HTTPS握手 -> 构造请求（Http报文-> TCP Head(端口号地址，切断) -> IP Head（IP地址）-> MAC(head， arp)  ）
1. 先解析URL地址，域名还是查询，提取协议，HTTP还是HTTPS，路径是什么
例子：https://www.xxx.com/path/to/page?x=1

| **部分**       | **含义**        | **浏览器要用来干嘛**        |
| ------------ | ------------- | ------------------- |
| 协议（Protocol） | https         | 决定用哪套通信规则（是否加密）     |
| 域名（Domain）   | www.xxx.com   | 用来找服务器（后面会做 DNS 解析） |
| 路径（Path）     | /path/to/page | 告诉服务器你要哪个资源         |
| 参数（Query，可选） | ?x=1          | 给服务器的附加信息           |

2. 获取IP地址。它会按顺序查：浏览器缓存->系统缓存（/etc/hosts）->路由器缓存。如果都没命中，就向 DNS 服务器发起查询，最终拿到目标 IP。
3. TCP三次握手
4. 如果是 HTTPS，在TCP 握手之后，还要进行 TLS四次握手，协商加密密钥，确保后续数据传输安全。
5. 构造 HTTP 请求。连接建好了，浏览器把用户的请求组装成一个 HTTP 报文（应用层数据）。
6. 协议栈封装。数据包要出门，得穿上几层“衣服”：
	1. ﻿﻿﻿传输层（TCP）：给 HTTP 数据加上TCP 头（源端口、目的端口、序列号），把大块数据切分成小段（Segment）。
	2. ﻿﻿﻿网络层（IP）：加上IP头（源IP、目的IP），这一层决定了数据包怎么在互联网上路由。
	3. ﻿﻿﻿数据链路层（MAC）：加上 MAC 头（源 MAC、目的 MAC）。
	4. ﻿﻿﻿补充细节（ARP）：这里有个问题，浏览器怎么知道下一跳（通常是路由器）的 MAC地址？这就需要ARP协议，通过IP 查到网关的MAC地址，填进去才能发出去。
## 网络传输（路上的事）

网卡 -> 交换机 ->路由器转发（看IP地址决定下一跳去哪，更换MAC地址）


1）网卡发送封装好的二进制数据，被网卡转化为电信号或光信号，发到网线或空气（Wi-Fi）中。
2）交换机转发（二层）数据先到交换机。交换机很单纯，只看 MAC地址。它查 MAC 表，把数据从对应的端口转发给路由器
3）路由器转发（三层）数据出了局域网，到了路由器。路由器看IP地址。它查路由表，决定下一跳去哪。

注意：路由器会把旧的MAC 头剥掉，换上自己的 MAC 作为源，下一跳设备的 MAC 作为目的，然后把包扔给下一个节点。同时，IP 头里的 TTL（生存时间）会减1

## 第三阶段：处理与响应（服务器端）

1. 服务器接收与解包。数据包历经干辛万苦到达服务器。服务器网卡收到后，操作系统开始一层层脱衣服（解封装）：去掉MAC 头-> 去掉IP 头-> 去掉 TCP 头，最后把纯净的 HTTP 数据交给监听 80（http）或443（https）端口的Web 服务器进程（如Nginx) 
2. 处理与响应。服务器处理请求（读数据库、运行逻辑），生成 HTTP 响应报文。这个响应报文会经历同样的封装过程，原路返回给浏览器。
3. 浏览器渲染。浏览器收到 HTML代码，开始解析 DOM 树，下载css和JS，最后把漂亮的网页渲染出来展示给用户。



# DNS的解析流程，怎么获取DNS的，怎么递归查询的，举个例子


- ﻿根DNS服务器。根 DNS服务器提供 TLD 服务器的IP地址。目前世界上只有13 组根服务器，我国境内目前仍没有根服务器。
- ﻿﻿顶级域 DNS服务器（TLD 服务器）。顶级域是指域名的后缀，如com、org、net 和edu 等。国家也有自己的顶级域，如uk、fr 和 ca。TLD服务器提供了权威 DNS 服务器的IP地址。
- ﻿权威 DNS服务器。在因特网上具有公共可访问主机的每个组织机构必须提供公共可访问的 DNS记录，这些记录将这些主机的名字映射为IP地址。
- ﻿﻿本地 DNS服务器。每个 ISP（互联网服务提供商）都有一个自己的本地 DNS服务器。当主机发出 DNS请求时，该请求被发往本地 DNS服务器，它起着代理的作用，并将该请求转发到 DNS层次结构中。严格说来，不属于 DNS 层级结构。



用户访问www.mianshiya.com 时，域名解析是这么走的：

1）浏览器先查本地缓存，有就直接用，没有就往下走
2） 操作系统查本地 hosts 文件和系统 DNS缓存
3）还是没有，就问本地DNS 服务器。这个服务器一般是运营商提供的，或者用户自己配的 8.8.8.8、114.114.114.114 这些公共DNS
4） 本地 DNS 服务器也没缓存的话，就开始==迭代查询==。先问根域名服务器，根服务器说"我不知道，但.com 的事你去问这个地址"
5） 本地 DNS 再去问.com 的顶级域服务器，TLD 服务器说"mianshiya.com 的权威服务器在这个地址"
6） 本地 DNS 最后去问权威 DNS 服务器，权威服务器手里有 www.mianshiya.com 的完整记录，返回IP地址
7）本地 DNS把结果缓存下来，然后返回给浏览器，浏览器也缓存一份整个过程涉及多级缓存，绝大多数请求在前几步就命中缓存了，真正走完全流程的不多。

DNS 查询类型：
- ﻿递归查询：客户端将域名查询请求发送给 DNS 服务器，由 DNS 服务器负责完成整个查询过程，最终返回结果给客户端。
- ﻿迭代查询：DNS 服务器向其他 DNS 服务器查询，当一个 DNS 服务器无法回答时，会返回下一步应该查询哪个 DNS 服务器的信息，再由客户端进行下一步查询。

提问：如果服务器响应很慢，怎么判断是网络问题还是服务端问题？

回答：先用 ping 或 traceroute 看网络延迟和丢包，如果网络正常，那大概率是服务端问题。也可以看浏览器开发者工具的Timing,TTFB 特别长说明服务端处理慢，Content Download 长说明响应体大或者带宽不够。服务端排查就要看日志、监控，看是慢查询、锁等待还是资源耗尽。
