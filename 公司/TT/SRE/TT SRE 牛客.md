# 2. 假如现在外部访问服务的url出现了502该怎么去排查  
下面把你贴出来的“两篇文章”的排查方法，融合成一套**可复制的 502 排查 SOP**（既能快速止血，也能定位根因并形成预防闭环）。我会按“先定位、再验证、再深挖连接层、最后修复/预防”的结构总结。

---

## **一、两篇文章的共同核心观点（你需要记住的3件事）**

1. **502=网关（Nginx）拿不到上游有效响应**
    
    不代表一定是业务报错，更多时候是“连接/超时/队列/配置”问题。
    
2. **排查起点永远是 Nginx error.log 的 upstream 报错行**
    
    那一行通常直接告诉你：是 connect refused、timed out、prematurely closed、还是 no live upstreams。
    
3. **“服务看起来正常”不等于“能接住流量”**
    
    第二篇的案例就是：后端进程活着、健康检查 200，但**accept 队列满**导致新连接进不来，Nginx 只能超时并返回 502。
    

---

## **二、统一成一个方法：502 排查 6 步法（从快到慢）**

  

### **Step 1：先止血 + 先确认“谁返回的502”**

- 用 curl -v 看响应头 server、是否 Nginx/Ingress/网关返回。
    
- 同时**确认影响面**：全量还是部分、只在高峰还是一直发生。
    

  

> 目的：避免一上来就盲查后端代码，先把边界锁定在“入口层 ↔ 上游”。

---

### **Step 2：第一优先级看 Nginx 错误日志，给问题“归类”**

  

文章都强调：先看 error.log 的 upstream 报错。

```
tail -n 50 /var/log/nginx/error.log | grep -i upstream
# 或实时
tail -f /var/log/nginx/error.log
```

把错误归为 4 类（非常实用）：

- **connect() failed / connection refused**：上游端口没人监听/服务挂/防火墙拦
    
- **upstream timed out**：连接慢或读慢（上游慢、队列满、网络问题）
    
- **upstream prematurely closed**：上游主动断开/崩溃/响应写到一半断了
    
- **no live upstreams**：upstream 全挂或健康检查判定全不可用
    

  

> 目的：把排查从“散弹枪”变成“狙击枪”。

---

### **Step 3：验证上游是否真的“可用”（活着 + 能直连 + 响应正常）**

  

你贴的第一篇给的是“最标准三件套”：

```
ss -tlnp | grep 8080        # 端口是否监听
ps aux | grep <进程名>       # 进程是否存在
curl -v http://127.0.0.1:8080/health  # 绕过nginx直连
# 可选：检查最近日志
journalctl -u myapp.service --since "5 minutes ago"
```

判断：

- **直连都不通**：上游服务/端口/网络就是根因
    
- **直连 200，但经 Nginx 502**：问题更可能在 Nginx 配置、连接层、队列、超时策略
    

---

### **Step 4：如果“直连正常但仍502”，立即查连接数与状态分布（定位连接层问题）**

  

第二篇的核心亮点就在这里：通过 ss 看出 **SYN_SENT 卡住**。

```
ss -ant | grep 8080 | wc -l
ss -ant | grep 8080 | awk '{print $1}' | sort | uniq -c
```

如何读：

- **SYN_SENT / SYN_RECV 多**：新连接建不上（网络/队列/backlog/防火墙）
    
- **ESTABLISHED 很多且持续不降**：连接堆积（上游处理不过来、连接没释放、线程池/连接池耗尽）
    
- **TIME_WAIT 爆炸**：短连接太多（缺少 keepalive、复用差，导致资源压力）
    

  

> 目的：确认你面对的是“业务慢”还是“连接层拥塞”。

---

### **Step 5：进一步锁根因：看 accept 队列是否满（文章2的关键）**

  

这是最容易被忽略但杀伤力很强的一步：

```
ss -lnt | grep 8080
```

看输出里的 Recv-Q 和 Send-Q：

- Recv-Q 接近/超过 Send-Q（上限）≈ **accept 队列满了**
    
- 队列满 → 新连接进不来 → Nginx 连接超时/读头超时 → 502
    

  

文章2还给了根因链路：

- Spring Boot/Tomcat accept-count 默认偏小
    
- 系统 net.core.somaxconn 默认 128
    
- 实际队列取二者较小 → 高峰时直接爆
    

---

### **Step 6：对症修复（按“短期止血 → 长期治理”）**

  

两篇文章的修复建议可以合并成一套“组合拳”：

  

#### **A. 配置事故类（文章1的教训）：**

#### **不要把超时改得离谱**

- 把 proxy_connect_timeout 调到 500s 会让“快速失败”机制失效，连接卡死堆积拖垮集群
    
    ✅ 正确做法：合理超时 + 重试 + 快速失败
    

  

示例（文章1风格）：

- proxy_connect_timeout 5s
    
- proxy_read_timeout 30s
    
- proxy_next_upstream ... + 限制重试次数
    

  

#### **B. 队列/连接层类（文章2根因）：**

#### **调大 backlog + 减少新建连接**

- 系统参数：net.core.somaxconn、tcp_max_syn_backlog
    
- 应用参数：Tomcat accept-count、max-connections、线程数
    
- Nginx：启用 upstream keepalive，减少频繁建连
    

  

#### **C. 资源耗尽类（文章1覆盖）：**

#### **查句柄/连接池/CPU内存**

- ulimit -n
    
- ss -s
    
- htop
    
- DB/Redis 连接池耗尽导致上游慢 → Nginx 超时 502（文章1的复盘案例）

##  jwt的组成结构是什么样的，以及整个生成和验证的流程是怎么样的  

用户登录成功后，服务端生成 JWT 并返回给客户端。客户端将该 Token 存储起来，并在后续请求中通过 Authorization 头发送给服务端。服务端收到后验证签名，若签名有效，则信任其中的用户信息，无需查询数据库，实现无状态认证。


一个 JWT 长这样：xxxxx.yyyyyy.zzzzzz，三部分用点号连接。
第一部分是 Header，存的是元数据，说明这是个 JWT、用的什么签名算法。
第二部分是 Payload，存的是声明信息，包括用户ID、过期时间这些数据。
第三部分是 Signature，把 Header 和 Payload 拼起来用密钥签名，用来验证令牌没被篡改。

## 4. redis做点赞缓存如何避免产生数据不一致的问题  

==先删除再修改的话，因为写Mysql比较久，可能会出现删了，写的过程中来了一个新的数据，会对Mysql读，读了之后之后，Mysql再更新，缓存和数据库会出现不一致==

另外的不一致只会发生在，更新数据库之前读，没读到，要写redis，然后再更新数据库之后+删除redis之后，才写回去，才会造成不一致。


1. **为什么是先更新MySQL，后删除Redis缓存？会不会有数据不一致的问题**

2. 首先更新MySQL数据 —— 删除Redis缓存，中间这一段时间当然是数据不一致的，但是因为是同步删除，间隔时间极短，几乎忽略不计。
3. 另外当高并发情况下，可能存在因为并发读写导致数据不一致的情况，分析如下

4. 读请求：缓存未命中，读取数据库的值为20
5. 写请求：更新数据库的值为21，删除缓存
6. 读请求：将20写回缓存

7. **但上述情况出现的概率很小**：因为 更新MySQL的耗时 >> 写入Redis缓存的耗时，正常流程下不应该在经过了写请求之后，读请求才把旧值写回缓存，当并发量不大的情况下，几乎不会发生

8. 但如果先删除Redis缓存，后更新MySQL的数据，虽然**理论上**避免了一小段时间的数据不一致，但是在并发读写的场景下又会**出现数据不一致的概率会大很多**，分析如下

9. 写请求：删除缓存
10. 读请求：缓存未命中、读取数据库的值，更新缓存值为20
11. 写请求：更新数据库的值为21

12. 因为更新MySQL的耗时 >> 删除Redis缓存的耗时，所以很有可能会有并发读请求在"删除缓存"和"更新数据库" 之间执行，导致数据不一致。

13. **为什么是删除Redis缓存，而不是更新？**

14. 删除缓存更简单耗时更短，而更新缓存如果涉及到复杂的数据聚合，则耗时很长
15. 如果在写写并发的场景下更新Redis缓存，可能会出现数据不一致的问题

  

16. **如果删除Redis缓存失败了怎么办？**

17. 依赖TTL到期自动删除缓存
18. 两种改造方法：

19. 消息队列重试
20. 订阅BinLog + 消息队列重试=

## 你是一个站点的管理员，现在用户访问你这个站点特别慢，该怎么去排查  ,磁盘IO

网络连接(DNS解析速度，网络连接ping), CPU内存占用率(top，htop)

需要从最基础的网络连接开始检查。网站访问速度的基础是网络连接质量，包括 DNS 解析速度、网络带宽、网络稳定性等。如果 DNS 解析速度过慢，网站在用户访问时就会产生延迟，影响整体的访问体验。可以使用 dig 命令检查 DNS 解析时间，确认是否存在解析延迟。如果发现 DNS 解析速度过慢，可以考虑更换更高效的 DNS 服务提供商，或者配置本地缓存 DNS 服务器。

同时，测试网络连接的带宽和延迟也是很重要的一步。可以使用 ping 命令检查服务器的响应时间，确认是否存在丢包或超时现象。通过这种方式，可以初步判断网络质量是否存在问题。如果网络质量不佳，可能需要与网络服务商联系，检查是否有带宽限制或链路不稳定的问题。如果服务器与用户之间的网络路径较长，或者涉及跨国访问，还需要考虑使用  CDN(内容分发网络)来加速网站访问，减少跨境延迟。
 
接下来，如果确认网络连接没有问题，应该检查服务器的性能。服务器性能问题通常体现在 CPU 使用率、内存占用、磁盘 I/O、网络带宽等方面。通过命令如 top 或 htop 可以查看 CPU 和内存的使用情况： 

如果 CPU 使用率过高，可能是因为服务器处理的请求过多，或者某些进程占用了过多的 CPU 资源。此时需要进一步分析高负载进程，优化不必要的后台进程或调整请求的负载均衡策略。如果内存使用率过高，可能是由于应用程序内存泄漏、缓存未能及时清理，或者应用处理的数据量超出了服务器的内存容量。检查并优化应用程序的内存管理，或者考虑扩展服务器内存，是解决该问题的有效手段。

磁盘 I/O 性能差也是导致服务器响应慢的常见问题之一。如果磁盘 I/O 性能不足，应用读取和写入数据时会产生延迟，导致用户访问时出现明显的卡顿或慢响应。可以通过 iostat 或 iotop 命令检查磁盘 I/O 性能，确认是否存在磁盘瓶颈。如果发现磁盘使用率过高，可以考虑优化磁盘 I/O，例如使用 SSD 替代机械硬盘、优化数据库索引或查询、增加 RAID 磁盘阵列来提升 I/O 性能。如果服务器存储过于分散，可能还需要重新配置存储架构，合并分散的存储资源，减少磁盘访问的路径和时间。

当网络和硬件资源没有问题时，接下来的排查重点应该转向服务器的 Web 应用和数据库。应用层的优化是提高网站性能的关键环节之一。许多网站的访问慢是由于 Web 服务器处理请求的效率低，或者应用程序代码执行不够高效。首先，可以检查 Web 服务器(如Nginx的配置，确认是否开启了请求缓存、压缩、HTTP/2 等性能优化功能。

数据库性能往往是影响网站速度的瓶颈之一。数据库查询慢，或者数据库连接数过多时，往往会导致页面加载时间显著增加。首先需要检查数据库的性能指标，使用 SHOW STATUS 或 EXPLAIN 命令检查是否存在慢查询。慢查询日志可以帮助你找出那些性能差的 SQL 语句，优化查询或添加索引是常见的解决方法。例如，确保常用的查询字段有索引，并尽量避免全表扫描。

如果数据库连接数过多，可能是由于数据库连接池配置不合理或应用程序未能合理复用数据库连接。此时，优化数据库连接池的配置，确保连接的高效复用，避免频繁的连接和断开操作。

除了数据库查询，缓存机制也是提高网站响应速度的重要手段。对于频繁访问的数据，使用缓存可以显著减少数据库查询的次数，提升网站响应速度。常见的缓存技术包括  Memcached、Redis等。在应用层增加缓存层，可以通过减少不必要的数据库访问来显著提高响应速度。如果没有使用缓存，建议根据业务场景配置合适的缓存策略。例如，页面缓存、数据库查询缓存、API 响应缓存等。

在排查和优化完服务器配置后，另一个可能影响访问速度的因素是应用程序代码质量。代码中的冗余操作、无效循环、不必要的请求、重复的数据库查询等都会导致服务器负载过高，从而使得网站响应慢。定期进行代码审查，发现并优化性能瓶颈是非常重要的。常见的优化策略包括：减少 HTTP 请求的数量、精简 JavaScript 和 CSS 文件、延迟加载非核心资源、使用图片压缩等。

### cpu load是什么概念，怎么能看到load  


在 Linux 里大家说的 **CPU load（系统负载）**，通常指 **load average**：

它表示在一段时间内，系统里平均有多少“活儿”处于 **需要 CPU 或者被 I/O 卡住但不可中断** 的状态。

更具体一点：load 统计的主要是两类任务数量（取平均）：

1. **正在运行**或**准备运行（run queue）**的任务（R）——“在排队等 CPU”
    
2. **不可中断睡眠（D）**的任务——通常是在等磁盘/存储 I/O（“卡在 I/O 上，CPU也干不了下一步”）

13. 讲一讲零拷贝  

零拷贝（Zero-Copy）是一种高效的数据传输技术，它可以将数据从内核空间直接传输到应用程序的内存空间中，避免了不必要的数据拷贝，从而提高了数据传输的效率和性能。

在传统的数据传输方式中，当应用程序需要从磁盘、网络等外部设备中读取数据时，操作系统需要先将数据从外部设备拷贝到内核空间的缓冲区，然后再将数据从内核空间拷贝到应用程序的内存空间中，这个过程中需要进行两次数据拷贝，浪费了大量的 CPU 时间和内存带宽。

而使用零拷贝技术，数据可以直接从外部设备复制到应用程序的内存空间中，避免了中间的内核空间缓冲区，减少了不必要的数据拷贝，提高了数据传输的效率和性能。

在网络编程中，零拷贝技术可以用于大文件的传输、网络文件系统的读写、数据库查询等场景中，提高数据传输的效率和响应速度。同时，零拷贝技术也可以减少系统内存的开销，提高系统的稳定性和可靠性。

简单来说 原本是 外部空间 -> 内核空间 -> 用户空间
		现在是  外部空间->用户空间（内核空间）

以“把文件发到网络上（HTTP 静态文件）”为例，传统路径常见是：

1. 磁盘 → **内核页缓存**（DMA）
2. 内核页缓存 → **用户态 buffer**（一次拷贝）
3. 用户态 buffer → **内核 socket buffer**（一次拷贝）
4. 内核 socket buffer → 网卡（DMA）

这里的浪费主要在：

- **两次用户态参与的 memcpy**
- **频繁的用户态/内核态切换（syscall）**

- 大量 CPU 时间花在“搬运数据”而不是业务逻辑

`mmap`是将读缓冲区的地址和用户缓冲区的地址进行映射，内核缓冲区和应用缓冲区共享，所以节省了一次CPU拷贝‘’并且用户进程内存是**虚拟的**，只是**映射**到内核的读缓冲区，可以节省一半的内存空间。


sendfile表示在两个文件描述符之间传输数据，它是在**操作系统内核**中操作的，**避免了数据从内核缓冲区和用户缓冲区之间的拷贝操作**，因此可以使用它来实现零拷贝。

可以发现，`sendfile`实现的零拷贝，I/O发生了**2**次用户空间与内核空间的上下文切换，以及3次数据拷贝。其中3次数据拷贝中，包括了**2次DMA拷贝和1次CPU拷贝**。那能不能把CPU拷贝的次数减少到0次呢？有的，即`带有DMA收集拷贝功能的sendfile`！

少了两次状态切换，这主要是源于直接把磁盘数据搬到了网卡，不需要用户态处理什么操作。

**sendfile+DMA scatter/gather实现的零拷贝**

1. 用户进程发起sendfile系统调用，**上下文（切换1）从用户态转向内核态**
2. DMA控制器，把数据从硬盘中拷贝到内核缓冲区。
3. CPU把内核缓冲区中的**文件描述符信息**（包括内核缓冲区的内存地址和偏移量）发送到socket缓冲区
4. DMA控制器根据文件描述符信息，直接把数据从内核缓冲区拷贝到网卡
5. **上下文（切换2）从内核态切换回用户态**，sendfile调用返回。

可以发现，`sendfile+DMA scatter/gather`实现的零拷贝，I/O发生了**2**次用户空间与内核空间的上下文切换，以及2次数据拷贝。其中2次数据拷贝都是包**DMA拷贝**。这就是真正的 **零拷贝（Zero-copy)** 技术，全程都没有通过CPU来搬运数据，所有的数据都是通过DMA来进行传输的。

##  进程和线程有什么区别  

进程是操作系统分配资源的最小单位，线程是 CPU 调度的最小单位。

每个进程有自己独立的内存空间，包括代码段、数据段、堆栈等，可以看作是一个正在运行的程序实例。进程之间相互独立。

线程属于进程，一个进程可以包含多个线程。线程共享进程的内存空间和资源，比如文件句柄、数据段，但每个线程有自己独立的栈和寄存器。


线程共享进程的堆，但有自己私有的栈。
![[Pasted image 20260205161155.png]]



线程共享内存空间，所以通信用全局变量就可以通信，比较方便，但需要注意同步机制。线程创建的开销比进程小。


#### 协程：比线程更轻量的方案

线程虽然比进程轻量，但还是有开销。每次线程切换要陷入内核态，保存和恢复寄存器、栈指针，还要刷新TLB。高并发场景下，比如10万个连接，每个连接一个线程，光线程切换就能把 CPU 吃满。

协程就是为了解决这个问题。协程完全在用户态调度，一个线程里可以跑几十万个协程，切换时只需要保存几个寄存器，开销比线程小两个数量级。

Go 语言的 goroutine 就是协程的典型实现，用的是 M:N 模型，M 个协程映射到 N个操作系统线程上。
