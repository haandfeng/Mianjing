11. **你简历提到 RAG，你在 RAG 里做了什么？**

    - **文档预测**：用户问题可能比较短，文档可能比较长，根据用户问题预测可能提取的文档，与数据库里 embedding 的文档进行匹配

    - **重排序器**：使用 Bert 模型进行用户问题和文档之间相关度匹配，对所有方法提取的文档重新排序，找到最相关的文档，最后喂给大语言模型

	1. **能不能做成异步（async）？为什么你们最后没做？**
		可以，OOM的问题
	改用**异步（Asynchronous）请求的核心优势在于实现“计算”与“IO”的全面并联**，彻底消除 CPU 和 GPU 互相等待的死时间。
	
	以下是具体的优势分析：
	
	---
	
	### 1. 消除“木桶效应”：实现流水线并行（Pipelining）
	
	在**同步**模式下，你的链路是串行的： `[读取/预处理] -> [网络发送] -> [GPU 计算(死等)] -> [写入 Milvus] -> [开始下一个 Batch]` 此时，当 GPU 在计算时，你的 CPU 和网卡在闲置；当你在写入 Milvus 时，昂贵的 GPU 在等下一波数据。
	
	在**异步**模式下，链路变成了**流水线**：
	
	- **线程 A：** 正在把 Batch 1 的结果写入 Milvus。
	    
	- **线程 B（异步 Task）：** 正在等待 GPU 返回 Batch 2 的结果。
	    
	- **线程 C：** 正在预处理 Batch 3 的文本并准备发送。
	    
	
	**优势：** 你的总耗时不再是所有环节的加和，而是取决于**最慢的那个环节**（通常是 GPU 计算）




面对这种“高延迟、长耗时”的同步请求，确实有很多工程上的陷阱。如果只是简单地发一个请求然后死等，系统极易出现雪崩。

## 1. 编程层面的“坑”与对策

### ❌ 坑点：超时时间设置（Timeouts）

绝大多数 HTTP 客户端（如 `requests`, `httpx`）都有默认超时时间（通常是 30s 或 60s）。Embedding 一个大 Batch 往往会超过这个时间。

- **对策：** 不要简单地设置 `timeout=None`。你应该区分**连接超时**（Connect Timeout）和**读超时**（Read Timeout）。将读超时设置得长一些（如 300s），但连接超时要短。
    

### ❌ 坑点：内存溢出（OOM）

如果你一次性把几十万个文档加载到内存中准备发送，或者并发量过高导致返回的 Embedding（这可是巨大的浮点数矩阵）堆积在内存里，你的客户端会直接崩溃。

- **对策：** 使用**生成器（Generator）**或**游标（Cursor）**流式读取文档，并结合 `asyncio.Semaphore` 限制内存中的活动请求数量。
    

### ❌ 坑点：失败重试逻辑

如果一个包含 64 条文档的 Batch 在第 59 秒因为网络抖动断了，你是全部重发还是放弃？

- **对策：** 必须实现**指数退避（Exponential Backoff）**的重试机制。同时，确保服务端操作是**幂等**的（对于 Embedding 来说天然幂等，只要输入一致，输出就一致）。

## **4) Coding 题**

24. **开始 coding：确认输入条件**
    
    **回答：** 
    - 确认 M、N 的长度是否 >= 1？（面试官说可以这么认为）
    - 确认"包含"的意思：窗口里字符出现次数可以多于 T，只要满足至少需要的数量（如第二个测试样例可以有 double e，只要目标字符串的字符数量满足即可）

25. **你先讲一下思路**（滑动窗口 + Counter + less/need）
    
    **回答：** 
    - 用一个 counter 统计 N 字符串的字符出现数量
    - 用滑动窗口维护左指针和右指针，统计窗口里 N 字符串字符出现的次数
    - 用变量 `less` 判断有多少个数字至今为止还是大于 0 的
    - 如果 `less > 0`，说明还没完全找到子串；如果 `less = 0`，说明找到子串了
    - 找到子串后获取答案，判断答案大小，滑动窗口不断收缩找到最小子字符串
    - 当 `less` 不再等于 0 时，再向右滑，最终滑动整个字符串，获得最小的子字符串
    
26. 追问：**滑动窗口这个概念你最早从哪学到的？**
    
    **回答：** 最早是在计算机网络 TCP 的可靠性保证数据交付一致的时候学到的，后来做力扣题的时候也学到这个思路。
    
27. **Follow-up：如果要求 substring 还要保持 T 的字符顺序，怎么做？**
    
    **回答：** 
    - 在窗口里再放一个指针，从 T 的 0 位置开始指
    - 遇到一样的字符，T 的指针就往前 move 加一，直到 move 到结尾，就是一个结果
    - 找到答案后需要缩小窗口，考虑重复问题（如中间再加一个 t）
    - 从头再开始匹配，找到下一个开头的位置，然后再进行一次匹配
    - 先找到第一个字符串，记录位置，然后 left 指针跳到字符串 T 的开头，从第二个 t 开始重新匹配
    - 不确定时间复杂度是否最优，因为可能有重复比较
    
28. 追问：**复杂度是多少？**（并进一步纠正你：最终是 **O(MN)**，你认同）
    
    **回答：** 
    - 最初认为是 O(N)，因为第一次比较是 N，第二次是 N/2，第三次是 N/4，是等比数列求和，提取公因式后是常数，所以是 O(N)
    - 但面试官指出：每次遍历 T 里的每个字符（T、T、G）都需要遍历一遍 M 去找它的位置，这一步是省不了的，所以应该是 O(MN)
    - 最终认同是 O(MN)
    - 面试官提到 DP 可能会简单一些
