#  以 Java 为例，描述一个应用程序从编码到最终运行的过程和原理。 


Java 源码（.java）→ javac 编译为字节码（.class）→（可打包 jar）→ java 启动 JVM → 类加载/验证/链接/初始化 → 解释执行 + JIT 编译成机器码 → 运行期间由 GC 管理堆内存 → 所有非守护线程结束后 JVM 退出。

# 操作系统里的“中断”是什么？为什么会发生中断？

CPU停下当前的工作任务，去处理其他事情，处理完后回来继续执行刚才的任务，这一过程便是中断。

中断使得计算机系统具备应对对处理突发事件的能力，提高了CPU的工作效率，如果没有中断系统，CPU就只能按照原来的程序编写的先后顺序，对各个外设进行查询和处理，即轮询工作方式，轮询方法貌似公平，但实际工作效率却很低，却不能及时响应紧急事件。

# 常见的 IO 模型有哪些？（内存/网络都可以）

| **IO 模型**                      | **是否阻塞** | **是否同步** | **常见场景**           |
| ------------------------------ | -------- | -------- | ------------------ |
| 阻塞 IO（Blocking IO）             | 阻塞       | 同步       | 最基础 socket / 文件 IO |
| 非阻塞 IO（Non-blocking IO）        | 不阻塞      | 同步       | 轮询 + socket        |
| IO 多路复用（select / poll / epoll） | 阻塞（等待事件） | 同步       | 高并发网络服务器           |
| 信号驱动 IO（Signal-driven IO）      | 不阻塞      | 同步       | 较少使用               |
| 异步 IO（Asynchronous IO）         | 不阻塞      | 异步       | AIO、Windows IOCP   |


# 网络 IO 的 select / poll / epoll 是什么思路？（IO 多路复用怎么处理事件）

select 实现多路复用的方式是，将已连接的 Socket 都放到一个文件描述符集合，然后调用 select 函数将文件描述符集合拷贝到内核里，让内核来检查是否有网络事件产生，检查的方式很粗暴，就是通过遍历文件描述符集合的方式，当检查到有事件产生后，将此 Socket 标记为可读或可写，接着再把整个文件描述符集合拷贝回用户态里，然后用户态还需要再通过遍历的方法找到可读或可写的 Socket，然后再对其处理。


所以，对于 select 这种方式，需要进行 2次「遍历」文件描述符集合，一次是在内核态里，一个次是在用户态里，而且还会发生2次「拷贝」文件描述符集合，先从用户空间传入内核空间，由内核修改后，再传出到用户空间中。


poll 不再用 BitsMap 来存储所关注的文件描述符，取而代之用动态数组，以链表形式来组织，突破了select 的文件描述符个数限制，当然还会受到系统文件描述符限制。

但是 poll 和 select 并没有太大的本质区别，都是使用「线性结构」存储进程关注的 Socket 集合，因此都需要遍历文件描述符集合来找到可读或可写的 Socket，时间复杂度为 0（n），而且也需要在用户态与内核态之间拷贝文件描述符集合，这种方式随着并发数上来，性能的损耗会呈指数级增长。

# 支付环节涉及对账/对接银行：如何保证系统数据一致性？能想到哪些方法？



这个问题本质上是在问：两个独立系统（你的系统和银行）之间，网络不可靠、任何一步都可能失败，怎么保证钱不错。

**先理解问题的根源：** 你的系统和银行是两个独立的数据库，没法用一个事务同时操作两边。你扣了用户余额，调银行接口转账，银行可能超时了——你不知道钱到底转没转。这就是分布式一致性问题。

**方法一：本地消息表（最常用）**

核心思路是把"扣余额"和"记一条待支付消息"放在同一个本地事务里，保证要么都成功要么都失败。然后由定时任务异步地去调银行接口完成支付。

流程：用户发起支付 → 本地事务里同时做两件事：扣余额 + 往消息表插一条状态为"待支付"的记录 → 事务提交 → 异步调银行接口 → 银行返回成功 → 更新消息状态为"已完成"。如果调银行超时或失败，定时任务扫描消息表里状态还是"待支付"的记录，重新调银行查询接口确认结果，再更新状态。

关键点：调银行的操作必须幂等（用唯一的支付流水号），重试多少次都不会重复扣款。

**方法二：对账（兜底机制，和银行对数）**

不管上面做得多好，都可能有边界情况漏掉。所以必须有独立的对账系统做最后一道防线。

做法：每天（或每小时）从银行拉一份账单文件（银行侧所有成功的交易记录），和我们系统里的支付流水逐条比对。比对的维度是：支付流水号、金额、时间。

会出现三种差异：

"我们有银行没有"——我们记录支付成功了，但银行那边没这笔。可能是我们状态更新错了，需要给用户退款或重新发起。

"银行有我们没有"——银行扣款成功了，但我们没收到回调（网络丢了）。需要补一条支付成功记录，把用户的订单状态改成已支付。

"金额不一致"——最严重，必须人工介入排查。

差异记录写入差异表，小额自动补偿，大额告警人工处理。

**方法三：TCC（Try-Confirm-Cancel）**

适用于多方参与的复杂场景（比如同时涉及余额、优惠券、积分）。

三个阶段：Try 阶段冻结资源（余额减掉但标记为冻结，不是真扣）→ Confirm 阶段所有参与方都冻结成功后，一起确认（冻结变真扣）→ Cancel 阶段任何一方 Try 失败，所有已冻结的资源释放回去。

好处是不会出现"钱扣了但优惠券没扣"的半成功状态。坏处是实现复杂，每个参与方都要实现 Try / Confirm / Cancel 三个接口。

**方法四：最大努力通知**

银行侧支付完成后，会主动回调通知我们的系统。如果我们没响应，银行会按策略重试（比如 1s、5s、30s、5min）。我们收到通知后更新订单状态，返回成功，银行停止重试。

这是最简单的方案，但依赖银行的重试机制，不够可靠，所以一定要配合方法二的对账兜底。

**面试时怎么组织回答：**

实际系统中这四种不是选一个，而是分层组合——本地消息表保证本地和远程操作的最终一致，银行回调 + 重试保证实时性，对账系统做每日兜底，TCC 用在需要多方同时操作的场景。一句话总结：实时靠本地消息表 + 幂等重试，兜底靠独立对账系统，核心原则是"宁可多查一次也不能漏一笔"。
