# 你用 cursor 过程中有没有遇到实际问题？

主要遇到几类问题：

**1. 上下文与大型项目的问题**

| **问题类型** | **具体表现** | **本质原因** |
|------------|------------|------------|
| **局部视角局限** | 在 monorepo 或多模块项目中，Cursor 往往只看局部文件就开始改，忽略其他模块或配置 | Cursor + LLM 没法像人那样真正"整仓库全盘加载后再推理" |
| **上下文窗口不够全** | 虽然感觉"项目都打开了，它应该知道"，但实际上它只看了少数被高亮或最近改过的文件 | 只是一个有限窗口的"聪明文本编辑器" |

**2. 代码质量 & 可控性问题**

| **问题** | **表现** | **影响** |
|---------|---------|---------|
| **一改就改太多** | 给个模糊任务"帮我优化下这个函数"，结果函数重写了、接口签名顺带改了、调用方也"顺手修了" | review diff 的压力直线上升 |
| **代码风格不一致** | 项目有特定的命名习惯、错误处理方式、日志规范，但它有时候按自己的"通用习惯"写 | 同一项目里风格混杂 |
| **不太管测试 & 边界** | 会生成看起来很漂亮的代码，但没有跑测试的意识 | 边界条件、极小概率分支经常要自己补 |

**结果**："AI 写得很快，人类 review 更累"，如果不给它约束，就会有这种感觉

**3. 数据验证和边界条件处理**

| **问题** | **表现** | **解决方案** |
|---------|---------|------------|
| **假设理想情况** | Cursor 生成的代码往往假设理想情况，比如简历解析时，它假设所有字段都存在 | 需要手动添加大量的边界检查和异常处理 |
| **Pydantic 字段设置问题** | 使用 pydantic 做结构化输出时，它默认所有字段都是 required，但实际需要很多字段设为 Optional | 手动设置为 Optional，允许 None 值 |

**4. 工具选择不当**

- **初始问题**：一开始 Cursor 生成代码时没有用 pydantic，导致结构化输出格式不稳定
- **解决方案**：后来我根据项目需求（需要保证 JSON 格式严格符合要求）改用了 pydantic，这是基于我之前实习经验和对工具特性的理解

**5. 和 Git / 构建系统配合时的小坑**

| **问题** | **表现** |
|---------|---------|
| **Git 冲突** | 本地改了一些东西，它在旧版本基础上再改，提交/合并的时候冲突一堆 |
| **对构建系统不敏感** | 比如 Monorepo 下不同 package 的 build 脚本、特定 CI 规则，它经常"只改源码"，但没想到还有代码生成、编译参数、CI 校验等等 |

**本质问题**：它在文件层面很聪明，在工程化层面比较"半懂不懂"

**6. 代码可读性和维护性**

- **代码结构问题**：Cursor 一次性生成几百上千行代码，虽然功能能跑，但代码结构、命名风格不够统一
- **解决方案**：我让 Cursor 生成后，再要求它写一份架构文档和 onboarding 文档，包括：项目结构、启动方式、关键模块说明
- **效果**：这样既方便我自己理解代码，也方便团队协作和后续迭代


**总结**：Cursor 适合快速搭建框架和生成样板代码，但在大型项目的全局理解、代码质量控制、工程化配合、细节优化、边界处理、测试验证这些方面，还是需要人工介入和迭代优化。

# 你调了有多少版提示词？

主要就是修改和评价这两个提示词比较重要。我采用了系统化的 prompt 工程方法来优化：

**1. 结构化 Prompt 设计**

| **要素** | **说明** |
|---------|---------|
| **角色定义** | 明确 agent 的角色（如"资深简历优化专家"），设定专业背景和职责边界 |
| **任务分解** | 将复杂任务拆分为清晰的步骤（解析 JD → 提取关键词 → 匹配简历 → 优化表达 → 格式检查） |
| **输出格式规范** | 使用结构化输出（JSON Schema 或 pydantic），确保输出格式稳定可解析 |
| **Few-shot 示例** | 提供 2-3 个高质量的输入输出示例，让模型理解期望的效果 |

**2. 迭代优化流程**

| **阶段** | **方法** | **结果** |
|---------|---------|---------|
| **初版生成** | 修改的提示词一开始让 cursor 生成 | 效果很一般，没有对简历进行很大修改 |
| **基准测试** | 用 10-20 份不同类型的简历作为测试集，评估输出质量 | - |
| **A/B 测试** | 对比不同版本的 prompt（如是否加入 Few-shot、是否强调关键词匹配），量化评估指标 | 关键词覆盖率、格式正确率、用户满意度 |

**3. 最终优化**

- **基于他人经验**：后来朋友之前也用 AI 改简历，他把自己的 prompt 贡献出来，我们基于他的版本进一步优化
- **调整优化**：结合我们的测试结果，调整了关键词提取的权重、优化了输出格式的约束
- **评价 Agent**：评价的 agent prompt 相对简单，主要是打分标准和反馈格式，所以初版就基本满足需求，没有再大改

**经验总结**：
- Prompt 不是一次性写好的，需要基于实际效果迭代优化
- 建立评估体系很重要，不能只凭感觉判断好坏
- 结构化设计（角色、步骤、格式）比自由描述效果更好
- 版本管理帮助追踪哪些改动真正有效

# 你们当时为什么选择用 LangGraph 去写？

在选框架的时候，我其实对比过几类方案：

**1. 主流框架分类**

| **类型** | **框架** |
|---------|---------|
| **代码型框架** | LangGraph、LlamaIndex |
| **低/无代码平台** | Dify、Flowise 等 |

**2. 各框架定位对比**

**LangChain vs LangGraph 的主要区别**：

| **特性** | **LangChain** | **LangGraph** |
|---------|--------------|--------------|
| **定位** | 通用 LLM 应用框架，提供 models、tools、chains、agents 等一整套高层抽象 | LangChain 团队出的有状态流程/Agent 编排框架，专门解决长流程、可恢复、可回溯的控制流问题 |
| **特点** | 组件化设计，适合快速搭建简单的 agent 或 chatbot，生态丰富、例子多 | 基于图（Graph）的编排方式，把应用画成一张图（节点+边），然后它帮你管状态、重试、恢复等 |
| **控制流** | 主要是链式调用（Sequential Chains）或简单的 Agent 循环，控制流相对简单 | 支持复杂的条件分支、循环、并行执行、动态路由等 |
| **状态管理** | 在复杂、有状态的长流程里，控制流和状态管理要自己补很多东西 | 内置状态持久化、检查点（checkpoint）机制，支持流程中断后恢复 |
| **可恢复（Resumable）** | ❌ 不支持：如果服务中断，必须从头开始执行 | ✅ 支持：通过 checkpoint 机制，可以从上次中断的节点继续执行，不需要重新开始 |
| **可回溯（Traceable）** | ❌ 不支持：需要自己实现日志记录，无法完整追踪执行历史 | ✅ 支持：自动记录每个节点的执行历史（输入、输出、耗时），可以完整回溯整个流程 |
| **Human-in-the-loop** | ❌ 不支持：无法在流程执行过程中暂停等待人工介入 | ✅ 支持：可以在特定节点暂停，等待人工审核、反馈或决策后再继续执行 |
| **适用场景** | 快速原型、简单的问答系统、基础的 RAG 应用 | 多步骤长流程、需要状态管理的复杂 Agent、生产环境部署 |

**核心区别总结**：
- LangChain = "组件库 + 简单链式调用"，适合快速搭建
- LangGraph = "图编排 + 状态管理"，适合复杂流程
- LangGraph 可以看作是 LangChain 在复杂流程编排上的"增强版"，两者可以配合使用（LangGraph 底层依赖 LangChain 的组件）

**具体代码示例对比**：

以"改简历项目"为例，展示两种实现方式的区别：

**使用 LangChain 实现（简单链式）**：

```python
from langchain.chains import SequentialChain
from langchain.prompts import PromptTemplate
from langchain.llms import OpenAI

# 定义各个步骤的 prompt
parse_prompt = PromptTemplate(...)
evaluate_prompt = PromptTemplate(...)
modify_prompt = PromptTemplate(...)

# 创建链式调用
chain = SequentialChain(
    chains=[
        parse_chain,      # 解析简历
        evaluate_chain,   # 评价简历
        modify_chain      # 修改简历
    ],
    input_variables=["resume", "jd"],
    output_variables=["modified_resume"]
)

# 执行（一次性完成，无法循环）
result = chain.run(resume=resume, jd=jd)
```

**问题**：
- ❌ 无法实现"如果分数 < 9 分就重新改"的循环逻辑
- ❌ 无法保存中间状态（如果服务中断，需要从头开始）
- ❌ 无法回溯执行历史（不知道哪个步骤出了问题）
- ❌ 不支持可恢复：服务崩溃后必须重新执行整个流程
- ❌ 不支持可回溯：需要自己实现日志记录，无法查看完整的执行轨迹
- ❌ 不支持 Human-in-the-loop：无法在流程执行过程中暂停等待人工审核或反馈

**使用 LangGraph 实现（图编排 + 状态管理）**：

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

# 定义状态
class ResumeState(TypedDict):
    resume: str
    jd: str
    parsed_resume: dict
    score: float
    modified_resume: str
    iteration: int

# 定义节点函数
def parse_node(state: ResumeState):
    # 解析简历
    parsed = parse_resume(state["resume"])
    return {"parsed_resume": parsed}

def evaluate_node(state: ResumeState):
    # 评价简历
    score = evaluate_resume(state["parsed_resume"], state["jd"])
    return {"score": score}

def modify_node(state: ResumeState):
    # 修改简历
    modified = modify_resume(state["parsed_resume"], state["jd"])
    return {"modified_resume": modified, "iteration": state.get("iteration", 0) + 1}

def should_continue(state: ResumeState):
    # 条件判断：如果分数 >= 9 或迭代次数 >= 3，则结束
    if state["score"] >= 9 or state.get("iteration", 0) >= 3:
        return "end"
    return "modify_again"

# 构建图
workflow = StateGraph(ResumeState)
workflow.add_node("parse", parse_node)
workflow.add_node("evaluate", evaluate_node)
workflow.add_node("modify", modify_node)

# 添加边
workflow.set_entry_point("parse")
workflow.add_edge("parse", "evaluate")
workflow.add_edge("evaluate", "modify")

# 条件边：根据分数决定是否继续
workflow.add_conditional_edges(
    "modify",
    should_continue,
    {
        "end": END,
        "modify_again": "evaluate"  # 循环回到评价节点
    }
)

# 编译并运行（支持状态持久化和恢复）
from langgraph.checkpoint.memory import MemorySaver

# 使用 MemorySaver 作为 checkpoint 存储（生产环境可以用数据库）
checkpointer = MemorySaver()
app = workflow.compile(checkpointer=checkpointer)

# 配置 thread_id，用于标识不同的执行会话
config = {"configurable": {"thread_id": "resume_workflow_1"}}

# 第一次执行
result = app.invoke({"resume": resume, "jd": jd}, config=config)
```

**优势详解**：

**1. 支持循环**：
- ✅ 如果分数 < 9，自动回到评价节点重新改
- ✅ 通过条件边（conditional edges）实现动态路由

**2. 状态管理**：
- ✅ 每个节点的输入输出都保存在状态中
- ✅ 状态是类型安全的（通过 TypedDict 定义）

**3. 可恢复（Resumable）**：

```python
# 场景：服务在执行到 evaluate 节点时崩溃了

# 第一次执行（执行到 evaluate 节点后崩溃）
try:
    result = app.invoke({"resume": resume, "jd": jd}, config=config)
except Exception:
    # 服务崩溃，但状态已保存到 checkpoint
    pass

# 服务恢复后，可以从上次中断的地方继续
# 获取当前状态
current_state = app.get_state(config)
print(f"当前节点: {current_state.next}")  # 输出: ['modify']
print(f"已执行节点: {current_state.values}")  # 包含 parse 和 evaluate 的结果

# 继续执行（从 modify 节点开始）
result = app.invoke(None, config=config)  # 传入 None 表示使用已保存的状态
```

**4. 可回溯（Traceable）**：

```python
# 查看完整的执行历史
from langgraph.graph import START

# 获取所有执行步骤
state_history = app.get_state_history(config)

for i, state in enumerate(state_history):
    print(f"步骤 {i}:")
    print(f"  节点: {state.next}")
    print(f"  状态值: {state.values}")
    print(f"  元数据: {state.metadata}")  # 包含每个节点的执行时间、token 消耗等

# 也可以查看特定节点的执行详情
evaluate_state = app.get_state(config, as_node="evaluate")
print(f"评价节点的输入: {evaluate_state.values['parsed_resume']}")
print(f"评价节点的输出: {evaluate_state.values['score']}")
```

**5. Human-in-the-loop**：

```python
from langgraph.graph import END
from langgraph.prebuilt import interrupt

# 在评价节点后添加人工审核步骤
def human_review_node(state: ResumeState):
    # 这里会暂停执行，等待人工介入
    # 在实际应用中，可以通过 API 或 UI 获取人工反馈
    return {"human_feedback": "需要优化工作经历部分"}

# 修改图结构，添加人工审核节点
workflow.add_node("human_review", human_review_node)

# 在 evaluate 和 modify 之间插入人工审核
workflow.add_edge("evaluate", "human_review")

# 使用 interrupt 让流程在 human_review 节点暂停
workflow.add_edge("human_review", interrupt)  # 暂停，等待人工反馈

# 或者使用条件边，根据人工反馈决定下一步
def after_human_review(state: ResumeState):
    if state.get("human_feedback") == "通过":
        return "end"
    else:
        return "modify"

workflow.add_conditional_edges(
    "human_review",
    after_human_review,
    {"end": END, "modify": "modify"}
)

# 执行时，流程会在 human_review 节点暂停
result = app.invoke({"resume": resume, "jd": jd}, config=config)
# 此时流程暂停，等待人工反馈

# 人工审核完成后，更新状态并继续执行
app.update_state(
    config,
    {"human_feedback": "需要优化工作经历部分"}
)
result = app.invoke(None, config=config)  # 继续执行
```

**6. 条件分支**：
- ✅ 可以根据不同条件走不同的路径
- ✅ 支持复杂的业务逻辑判断

**实际场景对比**：

| **场景**      | **LangChain**    | **LangGraph**                      |
| ----------- | ---------------- | ---------------------------------- |
| **简单问答**    | ✅ 适合：链式调用即可      | ⚠️ 过度设计                            |
| **需要循环的场景** | ❌ 难以实现           | ✅ 天然支持                             |
| **需要状态持久化** | ❌ 需要自己实现         | ✅ 内置支持                             |
| **需要回溯调试**  | ❌ 需要自己记录日志       | ✅ 自动记录完整执行历史                       |
| **服务中断恢复**  | ❌ 必须从头开始         | ✅ 可以从 checkpoint 恢复                |
| **需要人工审核**  | ❌ 无法在流程中暂停等待人工介入 | ✅ 支持 interrupt 和 human-in-the-loop |
| **复杂条件分支**  | ❌ 实现复杂           | ✅ 条件边轻松实现                          |
| **生产环境部署**  | ⚠️ 需要自己处理容错和监控   | ✅ 内置 checkpoint、状态管理、可观测性          |

**LlamaIndex**：
更偏数据/RAG 编排，强在索引私有数据、构建知识助手，对向量库、文档结构有很多高级封装。但做复杂控制流不是它的主战场。

**Dify/Flowise**：
低代码平台，通过可视化方式搭工作流，适合快速 PoC 和团队内部使用。但复杂逻辑、版本控制、代码 review 会有点别扭，不太适合作为简历里展示工程能力的主角项目。

**3. 为什么最终选 LangGraph**

综合下来，我最终选了 LangGraph，主要有三点考虑：

**场景匹配度**：
我的流程本质上是一张图：解析 JD → 解析简历 → 匹配打分 → 生成反馈，中间还有"异常分支"和"兜底策略"（比如打分低于 9 分要重新改）。用 LangGraph 可以非常自然地把这些节点和边建模成一个有状态的工作流。

**工程能力**：

LangGraph 专门解决长流程、可恢复、可回溯的问题，具体来说：

| **特性** | **说明** | **价值** |
|---------|---------|---------|
| **长流程（Long-running Workflow）** | 传统的一次性调用（如简单 API）是"请求-响应"模式，但我们的简历评估流程需要多个步骤：解析 JD → 解析简历 → 匹配打分 → 生成反馈 → 如果分数不够还要循环重改 | 可以把整个流程建模成一个有状态的工作流，而不是简单的函数调用链 |
| **可恢复（Resumable）** | 如果流程执行到一半（比如刚完成简历解析）时服务崩溃或网络中断，传统方式需要从头开始 | LangGraph 支持状态持久化，可以把每个节点的执行状态（输入、输出、中间结果）保存到数据库，服务恢复后可以从上次中断的节点继续执行 |
| **可回溯（Traceable/Replayable）** | 可以完整记录整个流程的执行历史：哪个节点执行了、用了什么参数、输出了什么结果、花了多长时间 | 这对于调试、A/B 测试、性能分析、问题排查都很有价值 |

这些特性对后续做 A/B 测试或者线上化是有价值的。而且它是一个纯代码框架，既保持了 LangChain 生态的兼容性，又不会把架构锁死在某种黑盒 Agent 模式里，我可以自己设计 prompt 结构和打分逻辑，这更符合"做一个可维护、可拓展的工程项目"的目标。

**个人经验**：
我之前在西门子实习的时候用过 LangGraph，上手成本比较低。

**总结**：所以综合"场景匹配度 + 工程可维护性 + 个人经验"这几个因素，最后选择了用 LangGraph 来搭整个简历评估流程。

我会从架构、检索、生成、评估四个层面进行系统化改进：

**1. 架构层面：引入 Agent Loop 和自适应 Query Rewriting**

**动态 Query Rewriting**：

| **当前问题** | **解决方案** |
|------------|------------|
| 虽然之前也做了 query 改写，但主要是针对中英文 gap 的静态改写 | 现在会在 LangGraph 框架中加入一个 loop：先检索 → 判断检索结果是否足够回答问题 → 如果不够，动态改写 query（比如扩展同义词、添加专业术语、分解复杂问题）→ 再次检索 |

这样可以解决"差一两个关键词找不到文档"的问题，提高召回率。

**多轮检索策略**：
如果第一轮检索结果不理想，可以尝试不同的检索策略（关键词检索、语义检索、混合检索）。支持"检索-评估-再检索"的迭代过程，直到找到足够的相关文档。

**2. 检索层面：Embedding 优化和 Chunk 策略**

**Embedding 模型微调**：

| **当前问题** | **解决方案** |
|------------|------------|
| 通用 embedding 模型对中英文语义理解不一致，对 CT 机专业术语效果不好 | 用现有的文档数据对 embedding 模型进行领域微调（Domain-specific Fine-tuning），可以显著提升专业术语和领域特定表达的检索准确率 |

**Chunk 策略优化**：

**当前问题**：
- 当前是公司已经切好的 500 token 左右的 chunk，但可能不是最优的
- 分块策略直接影响检索效果，因为：
  - 如果 chunk 太大：可能包含多个不相关的主题，检索时噪声多
  - 如果 chunk 太小：可能丢失上下文信息，语义不完整
  - 如果按固定长度切：可能把一个完整的概念切断了

**解决方案**：可以尝试以下几种分块技术：

| **分块策略**                         | **原理**                         | **优点**                      | **适用场景**                |
| -------------------------------- | ------------------------------ | --------------------------- | ----------------------- |
| **语义分块（Semantic Chunking）**      | 不是按固定长度切，而是根据语义边界（句子、段落、主题）来切分 | 保证每个 chunk 语义完整，不会把一个完整概念切断 | 技术文档、长文章、需要保持语义连贯性的内容   |
| **滑动窗口（Sliding Window）**         | 相邻的 chunk 之间有重叠部分，避免边界信息丢失     | 避免边界效应，提高召回率                | 所有类型的文档，特别是需要高召回率的场景    |
| **层次化分块（Hierarchical Chunking）** | 根据文档的层次结构（标题、章节、段落）进行多粒度分块     | 既保证精确性（细粒度），又保证上下文完整性（粗粒度）  | 结构化文档（技术手册、API 文档、操作指南） |

**Hybrid Search 增强**：

| **优化方向**                   | **具体方案**                                                   | **优势**                                        |
| -------------------------- | ---------------------------------------------------------- | --------------------------------------------- |
| **GPU加速多向量检索**             | 使用GPU并行计算多个embedding模型的向量检索（如同时计算千问3稠密向量、BGE-M3稀疏向量、多向量模式） | 大幅提升检索速度，从秒级降到毫秒级，支持实时检索                      |
| **Reranker精筛**             | 在最终输入文档给大模型前，不要选那么多文档（如从TOP50降到TOP10），使用reranker进行精筛       | 减少输入token数，节省模型生成时间，降低API成本，提高答案质量（只保留最相关的文档） |



**3. 生成层面：上下文压缩和 Summary 优化**

**上下文压缩（Context Compression）**：

| **当前问题** | **解决方案** |
|------------|------------|
| 检索到的文档可能很多很杂，模型 summary 能力有限，导致准确率下降 | 在喂给大模型前，先用一个"压缩 agent"对检索到的文档进行预处理，提取最相关的片段；使用 LongContextReorder 等技术，优化文档在 prompt 中的排列顺序 |

**分阶段生成**：
对于复杂问题，可以拆分为多个子问题，分别检索和生成，最后再整合。使用 Chain of Thought (CoT) 或 Tree of Thoughts (ToT) 让模型更好地处理多文档信息。

**Chain of Thought (CoT) 思维链技术**：

**原理**：
- 传统方式：模型直接给出最终答案，中间推理过程不透明
- CoT：要求模型展示推理过程，一步一步思考，最后得出结论
- 核心思想："Let's think step by step"（让我们一步步思考）

**在 RAG 中的应用**：

| **步骤** | **说明** |
|---------|---------|
| **步骤 1** | 理解用户问题，识别需要哪些信息 |
| **步骤 2** | 分析检索到的文档，提取相关信息 |
| **步骤 3** | 整合多个文档的信息，识别一致性和冲突 |
| **步骤 4** | 基于整合的信息，推理出答案 |
| **步骤 5** | 验证答案的完整性和准确性 |

**优点**：
- 提高推理准确性：让模型"慢思考"，减少错误
- 可解释性：可以看到模型的推理过程，便于调试
- 适合复杂问题：多步骤、需要综合多个文档的问题

**Tree of Thoughts (ToT) 思维树技术**：

**原理**：
- CoT 是线性的：A → B → C → 答案
- ToT 是树状的：从问题出发，探索多个可能的推理路径，然后评估每条路径，选择最优的
- 核心思想：不是只有一条推理路径，而是探索多条路径，选择最好的

**在 RAG 中的应用**：

| **步骤** | **说明** |
|---------|---------|
| **步骤 1** | 问题分解 - 将复杂问题拆分为多个子问题 |
| **步骤 2** | 多路径探索 - 对每个子问题，尝试不同的检索策略和推理路径 |
| **步骤 3** | 路径评估 - 评估每条路径的中间结果质量 |
| **步骤 4** | 路径选择 - 选择最优路径继续推理 |
| **步骤 5** | 答案整合 - 综合多条路径的信息，生成最终答案 |

**CoT vs ToT 选择建议**：

| **场景** | **推荐方法** | **原因** |
|---------|------------|---------|
| **简单到中等复杂度问题** | CoT | 成本低、效果好 |
| **非常复杂的问题** | ToT | 虽然成本高，但答案质量更好 |

**实际应用**：可以先尝试 CoT，如果效果不够好，再升级到 ToT

**4. 动态RAG：长期记忆和增量学习**

**核心思想**：
根据用户的问题和回答进行增量embedding，构建长期记忆系统。回答得好的问题会被引入知识库，持续优化系统。

**实现方案**：

| **功能** | **实现方式** | **价值** |
|---------|------------|---------|
| **高质量问答入库** | 当用户问题得到高质量回答（如用户点赞、评分高、或通过人工审核），将"问题-答案"对进行embedding并存入向量库 | 系统可以"记住"之前回答过的问题，避免重复计算，提高响应速度 |
| **增量embedding** | 定期（如每天/每周）对新增的高质量问答进行embedding，增量更新向量库，不需要全量重建 | 知识库持续增长，系统能力不断提升 |
| **问题相似度匹配** | 新问题进来时，先检索历史问答库，如果找到相似问题，直接返回历史答案或基于历史答案优化 | 提高回答质量，减少模型调用成本 |
| **用户反馈循环** | 收集用户反馈（点赞、点踩、修改建议），用于筛选哪些问答值得入库 | 确保入库的都是高质量内容，避免错误答案污染知识库 |

**技术实现**：
```python
# 伪代码示例
def dynamic_rag_pipeline(user_query):
    # 1. 先检索历史问答库
    similar_qa = search_qa_history(user_query, top_k=3)
    
    if similar_qa and similarity_score > 0.9:
        # 如果找到高度相似的历史问答，直接返回或优化
        return enhance_answer(similar_qa[0].answer, user_query)
    
    # 2. 如果没找到，走正常RAG流程
    docs = retrieve_documents(user_query)
    answer = generate_answer(docs, user_query)
    
    # 3. 如果回答质量高，存入历史问答库
    if evaluate_answer_quality(answer, user_query) > threshold:
        store_qa_pair(user_query, answer)
    
    return answer
```

**优势**：
- ✅ 系统具备"记忆"能力，越用越聪明
- ✅ 减少重复计算，提高响应速度
- ✅ 降低API调用成本
- ✅ 持续优化，知识库不断积累

**5. 图检索（Graph RAG）**

**核心思想**：
不仅检索文档片段，还利用文档之间的关联关系（如引用、主题、实体关系）进行图结构检索，提供更丰富的上下文信息。

**实现方案**：

| **功能** | **实现方式** | **价值** |
|---------|------------|---------|
| **知识图谱构建** | 从文档中提取实体（如设备名称、操作步骤、故障代码）和关系（如"设备A包含组件B"、"步骤1在步骤2之前"），构建知识图谱 | 理解文档之间的逻辑关系，提供结构化知识 |
| **图遍历检索** | 不仅检索相关文档，还沿着图结构检索相关的实体和关系（如检索到"故障代码X"，同时检索"故障代码X的解决方案"、"相关设备"等） | 提供更全面的上下文，回答更准确 |
| **多跳推理** | 通过图结构进行多跳推理（如"设备A → 组件B → 故障C → 解决方案D"） | 回答复杂问题，发现隐含关系 |
| **实体链接** | 将用户问题中的实体链接到知识图谱中的节点，基于实体关系进行检索 | 提高检索精度，理解问题意图 |

**技术实现**：
```python
# 伪代码示例
def graph_rag_pipeline(user_query):
    # 1. 实体识别和链接
    entities = extract_entities(user_query)  # 如：["CT机", "故障代码X"]
    linked_entities = link_to_knowledge_graph(entities)
    
    # 2. 图遍历检索
    # 从实体节点出发，沿着关系边检索相关节点
    related_nodes = graph_traverse(linked_entities, max_hops=2)
    # 如：故障代码X → 相关设备 → 操作手册 → 解决方案
    
    # 3. 获取相关文档
    related_docs = get_documents_from_nodes(related_nodes)
    
    # 4. 生成答案
    answer = generate_answer(related_docs, user_query)
    return answer
```

**适用场景**：
- ✅ **技术文档**：设备、组件、操作步骤之间的关联关系
- ✅ **知识库**：概念、术语、定义之间的层次关系
- ✅ **故障诊断**：故障现象、原因、解决方案之间的因果关系
- ✅ **操作流程**：步骤之间的先后顺序和依赖关系

**优势**：
- ✅ 提供更丰富的上下文信息
- ✅ 支持复杂问题的多跳推理
- ✅ 理解文档之间的逻辑关系
- ✅ 提高答案的准确性和完整性

**与向量检索的结合**：
- **混合检索**：向量检索找到相关文档 → 图检索找到关联实体和关系 → 综合两者结果
- **图增强向量**：在向量检索的基础上，用图结构扩展检索范围，找到间接相关的文档

检索的时候就是我嵌入的时候是能够加这个元数据的，然后我检索的时候就根据这个元数据来进行一个筛选 filter，把这些无关的数据全部 filter 掉。

**实现方式**：
```
嵌入时添加元数据（机型） → 检索时根据元数据 filter → 只返回对应机型的文档
```

# chunk 的话是怎样去切分的？

我们当时是没有切 chunk，就是因为他们给到我们的时候就是已经切好了，就是一小段一小段，就是按照公司的那种想法已经切好了，所以说我们就不方便说再进行再切了。他们有可能是一个一整个小章节切分，也可能是一个章节的一个小点切分，一个章节一个小点切分，但他们给到我的时候就是已经是这样的。他们文档也不算特别大，当时看的是一个文档大概就 500 个 TOKEN 左右，所以说我觉得已经没有必要再切了，就没有考虑过这件事。

**文档特点**：
- 公司已经切好，一小段一小段
- 可能是整个小章节切分，也可能是章节的小点切分
- 每个文档大概 500 个 TOKEN 左右
- 没有再进行二次切分

**常见的切分方法**：

| **切分策略** | **原理** | **优点** | **缺点** | **适用场景** |
|------------|---------|---------|---------|------------|
| **固定长度切分（Fixed-size Chunking）** | 按固定字符数或token数切分，如每500字符一段 | 实现简单，速度快 | 可能切断完整语义，丢失上下文 | 通用场景，对语义完整性要求不高的文档 |
| **语义分块（Semantic Chunking）** | 根据语义边界（句子、段落、主题）来切分，不是按固定长度 | 保证每个chunk语义完整，不会把一个完整概念切断 | 实现复杂，需要NLP模型判断语义边界 | 技术文档、长文章、需要保持语义连贯性的内容 |
| **滑动窗口（Sliding Window）** | 相邻的chunk之间有重叠部分（如重叠50-100字符），避免边界信息丢失 | 避免边界效应，提高召回率，确保重要信息不被遗漏 | 存储空间增加，可能有重复信息 | 所有类型的文档，特别是需要高召回率的场景 |
| **层次化分块（Hierarchical Chunking）** | 根据文档的层次结构（标题、章节、段落）进行多粒度分块 | 既保证精确性（细粒度），又保证上下文完整性（粗粒度），可以多级检索 | 实现复杂，需要解析文档结构 | 结构化文档（技术手册、API文档、操作指南、Markdown文档） |
| **递归分块（Recursive Chunking）** | 按层次递归切分：先按段落，段落太长按句子，句子太长按固定长度 | 兼顾语义完整性和灵活性，适应不同长度的文档 | 需要设置多个切分规则 | 混合类型的文档集合 |
| **基于标记符切分（Markdown/Code Splitting）** | 根据特定标记符切分，如Markdown的`#`、代码的`class`/`function` | 保持代码或文档的逻辑结构，便于理解 | 只适用于特定格式的文档 | 代码库、Markdown文档、结构化文档 |
| **句子分块（Sentence-based Chunking）** | 按句子边界切分，然后合并成固定大小的chunk | 保持句子完整性，比固定长度更自然 | 需要句子分割模型，可能仍有边界问题 | 自然语言文档、文章、对话记录 |

**选择建议**：
- **技术文档/操作手册**：推荐层次化分块 + 滑动窗口
- **代码库**：推荐基于标记符切分（按函数/类）
- **长文章/论文**：推荐语义分块 + 滑动窗口
- **通用场景**：推荐递归分块，兼顾灵活性和语义完整性
