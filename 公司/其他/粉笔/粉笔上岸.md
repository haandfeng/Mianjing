# 3. Redis缓存设计

当时是有用 Redis 做缓存，是吧？这当时是怎么设计的？

> 在**人力招聘系统**中，我通常会优先缓存**读频率高、变更相对可控、且对实时一致性要求不高**的数据。

> 主要包括四类：

> 第一是**招聘相关的字典和配置数据**，例如岗位类别、招聘阶段、候选人状态、渠道来源和面试结果类型，这些数据在列表和详情页频繁使用，但更新不频繁；

> 第二是**岗位和候选人的基础信息快照**，比如岗位名称、部门、地点、候选人姓名、当前阶段和负责招聘官，用于减少岗位、流程、用户等多表 Join；

> 第三是**流程与权限相关数据**，如用户角色、岗位可操作权限、面试官可见范围，在登录或上下文切换时加载并缓存；

> 第四是**高频统计类数据**，如我的待处理候选人数量、岗位下各阶段人数、今日面试数，这类数据我会采用短 TTL 或事件驱动方式缓存，在性能和一致性之间做权衡。

## 4.1 MySQL vs PostgreSQL

能讲一下，PgSQL和MySQL，你觉得它有什么区别吗？

**MySQL vs PostgreSQL 对比**：

| **特性**     | **MySQL**     | **PostgreSQL**                |
| ---------- | ------------- | ----------------------------- |
| **特点**     | 轻量、易用         | 功能丰富、可扩展                      |
| **适用场景**   | 学生项目、简单应用     | 复杂业务需求                        |
| **文档资源**   | 较多，易于查找       | 相对较少                          |
| **数据类型支持** | 主要支持关系型、结构化数据 | 通过插件支持多种数据类型（Redis、MongoDB 等） |
| **扩展性**    | 相对有限          | 通过插件扩展性强                      |


> PostgreSQL 和 MySQL 的主要区别在于定位。MySQL 更偏向高并发、简单 CRUD 的互联网场景，而 PostgreSQL 在 SQL 标准、事务一致性、复杂查询和扩展能力上更强。我在 HR 系统中更倾向使用 PostgreSQL，因为涉及多表关联、报表统计以及对数据一致性的要求更高。



- ﻿MySQL是应用开发者创建出来的DBMS；而PostgreSQL是由数据库开发者创建出来的DBMS。换句话说，MySQL倾向于使用者的角度，回答的问题是"你想解决的是什么问题"；而PostgreSQL倾向于理论角度，回答的问题是"数据库应该如何来解决问题"。
- ﻿﻿PostgreSQL 支持JSON、XML 等现代应用程序功能，同时 MySQL 仅支持 JSON。
- ﻿﻿PostgreSQL 完全符合 ACID 标准，同时 MySQL仅与 InnoDB 存储引擎一起使用时才符合 ACID 标准。
- ﻿﻿比较 PostgreSQL vs MySQL 性能，PostgreSQL 执行复杂查询时表现良好，而MySQL 在 OLAP 和OLTP 系统中表现良好。
- ﻿﻿MySQL一般会将数据合法性验证交给客户；PostgreSQL在合法性难方面做得比较严格。比如MySQL里插入"2012-02-30"这个时间时，会成功，但结果会是"0000-00-00"；PostgreSQL不允许插入此值。
- ﻿﻿MySQL 由于广泛的应用和支持，拥有庞大的用户社区和丰富的工具包。PostgreSQL社区相对较小，但是开发者活跃，技术较为严谨，适合对安全性、标准化要求高的场景
- ﻿﻿MySQL 更适合快速开发和高并发的应用，而PostgreSQL 则更适合需要高度稳定性和高级特性的企业级应用

## 4.3 MyISAM vs InnoDB

### 4.3.1 基本区别

MyISAM 支持这个事务吧？

不支持，就是只有 InnoDB 支持。


虽然，MyISAM的性能还行，各种特性也还不错（比如全文索引、压缩、空间函数等）。但是，MyISAM 不支持事务和行级锁，而且最大的缺陷就是崩溃后无法安全恢复。

MySQL5.5版本之后，InnoDB 是 MySQL的默认存储引擎。

### 4.3.2 详细对比

#### 4.3.2.1 是否支持行级锁

MyISAM只有表级锁（table-level locking），而 InnoDB 支持行级锁（row-level locking）和表级锁，默认为行级锁

也就说，MyISAM一锁就是锁住了整张表，这在并发写的情况下是多么滴憨憨啊！这也是为什么 InnoDB 在并发写的时候，性能更牛皮了！

#### 4.3.2.2 是否支持事务

MyISAM 不提供事务支持。

InnoDB 提供事务支持，实现了 SQL 标准定义了四个隔离级别，具有提交（commit）和回滚（rollback）事务的能力。并且，InnoDB 默认使用的 REPEATABLE-READ（可重读）隔离级别是可以解决幻读问题发生的（基于 MVCC和 Next-Key Lock）。


#### 4.3.2.3 是否支持外键

MyISAM 不支持，而InnoDB 支持。

外键对于维护数据一致性非常有帮助，但是对性能有一定的损耗。因此，通常情况下，我们是不建议在实际生产项目中使用外键的，在业务代码中进行约束即可！

不过，在代码中进行约束的话，对程序员的能力要求更高，具体是否要采用外键还是要根据你的项目实际情况而定。

![[Pasted image 20251216152255.png]]
总结：一般我们也是不建议在数据库层面使用外键的，应用层面可以解决。不过，这样会对数据的一致性造成威胁。具体要不要使用外键还是要根据你的项目来决定。

#### 4.3.2.4 是否支持数据库异常崩溃后的安全恢复

MyISAM不支持，而 InnoDB 支持。

使用 InnoDB 的数据库在异常崩溃后，数据库重新启动的时候会保证数据库恢复到崩溃前的状态。这个恢复的过程依赖于 redo log。

#### 4.3.2.5 是否支持 MVCC

MyISAM 不支持，而 InnoDB 支持。

讲真，这个对比有点废话，毕竟 MyISAM 连行级锁都不支持。MVCC 可以看作是行级锁的一个升级，可以有效減少加锁操作，提高性能。

#### 4.3.2.6 索引实现不一样

虽然 MyISAM 引擎和 InnoDB 引擎都是使用 B+Tree作为索引结构，但是两者的实现方式不太一样。

InnoDB引擎中，其数据文件本身就是索引文件。相比 MyISAM，索引文件和数据文件是分离的，其表数据文件本身就是按 B+Tree 组织的一个索引结构，树的叶节点 data 域保存了完整的数据记录。

#### 4.3.2.7 性能有差别

InnoDB 的性能比 MyISAM更强大，不管是在读写混合模式下还是只读模式下，随着 CPU 核数的增加，InnoDB 的读写能力呈线性增长。MyISAM 因为读写不能并发，它的处理能力跟核数没关系。

## 4.4 事务ACID特性

有了解过这个事务的 4 个特性吗？

事务特性就是 ACID，一个是原子性，一个 C 是一致性，I 是隔离性，D 是持久性。

然后就是原子性的话就是通过它的一个，我想想。持久性是根据原子性，然后一致性还有隔离性来实现的，然后。原子性是根据是使用它的这个 undo log 来实现它的事务回滚，然后保证了它的原子性。

然后持久性是使用这个 Redo log 来，就是说如果说我这个突然宕机了，我还使用根据这个 Redo log，然后重新执行，你去保证我数据的持久。

然后像一隔离性的话，就使用了这个 MVCC 的一个方式，就是通过记录我这个事务的一事务的这个索引编号，然后跟我现在在跑的事务进行一个比较，大概是这样，然后来实现说我每个事务能看到数据是不一样的，然后保证我这个数据是能够并发执行，不会出现说那种脏读、幻读这些问题，大概是这样。

**ACID 特性详解**：

| **特性** | **说明** | **实现机制** |
|---------|---------|------------|
| **原子性（Atomicity）** | 事务要么全部成功，要么全部失败 | 使用 undo log 实现事务回滚 |
| **一致性（Consistency）** | 事务执行前后数据保持一致 | 通过原子性、隔离性、持久性共同保证 |
| **隔离性（Isolation）** | 并发事务之间相互隔离 | 使用 MVCC（多版本并发控制），通过事务编号比较，避免脏读、幻读 |
| **持久性（Durability）** | 事务提交后数据永久保存 | 使用 Redo log，宕机后根据 Redo log 重新执行 |

## 4.5 索引数据结构

### 4.5.1 常见索引数据结构

数据库索引能知道有哪些数据结构用来实现这个索引功能？

主要实现索引功能的话，我知道是 MySQL 是用 B+ 树，然后可能还会有什么 LSM 树，还有什么？应该是这个是另外一种数据库，会用到它，忘了是为什么用到有种特性。

但 B+ 树的话，它就是它能够只在叶子节点保存信息，然后叶子节点是相互关联的，方便它进行一个范围查询，然后它整体的一个可以处层级是相比于二叉树或者其他树是层级是比较低的。然后能够说就是磁盘 IO 次数减少，就是更少。

然后相比于它 B 树的话，因为它叶子节点它是那个中间，B 树是中间节点会存储信息，然后 B+ 树是中间存节点不存储信息，好像是我这个更平均，这个就是访问的时间会能够更平均，然后说叶子节点存能存储的数据也是更多的，大概是这样。

**索引数据结构**：

| **数据结构**        | **使用场景**                 | **特点**                                       |
| --------------- | ------------------------ | -------------------------------------------- |
| **B+ 树**        | MySQL 主要使用               | 只在叶子节点保存信息，叶子节点相互关联，支持范围查询，层级低，减少磁盘 IO       |
| **LSM 树**       | 其他数据库（如 LevelDB、RocksDB） | 适合写多读少的场景                                    |
| **B 树 vs B+ 树** | -                        | B 树中间节点存储信息，B+ 树中间节点不存储信息，访问时间更平均，叶子节点存储数据更多 |

**B+ 树的优势**：
1. 只在叶子节点保存信息，叶子节点相互关联
2. 方便进行范围查询
3. 层级低，减少磁盘 IO 次数
4. 相比 B 树，访问时间更平均，叶子节点存储数据更多

### 4.5.2 哈希索引 vs B+树

有哈希来实现索引的话，那它的那个查询效率也很快，对吧？但它跟 B+ 树比，B+ 树有什么优点呢？

#### 4.5.2.1 为什么 InnoDB没有使用哈希作为索引的数据结构？

我发现很多求职者甚至是面试官对这个问题都有误解，他们相当然的认为 MySQL 底层并没有使用哈希或者B树作为索引的数据结构。

实际上，不论是提问还是回答这个问题都要区分好存储引擎。像 MBMORY 引擎就同时支持哈希和B树。

哈希索引的底层是哈希表。它的优点是，在进行精确的等值查询时，理论上时间复杂度是 O（1），速度极快。比如 WHERE id = 123。

但是，它有几个对于通用数据库来说是致命的缺点：

1. ﻿﻿﻿不支持范围查询：这是最主要的原因。哈希函数的一个特点是它会把相邻的输入值（比如 id=100 和id=101 ）映射到哈希表中完全不相邻的位置。这种顺序的破坏，使得我们无法处理像 WHERE age>30 或 BETWEEN 100AND200这样的范围查询。要完成这种查询，哈希索引只能退化为全表扫描。
2. ﻿﻿﻿不支持排序：同理，因为哈希值是无序的，所以我们无法利用哈希索引来优化 ORDER BY 子句。
3. ﻿﻿﻿不支持部分索引键查询：对于联合索引，比如（co11，co12），哈希索引必须使用所有索引列进行查询，它无法单独利用 co11 来加速查询。
4. ﻿﻿哈希冲突问题：当不同的键产生相同的哈希值时，需要额外的链表或开放寻址来解决，这会降低性能。

鉴于数据库查询中范围查询和排序是极其常见的操作，一个不支持这些功能的索引结构，显然不能作为默认的、通用的索引类型。


#### 4.5.2.2 为什么 InnoDB没有使用B树作为索引的数据结构？

B 树和 B+树都是优秀的多路平衡搜索树，非常适合磁盘存储，因为它们都很"矮胖"，能最大化地利用每一次磁盘1/0。

但B+树是B树的一个增强版，它针对数据库场景做了几个关键优化：

1. 1/0效率更高：在B+树中，只有叶子节点才存储数据（或数据指针），而非叶子节点只存储索引键。因为非叶子节点不存数据，所以它们可以容纳更多的索引键。这意味着 B+树的"扇出"更大，在同样的数据量下，B+树通常会比B树更矮，也就意味着查找数据所需的磁盘1/0次数更少。

2. ﻿﻿﻿查询性能更稳定：在B+树中，任何一次查询都必须从根节点走到叶子节点才能找到数据，所以查询路径的长度是固定的。而在B树中，如果运气好，可能在非叶子节点就找到了数据，但运气不好也得走到叶子，这导致查询性能不稳定。

3. ﻿﻿对范围查询极其友好：这是B+树最核心的优势。它的所有叶子节点之间通过一个双向链表连接。当我们执行一个范围查询（比如 WHERE id >100）时，只需要通过树形结构找到 id=100 的叶子节点，然后就可以沿着链表向后顺序扫描，而无需再回溯到上层节点。这使得范围查询的效率大大提高。

## 4.6 索引设计原则

一般如果要建索引，你会一般喜欢，会怎么设计呢？就是在选哪些字段加索引的时候一般会注意什么事情？

哪些字段适合创建索引？

- ﻿不为 NULL的字段：索引字段的数据应该尽量不为 NULL，因为对于数据为 NULL 的字段，数据库较难优化。如果字段频繁被查询，但又避免不了为 NULL，建议使用o,1,true,false 这样语义较为清晰的短值或短字符作为替代。
- ﻿被频繁查询的字段：我们创建索引的字段应该是查询操作非常频繁的字段。
- ﻿被作为条件查询的字段：被作为 WHERE 条件查询的字段，应该被考虑建立索引。
- ﻿频繁需要排序的字段：索引已经排序，这样查询可以利用索引的排序，加快排序查询时间。
- ﻿被经常频繁用于连接的字段：经常用于连接的字段可能是一些外键列，对于外键列并不一定要建立外键，只是说该列涉及到表与表的关系。对于频繁被连接查询的字段，可以考虑建立索引，提高多表连接查询的效率。

📌 面试官听到 **aggregation**，基本就会默认你是真的用过 ES。



> **聚合检索（Aggregation）就是：**
> **在"搜索结果的基础上"，对数据做统计、分组、计算，而不是返回具体文档。**


### 5.4.1 Terms 聚合（分组统计）


👉 **最常用，等价于 GROUP BY**

例：按品牌统计商品数

```json
"aggs": {
  "brand_count": {
    "terms": {
      "field": "brand"
    }
  }
}
```
返回的结果

```json
"buckets": [
  { "key": "海底捞", "doc_count": 120 },
  { "key": "蜀大侠", "doc_count": 80 }
]
```


### 5.4.2 指标聚合（数值计算）

👉 统计数值（sum / avg / max / min）
例：统计平均价格
```json
"aggs": {
  "avg_price": {
    "avg": {
      "field": "price"
    }
  }
}
```

**品牌分组 + 统计平均价格**
```json
"aggs": {
  "brand": {
    "terms": {
      "field": "brand"
    },
    "aggs": {
      "avg_price": {
        "avg": {
          "field": "price"
        }
      }
    }
  }
}
```



### 5.4.3 一个完整的「搜索 + 聚合」例子（重点）

```json
POST /product_index/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "product_name": "牛肉火锅" } }
      ],
      "filter": [
        { "range": { "price": { "gte": 50, "lte": 200 } } }
      ]
    }
  },
  "aggs": {
    "brand_group": {
      "terms": {
        "field": "brand"
      }
    }
  },
```

- hits：搜索到的商品列表（分页展示）
- aggregations.brand_group：
    - 当前条件
    - 每个品牌有多少商品


## 5.5 数据同步

### 5.5.1 数据同步方式

数据是怎么进 ES 的？（非常容易被追问）

增量同步即对MySQL 中新增，修改，删除的数据进行同步：

- ﻿﻿同步双写：修改数据时同步到 Elasticsearch。这种方式性能较差、存在丢数据风险且会耦合大量数据同步代码，一般不会使用。
- ﻿﻿异步双写：修改数据时，使用MQ异步写入Elasticsearch 提高效率。这种方式引入了新的组件和服务，增加了系统整体复杂性。
- ﻿定时器：定时同步数据到 Elasticsearch。这种方式时效性差，通常用于数据实时性不高的场景
- ﻿﻿binlog 同步组件Canal（推荐）：使用Canal 可以做到业务代码完全解耦，API 完全解耦，零代码实现准实时同步，Canal 通过解析 MySQL 的 binlog 日志文件进行数据同步。

## 6.1 重载和重写

你能讲就是重载和 Java 里边重载和重写的区别吗？

重载的话主要是一个，我想我要分一下两个区别。

重载的话，应重载的话是这个子类继承父类的时候，它们使用一个相同，就是父类可能提供接口，或父类提供这个一个函数，然后子类实现父类的时候，我可以把这个函数进行一个重载，重试，重载，然后重载完。他要重写。是重写，override 重写，我记得我加了那个 override 那个注释在这重写，然后重写，然后它就会，就是它会根据，就是在具体实现的，在具体扩这个函数的时候，因为我是给一个父类子，因为我一般它会，我们是调用父类指针，然后它会根据子，具体的子类的实现来此来这个扩不同的一个函数，这是这个重写。

然后重载的话是我相同的函数，然后它有不同的函数签名，就有不同的那个参数，然后不同，然后如果我使用不同的参数。不同的这个函数签名的就同样的名字，但不同的函参数。就有不同的一个特点表现出来，大概这样。

**重载 vs 重写**：

| **特性** | **重载（Overload）** | **重写（Override）** |
|---------|-------------------|-------------------|
| **定义** | 相同函数名，不同的函数签名（参数不同） | 子类继承父类，重写父类的方法 |
| **参数** | 必须有不同的参数列表 | 参数列表必须相同 |
| **返回值** | 可以不同 | 必须相同或兼容 |
| **调用方式** | 根据传入的参数类型和数量决定调用哪个方法 | 根据对象的实际类型决定调用哪个方法（多态） |
| **注解** | 不需要注解 | 可以使用 @Override 注解 |

**重载示例**：
```java
public void method(int a) { }
public void method(int a, int b) { }  // 重载：参数不同
```

**重写示例**：
```java
class Parent {
    public void method() { }
}
class Child extends Parent {
    @Override
    public void method() { }  // 重写：子类重写父类方法
}
```

## 6.2 JVM内存分区

我不知道对这个 Java，那个 JVM 它的这个内存分区有了解过吗？

这个很久之前了解过，现在又忘了。它应该是一个有新生代，然后有一个老生代，然后它会根据。据根据这个对象，因为他好像是他是经过不同的时间就会回收，就回收一次对象，然后他会根据这个对象存活的这个回收的轮次，然后把这个对象放到可能就把新生代对象，如果他存活轮次比较长的话，就放到这个老生代，然后如果不是的话，他就放在那，具体有什么？有什么优点我忘了。

**JVM 内存分区**：

| **区域** | **说明** | **特点** |
|---------|---------|---------|
| **新生代** | 新创建的对象存放在这里 | 经过多次 GC 后，存活的对象会晋升到老生代 |
| **老生代** | 存活时间较长的对象存放在这里 | 对象存活轮次比较长 |

**垃圾回收机制**：
- 根据对象存活的回收轮次决定对象存放位置
- 存活轮次长的对象会从新生代晋升到老生代
