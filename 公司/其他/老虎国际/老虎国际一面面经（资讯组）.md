## 2. 算法题 — 全排列

### 题目：

给定一个数组，输出所有全排列。

### 我的思路：

回溯法。每一轮从输入数字中选一个数，记录已选，下一轮从剩余的数中继续选。递归结束条件：已选数量等于数组长度。

### 结果：

一次通过，输出正确。

## 3. ARM 实习 — Profiling 工具开发

### Q：预热后启动分析，预热是怎么做的？

**我的回答：** 测试数据库在预热后高频写场景下的 TPS。原工具没有预热参数，我加了一个新的命令行参数，让采样线程 sleep 指定分钟数后再进行分析，避免预热阶段数据影响最终结果。具体是让分析采样结果的线程延迟启动，采样线程本身不受影响。

### Q：收集完整调用栈信息，是原来没有收集吗？

**我的回答：** 原来有调用栈信息（perf 自带 call chain），代码中也有用到，但问题是：工具在分析某一行代码时，没有考虑这行代码的调用栈上下文。在 MySQL 这种复杂系统中，同一行底层代码可能在不同场景下被调用（如读密集 vs 写密集 vs CPU 密集），需要区分不同调用路径。

**具体实现：** 通过 perf 的 call chain 获取每条采样的 PC（指令地址），结合程序的 debug 信息将 PC 映射到具体代码行，以此还原调用栈。以被分析的代码行作为 key，聚合其所有调用栈信息，统计每种不同调用路径的采样次数。

### Q：调用栈信息跟原来的指令能适配吗？

**我的回答：** 能适配。本质上只是在原有的采样结果上增加了调用栈维度的聚合。每采样到一行代码就获取其调用栈，记录在日志中，最终按代码行聚合，统计各调用路径出现的次数。

### Q：在项目中发现了哪些性能瓶颈？

**我的回答：** 主要定位到 CRC 校验是热点代码。MySQL 不管是写 binlog、redo log 还是 undo log，只要刷页落盘都会走 CRC 校验流程。x86 架构下 Intel 有专门的 CRC 加速指令，但 ARM 上没有，存在性能 gap。

工具分析结果显示：如果能优化这个 CRC 函数 20% 的执行速度，整体 TPS 能提升约 40%。我的工作主要是为团队提供佐证，证明这行代码确实是热点且优化空间大。

## 4. 西门子实习 — RAG 系统优化

### Q：问答系统是你独立开发的吗？

**我的回答：** 不是。进去时整体框架已搭好（文档提取 → embedding → 问答），我的工作是先评测系统准确率（初始 76%），然后负责优化。

### Q：怎么提升准确率的？

**我的回答：** 分几个阶段逐步优化：

**1. 修复文档处理缺陷：** 原有 XML 转 Markdown 的流程有细节问题，导致信息缺漏，先修复这些转换 bug。

**2. 更换 embedding 模型：** 发现核心问题是相关文档检索不到。换用千问三 embedding 模型（能力更强但向量维度更高），原来的 Elasticsearch 不支持高维向量，于是迁移到 Milvus。原有的 embedding 开源代码已停止维护、bug 很多，我重新搭建了整套 ETL 流水线（文档提取 → embedding → 落库），支持不同模型和数据库适配，也加了增量插入等功能。这步优化到 80%+。

**3. 多语言 Query Rewrite：** 文档是中文，用户习惯用英文专业术语提问，导致检索 gap。加了 query rewrite 环节，将用户问题改写为多语言版本后分别检索。

**4. 混合检索策略：** 纯语义检索容易忽略低频专业术语。加入 BM25 关键词检索 + sparse 检索（融合语义和关键词），三种检索方法混合后再用 reranker 重排序，最终喂给大模型生成回答。

**最终效果：** 准确率从 76% 提升到 92%。

### Q：BM25 关键词检索算法大概怎么实现的？

**我的回答：** 主要调 Milvus 的 API。底层原理是先对文档分词，然后通过公式计算：考虑词在所有文档中的稀有度（IDF）和文档长度的归一化。频繁出现的词不一定是关键词，少见的词如果出现在很长的文档里权重也会被调低。根据公式算出得分存入数据库用于检索。

### Q：还做过其他工作吗？

**我的回答：** 做了一个前端 AB 测试页面。因为评测系统用大模型打分，leader 不完全信任，所以做了一个前端页面让测试人员人工评测，然后对比人工评分和大模型评分的一致性。

## 5. 用友实习 — 数据迁移与 MySQL

### Q：大概讲述一下你做的工作？

**我的回答：** 新旧系统数据库的 table schema 和数据格式不同，我开发了一个数据迁移同步系统。

**迁移链路：旧 Java 系统（聚合数据）→ gRPC → Python（清洗/批处理）→ gRPC → 新 Java 系统（校验 + 插入 PgSQL）**

- **旧系统端：** 数据不是简单 SELECT 能拿到的，需要旧系统的 Java 业务逻辑做多表关联、枚举值转换等聚合，通过 gRPC 接口暴露聚合后的数据。
- **Python 中间层：** 作为 gRPC Client 拉取聚合数据，做 schema 映射和格式清洗（正则处理、数据类型转换、批处理组装），转换成新系统需要的格式。
- **新系统端：** 处理后的数据通过 gRPC 传给新系统的 Java 服务，走新系统已有的写入逻辑（字段合法性校验、关联关系检查）后插入 PostgreSQL，避免绕过业务层直连数据库导致脏数据。

保证写入幂等，事务失败能回滚。先小批量测试验证安全性，再进行大批量同步。

### Q：迁移过程中有没有遇到数据类型不匹配等冲突？

**我的回答：** 有。主要是格式化处理时定义不够清楚，导致数据类型不对，gRPC 接口报错无法正常插入。因为系统设计了事务回滚机制，所以不会造成脏数据。处理前也对数据表做了快照备份，确保安全。整体都是小问题，根据报错改代码即可。

### Q：迁移过程中系统是否提供服务？

**我的回答：** 不提供，是离线迁移。

### Q：Redis 缓存会不会出现和数据库数据不一致的情况？

**我的回答：** 缓存的是读多写少的数据（员工基本信息等不常改动的数据），正常不会有一致性问题。做法是用双删策略：先删数据库数据，再删 Redis 数据。因为 PostgreSQL 写入时间比 Redis 长，这个顺序能大概率保证一致性。人力系统没有热点数据，实际没遇到过一致性问题。

### Q：有没有优化过慢查询？

**当时的回答：** 有，主要是迁移后做数据比对时的 JOIN 语句比较慢。看慢查询日志发现耗时 JOIN，加了联合索引、调大 join buffer、避免 SELECT *。

**改进后的回答：** 有两类问题，一类是单条慢 SQL，一类是 N+1。

**场景一：迁移后数据比对 — 单条慢 SQL**

迁移完需要验证新旧数据一致，写了一条全表 JOIN + SELECT * 做 hash 比对，很慢。

排查方式：开启 PG 慢查询日志（`log_min_duration_statement`），直接看到这条耗时 JOIN。用 `EXPLAIN ANALYZE` 看执行计划，发现全表扫描 + 回表。

优化手段：
1. **加联合索引：** 让 JOIN 匹配走索引，避免全表扫描。
2. **调大 join buffer：** 没索引时走 Block Nested-Loop Join，join buffer 装外层驱动表的数据，扫一遍内层被驱动表时和 buffer 里的多行在内存中批量比较。buffer 越大，被驱动表从磁盘被扫描的轮次越少。
3. **避免 SELECT \*：** 只查 ID + 需要比对的字段，走覆盖索引避免回表，减少返回数据量。

**场景二：报表查询 — N+1 问题**

HR 查看招聘进度报表（每个岗位的候选人数量、各阶段人数统计）。代码里先查岗位列表（1 次），再循环每个岗位查候选人统计（N 次），50 个岗位就是 1 + 50 = 51 次查询。单条只要 2ms，但 51 次网络往返累加起来 100ms+。

排查方式：单条不慢，慢查询日志里看不到。通过 `pg_stat_statements` 统计发现同一类参数化 SQL 的 calls 高达几千次，总耗时很高，一看就是 N+1 模式。结合代码审查，定位到循环里的数据库查询。

优化手段：把循环里的 N 次查询改写成一条 JOIN + 聚合 SQL，一次查询搞定整个报表。在子表上加联合索引（`position_id, stage`）让 JOIN 和聚合走索引。101 次查询 → 1 次查询。

### Q：有没有遇到连接池报错或连接断开等问题？

**当时的回答：** 没有，数据量不大，没遇到高 QPS 场景下的连接问题。

**改进后的回答（如果追问怎么解决）：** 常见连接池问题分两类：

**1. 连接池耗尽（拿不到连接）：** 连接池大小固定，并发请求超过上限后排队超时报错。常见原因是慢查询占着连接不释放、代码异常分支没有 finally 关闭连接导致连接泄漏、或连接池配置太小。解决思路：先排查慢查询和连接泄漏（根因），再调整连接池最大连接数和等待超时参数，并发量确实大的话可以引入 PgBouncer 做连接复用。

**2. 连接断开（拿到死连接）：** 连接池里缓存的连接因网络抖动、数据库重启、防火墙超时等原因已断开，但连接池不知道，把死连接给了应用。解决思路：开启连接池空闲检测（定期 `SELECT 1` 探活剔除死连接），设置连接最大空闲时间（超时自动关闭），应用层做重试（拿到连接执行失败时丢弃并重新获取）。

### 补充准备 — 装饰器

装饰器是一个函数，接收一个函数作为参数，返回一个新函数，在不改动原函数代码的情况下给它加功能。`@log` 本质上就是 `func = log(func)`。常见用途：打日志、计时、权限校验、重试逻辑。Web 框架里到处都是（路由、认证）。

### 补充准备 — GIL（Global Interpreter Lock）

CPython 有一把全局锁，同一时刻只允许一个线程执行 Python 字节码。原因是 CPython 的引用计数内存管理不是线程安全的，GIL 是最简单的保护方式。

**影响 CPU 密集型任务：** 4 核 CPU 开 4 个线程，同一时刻只有 1 个在跑，多线程不会比单线程快。解决方案是用多进程（`multiprocessing`），每个进程有自己的 GIL。

**不影响 IO 密集型任务：** 线程等 IO（网络、磁盘、数据库）时会释放 GIL，其他线程可以拿锁继续执行。所以爬虫、Web 后端这种 IO 密集场景多线程还是有用的，或者用更轻量的协程。

### 补充准备 — 协程

**协程的定义：** 一个可以暂停和恢复的函数。普通函数从头跑到尾不能停，协程可以执行到一半主动暂停（让出控制权），等条件满足后从暂停的地方继续。

**Python 协程使用方式：** 用 `async def` 定义协程函数，`await` 等待异步 IO 时让出控制权，`asyncio.gather()` 并发调度多个协程。核心是一个事件循环（event loop）在单线程里调度所有协程。

**关键语法：**
- **`with`：** 自动资源收尾。退出 with 块时自动调用 `__exit__`（关闭文件/连接等），哪怕异常也会关。
- **`await`：** 告诉协程"这里等 IO，我先让出去，别的协程先跑，IO 完成后切回来"。只能在 `async def` 里用。
- **`yield`：** 让函数变成生成器，执行到 yield 暂停并返回一个值，下次调用从暂停处继续。核心价值是惰性求值——要一个给一个，不一次性加载到内存。`yield` 是最原始的协程思想，`async/await` 只是暂停原因从"产出值"变成"等 IO"。

### 补充准备 — Python 协程 vs Go 协程（goroutine）

| | Python 协程 | Go goroutine |
|--|------------|-------------|
| 启动方式 | `async def` + `await` + 事件循环 | `go func()` 一个关键字 |
| 调度 | 单线程，手动 await 让出 | 多线程，运行时自动调度（GMP 模型） |
| 多核 | 不能，受 GIL 限制 | 能，天然多线程并行 |
| 代码侵入 | async 传染（用了一个 async，调用链一路都要 async，同步异步生态割裂） | 无侵入，普通函数前面加 go 就行 |
| 阻塞影响 | 忘写 await 或调同步阻塞函数会卡住整个事件循环 | goroutine 阻塞时调度器自动把其他 goroutine 调到别的线程 |
| 适合 | IO 密集型，中等并发 | IO + CPU 密集型，高并发 |

面试官团队用 Python + Go，IO 密集型服务都用协程。Python 协程够用但有 GIL 和传染性限制，Go 在这方面更自然，这也是很多团队 Python 做快速开发、Go 做高性能服务的原因。
