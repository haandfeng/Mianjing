## 2. 西门子实习 — RAG 系统深度讨论

### Q：整个 RAG 系统是你搭建的吗？

**我的回答：** 不是，进去时团队已完成基础链路（用户提问 → 文档检索 → 大模型回答）。我的工作是在已有系统上做优化。

### Q：优化之前是什么检索方式？

**我的回答：** 优化前只有稠密向量检索——把文档转成语义向量，用 query 向量和文档向量算余弦相似度，返回最相关的文档。

### Q：为什么要加 BM25 关键词检索？

**我的回答：** 用户会问到 CT 操作的术语关键词，可能就几个中文字，但文档 chunk 有 500 个 token。术语本身语义转换不好，加上 token 数量悬殊，稠密向量对这类短关键词的匹配效果不好。引入关键词检索可以弥补这种场景。

### Q：稠密向量用的什么模型？

**我的回答：** 一开始用 BGE-M3（1024 维），后来发现效果不够好，换成了千问 3（4096 维）。但 Milvus 当时不支持这么高维度的向量，所以为了适配重构了 ETL pipeline，改造了向量数据库的存储方案。

### Q：Reranker 用了哪些？是同时用还是选了其中一个？

**我的回答：** 最终只用了 BGE-M3 Reranker。测试了三种方案：

1. **RRF（Reciprocal Rank Fusion）：** 每个文档在各检索方式中有排名，取排名倒数累加得分。最简单最快，但实测排序效果不好，不能有效把相关文档推到前面。
2. **千问 3 Reranker：** 效果可以但慢，约 0.6s。
3. **BGE-M3 Reranker：** 约 0.2s，速度是千问 3 的三倍。体感上两者准确率差距不大，但速度优势明显，最终选了 BGE-M3。

### Q：Embedding 模型和 Reranker 模型底层区别是什么？⚠️（回答不好）

**当时的回答：** Embedding 模型走 BERT 编码器架构，输入文档输出向量；Reranker 更像 GPT 解码器架构做语义理解。体感区别是输出不同——一个返回向量，一个返回相关性分数。

**改进后的回答：**

两者的核心区别在于建模方式：

**Embedding 模型（Bi-Encoder）：** Query 和 Document ==分别独立编码==成向量，再通过余弦相似度等度量计算相关性。优势是 Document 向量可以离线预计算存入向量数据库，在线只需编码 Query 然后做 ANN 检索，速度极快，适合从海量文档中召回候选集。

**Reranker 模型（Cross-Encoder）：** 将 Query 和 Document ==拼接在一起==作为输入，让模型内部的 attention 机制在 query token 和 document token 之间充分交互，直接输出一个相关性分数。因为有了 token 级别的交叉注意力，语义理解更精细，排序更准确。但代价是每个 (query, document) 对都要过一遍模型，无法预计算，所以只适合对召回后的少量候选（如 top-20）做精排。

**本质区别不是 BERT vs GPT，而是 Bi-Encoder vs Cross-Encoder。** 实际上大多数 Reranker（包括 BGE Reranker）也是基于 BERT 类编码器架构，只是输入方式不同。

> 详见 [[专栏/Agent/Embedding模型与Reranker模型的区别]]

### Q：既然 Embedding 也能算余弦相似度得到分数，能不能用 Embedding 做 Reranker？劣势是什么？⚠️（回答不好）

**当时的回答：** 可以做，但训练目的不同，Embedding 模型更轻量，Reranker 模型训练目标就是生成精确相关性分数，精细度更高。

**改进后的回答：**

技术上可以——把候选文档逐个编码，和 query 向量算余弦相似度来排序。但劣势明确：

1. **精度差距：** Bi-Encoder 的 query 和 document 独立编码，无法做 token 级别的交叉注意力。Cross-Encoder 把两者拼接输入，attention 层能直接建模 query 中某个词和 document 中某个词的关系，语义匹配更精细。这就是为什么 Reranker 在精排阶段效果更好。
2. **信息压缩损失：** Embedding 把整个文档压缩成一个固定维度的向量（如 1024 维），不可避免地丢失细节信息。Cross-Encoder 直接在原始 token 序列上做推理，保留了全部信息。
3. **训练目标不同：** Embedding 模型的训练目标是让语义相似的文本在向量空间中靠近（对比学习），优化的是向量空间的全局结构；Reranker 的训练目标是直接判断一对 (query, document) 的相关程度（分类/回归），优化的是排序精度。

**总结：Embedding 适合从百万文档中快速召回 top-k，Reranker 适合对 top-k 做精排。两者是互补关系，不是替代关系。**

> 详见 [[专栏/Agent/Embedding模型与Reranker模型的区别]]

### Q：文档摘要检索和 Query Refine 是怎么做的？

**我的回答：**

**Query Refine：** 解决中英文术语 gap。用户倾向用英文问 CT 术语（操作界面是英文），但文档以中文为主。用小模型将用户 query 改写成语义相似的不同语言版本，用多个 query 分别检索再汇总结果送入重排。

**文档摘要检索：** Dense embedding 本质是取 Transformer 的 CLS token（或 head token），当文档 chunk 过长（如 500 token），关键信息被稀释，向量质量下降。思路是用 LLM 把 500 token 的文档压缩成约 50 token 的摘要，用摘要向量作为索引。检索时 query 和摘要的 token 数量级相近，语义匹配更精准。类似 LlamaIndex 的 Document Summary Index 方案。实测效果不如预期，最终没采用。

### Q：三个评估指标分别是什么含义，怎么评估的？

**我的回答：**

- **Correctness（正确率）：** 有标准答案（测试中心提供），将系统回答和标准答案都喂给 LLM 做 judge，判断是否一致、有无矛盾或遗漏。
- **Groundedness（忠实度）：** LLM 判断回答是否忠于检索到的文档内容，有无幻觉。
- **Relevance（相关性）：** LLM 判断回答是否真的在回答用户的问题。

## 3. 模型与算法基础

### Q：Transformer 架构、大模型训练流程了解吗？

**我的回答：** 坦诚说这方面看得比较少，主要偏工程方向，算法学习较少。

### 面试官反馈：

岗位面向算法方向，不仅需要了解 RAG 链路，还需要深入到链路中每个环节涉及的模型算法和架构。具体包括：

- **向量模型微调：** 针对特定领域（如西门子 CT）对 Embedding 模型做微调，而不只是引入 BM25 互补。
- **Reranker 微调：** 开源 Reranker 在垂直领域效果不一定够好，需要针对性训练。
- **生成模型训练：** RAG 的 G（Generation）阶段，除了用好的模型，还可能自己训练模型，结合强化学习微调使输出符合业务规范。
- **评估模型迭代：** LLM-as-Judge 本身也需要迭代优化，确保评估准确性。

> [!warning] 关键差距
> 面试官明确指出：整个 RAG 链路上涉及模型的地方，都可能需要算法同学去优化。纯工程落地（套用开源方案）不够，需要具备分析问题 → 训练/微调模型的能力。
