## 2. 西门子实习 — RAG 问答系统

面试官对西门子项目问得最深，持续追问了很多细节。

### Q：能详细展开一下在西门子的项目吗？

**我的回答（梳理版）：**

进去时 RAG 已搭好，但准确率只有 **76%**。排查后发现主要是文档检索不到的问题，于是分几步优化：

1. **换 Embedding 模型：** 5月千问三发布，切换到千问三的 Embedding 模型，向量维度更高，语义更准，但原来的向量数据库不支持，重构了整个 ETL pipeline（文档转换 → Embedding → 落库）。准确率提升到 **80%+**。
2. **Query Refine：** 用户用英文问，文档是中文，存在语言 gap。用大模型对 query 生成语义相近但语言不同的多个版本，扩大检索覆盖面。
3. **混合检索：** 原来只有 dense 向量检索，加入 BM25 关键词检索 + sparse 向量检索，三路融合后再经 Reranker 重排，取最相关文档喂给 LLM 生成回答。
4. 最终准确率提升到 **92%**，达到上线标准。

### Q：这个项目的用户是谁？

CT 机操作人员。公司有在线培训平台，用户在上面看操作手册和培训视频，我们给它加了一个 RAG 问答客服功能，方便用户直接提问。

### Q：文档数据源是什么？

公司内部的 CT 机操作手册，由文档组维护，定期更新后交给我们做 Embedding。

### Q：文档转换那一步是在做什么？

手册原本是 HTML 格式，HTML 标签多、耗 token、语义噪声大。我们把它转成 Markdown，更简洁、更语义化，embedding 效果更好。

### Q：准确率是怎么度量的？

测试中心和产品经理基于用户反馈整理了约 **300 道题**（200 道中文、100 道英文），每道题有标准答案。把问题喂给 RAG 系统生成回答，再用 LLM 对比 RAG 回答与标准答案，判断是否存在缺漏或不一致，统计正确率。按 **Correctness、Relevance、Groundedness** 三个维度打分，用 LangSmith 做追踪评估。

### Q：团队组成是什么样的？你承担什么角色？

团队共约 4 名开发，但他们主业是 CT 机操作软件研发，同时还在做 Agent 控制 CT 机的方向，精力有限。RAG 系统基本由我一人负责，向 mentor 汇报进展，周会向全组同步。

### Q：有涉及大模型选型吗？

有。公司是医疗企业，数据不能上传到外部服务，必须用本地部署的开源模型。我们测试了千问、LLaMA、DeepSeek 小模型，也测了 GPT（闭源，作为基线对比）。最终选了 **千问的 120B 开源模型**（7 月份开源的版本），在效果和本地可部署性上综合最优。

### Q：用户访问方式是什么？

访问一个内部 Web 站点，在上面直接提问。

### Q：Memory 怎么管理？

做得比较轻量。对话超过 **10 轮**时会触发 Summary 路径，把历史对话压缩成摘要再传给 LLM。因为我们主要测试单轮问答，多轮对话的幻觉问题比较严重（模型会把历史文档的相关内容错误地拿来回答当前问题），所以没有做深度的多轮优化。

### Q：有权限管理吗？

有按机型分权限。不同机型的文档存在独立的表里，用户查询时只能指定某些机型，系统通过元数据过滤，只检索对应机型的文档。如果要做更细粒度的用户级权限，也很好实现——给每个文档 chunk 加上权限元数据，检索时按元数据过滤即可。

这也是我重构 pipeline 的原因之一：原来的开源方案不支持按 ID 删除或增量更新，只能全量重建，业务上不可接受，所以给每个文档分配唯一 ID，实现了定向删除和增量更新。

### Q：准确率还想往上提的话，会碰到哪些瓶颈？

主要瓶颈是**用户问题质量**。用户可能问非常短的句子（如"这是什么？"），很少的 token 加上表述差异，导致语义检索和关键词检索都难以精准匹配文档。

后续改进思路是引入 **ReAct Agent 模式**：把 RAG 从线性 pipeline 改成循环推理。如果当前检索结果不足以回答问题，让 LLM 自主判断如何改写 query，重新检索，反复 loop，直到能够回答为止。

### Q：整个项目最大的挑战是什么？

从零进入一个全新的领域（LLM/RAG 工程），组内也没有人熟悉这个方向，所有技术选型、优化路径都靠自己阅读 LangChain、LlamaIndex 等框架的文档和 Paper 来摸索。这是第一次真正意义上自主解决工程问题，而不是按已有框架执行任务。

## 3. 技术问题

### Q：PostgreSQL 和 MySQL 怎么选型？

**我的回答：**

大多数场景下 PostgreSQL 是 MySQL 的上位替代：
- 支持更丰富的数据类型（JSON、数组、自定义类型等）。
- Join 性能更强：支持 Hash Join 和 Sort-Merge Join，MySQL 旧版本只有 Nested Loop，新版本才引入 Hash Join。
- MVCC 实现更完善，事务隔离性更强。

选 MySQL 的理由：国内社区更活跃、技术沉淀丰富、遇到问题网上解法多，对研发能力相对弱的团队更友好。

**改进后的回答（补充）：**

还需要考虑 **运维能力与场景匹配**。PostgreSQL 功能强大，但 autovacuum、连接池（需搭配 PgBouncer）等运维细节要处理好，否则会踩坑。如果团队 DBA 经验主要在 MySQL 上，盲目迁移反而引入风险。

### Q：读多写少的场景，PostgreSQL 有什么坑？

**当时的回答：** 没能答出来。

**改进后的回答（课后补充）：**

PostgreSQL 的 MVCC 实现方式是**保留旧版本行（dead tuple）在表内**，不像 MySQL 的 undo log 存在独立空间。对于读多写少的场景，虽然写入少，但每次更新都会产生 dead tuple，需要 **VACUUM** 来回收。

潜在坑点：
- **表膨胀（table bloat）：** autovacuum 不够及时时，dead tuple 堆积，表物理大小持续增长，全表扫描变慢。
- **索引膨胀（index bloat）：** 索引也会保留对 dead tuple 的引用，空间浪费且查询性能下降。
- **长事务阻塞 VACUUM：** 只要有一个长事务存在，VACUUM 就无法回收该事务启动前的所有 dead tuple。读多写少场景下容易出现长时间持有 snapshot 的只读事务，反而阻塞 VACUUM 运行。
- **对策：** 合理配置 autovacuum 参数（`autovacuum_vacuum_scale_factor`、`autovacuum_vacuum_cost_delay`），避免长事务，定期监控表和索引膨胀情况。

### Q：Redis 主从同步怎么做？

**我的回答：**

分两个阶段：

**全量同步（初始化）：** 从节点向主节点发起同步请求，主节点生成 **RDB 快照**，通过网络传输给从节点加载，从节点完成全量数据初始化。

**增量同步（持续）：** 主节点执行写命令后，通过 **Replication Buffer** 将命令异步传播给从节点，从节点按序重放命令，保持数据同步。Redis 通过 `repl_backlog_buffer`（环形缓冲区）支持断线重连后的增量同步（`PSYNC`），避免重新全量同步。

**高可用：** 哨兵（Sentinel）模式监控主节点健康状态，主节点故障时自动选举新主节点，并更新从节点的复制目标。

### Q：Redis 的 value 非常大时（大 key）会有什么坑？

**我的回答：**

- **阻塞主线程：** Redis 单线程处理命令，删除大 key（如大 Hash、大 List）是 O(n) 操作，会长时间阻塞所有其他命令。解决方案：用 `UNLINK`（异步删除）代替 `DEL`。
- **网络传输开销：** 大 value 导致单次 GET 占用大量网络带宽，在高 QPS 下会成为瓶颈。
- **主从同步延迟：** 大 key 的同步数据量大，可能导致主从延迟升高，从节点数据短暂滞后。
- **内存分配问题：** 大对象分配和释放会加剧内存碎片化。

**发现大 key 的方法：** `redis-cli --bigkeys` 扫描，或定期用 `MEMORY USAGE key` 检查。

### Q：常见的负载均衡策略有哪些？

**我的回答：**

- **轮询（Round Robin）：** 请求依次分配给各服务器，适合服务器性能均匀的场景。
- **加权轮询（Weighted Round Robin）：** 性能好的服务器分配更高权重，接收更多请求。
- **最少连接（Least Connections）：** 将新请求发给当前活跃连接数最少的服务器，适合请求处理时间差异大的场景。
- **一致性哈希（Consistent Hashing）：** 将哈希空间映射为一个环，服务器节点均匀分布，请求按 hash(key) 顺时针找最近节点。扩缩容时只影响相邻节点的数据，不会全量重映射。

### Q：什么场景特别适合用一致性哈希？（追问）

**当时的回答：** 只回答了"弹性扩缩容场景"，过于运维视角。

**面试官引导方向：** 让同一用户的请求始终打到同一节点的场景。

**改进后的回答：**

**有状态会话保持**是最典型的场景。例如：
- 服务端存储了用户 Session（未做分布式 Session），需要同一用户请求打到同一台机器，否则 Session 丢失。
- **WebSocket / SSE 长连接：** 连接是有状态的全双工通道，必须保持在同一节点。面试中我提到了 MCP Server 走 SSE 协议的场景，连接建立后所有后续消息都必须发到同一个服务实例。
- 分布式缓存（如 Redis 集群）：用一致性哈希决定 key 落在哪个 shard，扩缩容时减少 key 迁移量。

## 4. 算法题

### 题目一：矩阵连通块计数

给定一个 0/1 矩阵，上下左右相连的 1 视为一个连通块，输出每个连通块内 1 的数量。

**我的思路：** 经典岛屿问题，DFS 遍历。遇到未访问的 1，进入 DFS 并将访问过的节点标记为 2，同时计数。返回每次 DFS 的计数即为该连通块大小。

**结果：** 实现过程中把 `i`、`j` 写混了，debug 后找到并修正，输出正确。

### 题目二：翻转一个 0 使最大连通块最大（进阶）

在题目一基础上，允许把任意一个 0 翻转为 1，问操作后最大连通块能有多大。

**我的思路（口述）：**

1. **染色预处理：** 先跑一遍 DFS，对每个连通块内的所有格子，把值染成该连通块的大小（用独立 ID 区分面积相同的不同连通块，避免混淆）。
2. **遍历所有 0：** 对每个 0 格子，查看其上下左右四个邻居的连通块 ID（用 Set 去重，避免重复统计同一块），将这些连通块大小求和再加 1（翻转当前 0），取所有 0 中的最大值。

**实现：** 用独立 `visited` 数组代替原地标记（避免面积相同时的歧义），用 Set 去重相邻的相同连通块，最终输出正确（4）。
