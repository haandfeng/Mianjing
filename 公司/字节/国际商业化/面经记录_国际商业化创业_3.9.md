## 2. ARM 实习 — Profiling 工具（Coz / off-CPU）

### Q：能具体展开说一下你在做的 off-CPU profiling 工作吗？

**我的回答：**
- 传统 profiler（如 perf）只能采样 on-CPU 的信息，无法捕获线程 sleep、锁等待、IO 等待的时间。
- Coz（Causal Profiling）的核心思想是"虚拟加速"——不能直接加速某段代码，所以通过给其他线程加 sleep/delay 来模拟加速效果，量化某段代码对整体吞吐量的影响。
- 这来源于 2015 年 SOSP best paper，AMD 已在 LCM 中实现过。
- 我的工作是将其移植到 ARM 平台，并加入 off-CPU 功能：在线程被 schedule out 和重新 schedule in 时分别加时间戳，通过计算等待时间除以采样频率，得到该段时间内的等效采样数，从而实现对 IO/锁等待的虚拟加速。

## 4. 用友实习 — SQL 优化

### Q：N+1 查询是什么？怎么解决的？

**我的回答：**
- 比如先查有多少个部门，再对每个部门单独发一次查询，50 个部门就是 50+1 次查询。
- 用 JOIN 语句合并为一次查询解决。

**改进后的回答：**
- 先查部门列表拿到 N 条记录，然后循环每条记录去查子表，就变成了 1 + N 次查询。解决方式是用 JOIN 或 IN 子句合并成一次查询。

### Q：效果有 Benchmark 吗？

**我的回答：**
- 通过 pg_stat_statements 统计日志发现该 SQL 出现频率高、累计耗时长，优化后没有特别关注具体数值，但确实从统计排名前列消失了。

**改进后的回答：**
- 优化前是 N+1 查询，单条 SQL 很快大概 3-4ms，但 50 个部门要发 51 次，累计一个请求就要 200ms 左右，在 `pg_stat_statements` 里累计耗时排前五。改成 JOIN 之后合并成一次查询，单次大概 15ms，总耗时直接降了一个数量级，从 top 排名里消失了。同时用 `EXPLAIN ANALYZE` 确认了执行计划走了索引，没有全表扫描。

### Q：单条 SQL 查询你会怎么优化？

**我的回答：**
- 先用 EXPLAIN ANALYZE 看执行计划，判断是否用了索引、是否有 filesort、是否回表。
- 没用索引就加索引；有回表就建覆盖索引；有 filesort（ORDER BY）也可以通过加索引解决。
- JOIN 没走索引时会用 Block Nested Loop（BNL），可以调大 join_buffer_size 减少扫描次数。

**改进后的回答：**
- 先用 `EXPLAIN`（MySQL）或 `EXPLAIN ANALYZE`（PG）看执行计划，确认是否走了索引、有没有全表扫描、排序是否走了索引。
- 没走索引就加索引；只需要部分列就建覆盖索引避免回表（MySQL 的 `Using index` / PG 的 `Index Only Scan`）。
- 查询改写：避免 `SELECT *`，子查询改 JOIN，深分页用游标分页优化（`WHERE id > last_id LIMIT N`）。
- JOIN 优化：MySQL 确保连接列有索引（否则走 BNL/Hash Join），可调大 `join_buffer_size`；PG 关注 Nested Loop / Hash Join / Merge Join 的选择，可调大 `work_mem`。
- MySQL 特有：利用索引下推（ICP，5.6+）减少回表，联合索引中范围查询后的列虽然不能用于索引查找，但能在索引层提前过滤。

### Q：索引失效的场景有哪些？

**我的回答：**
- 联合索引不满足最左匹配原则（范围查询时后续列失效）。
- 对索引列使用函数导致无法命中索引。

**改进后的回答：**
1. **联合索引不满足最左匹配**：跳过最左列，如 `(A,B,C)` 只查 `WHERE B=x`。
2. **范围查询后的列失效**：`WHERE A>1 AND B=2`，B 走不了索引（因为跨多个 A 值后 B 不再有序）。
3. **对索引列用函数**：`WHERE YEAR(create_time) = 2024`。
4. **隐式类型转换**：列是 varchar 但 `WHERE phone = 13800138000`（没加引号）。
5. **LIKE 左模糊**：`WHERE name LIKE '%张'`。
6. **OR 连接非索引列**：`WHERE A=1 OR D=2`，D 没索引则整体不走索引。

### 追问：联合索引 (A,B,C)，查询条件是 B=x AND A=y AND C=z，都是等值查询，能命中索引吗？

**我的回答：** 可以，优化器会自动调整顺序为 A,B,C。SQL 里 WHERE 条件的书写顺序不影响索引使用。但如果是 `A=y AND B>x AND C=z`，C 就失效了（范围查询后面的列），跟写的顺序无关。

## 5. 仿大众点评项目 — 秒杀与消息队列

### Q：异步秒杀下单怎么实现的？

**我的回答：**
- Lua 脚本在 Redis 中做两件事：①一人一单校验（set 判断 userId）；②库存扣减（判断库存 > 0 则减 1）。利用 Redis 单线程串行执行保证原子性。
- 订单落库通过 RabbitMQ 异步完成，解耦下单与落库。
- 库存仅在 Redis 中扣减，DB 只保存订单信息，未做 Redis 与 DB 库存的一致性同步。

### Q：RabbitMQ 失败怎么办？

[[阿里云消息团队一面面经#Q：扣库存成功但发消息失败，或消费失败怎么处理？]]
**我的回答：**
- **生产者端：** 投放失败时 RabbitMQ 返回 NACK，重复投放；多次失败则写入数据库兜底表，定时任务轮询重试。
- **Broker 端：** 消息持久化，投放成功即不会丢失。
- **消费者端：** 手动 ACK 模式，消费成功才确认；多次消费失败则转入死信队列，由另一个消费者处理或转人工。
- 订单 ID 做幂等，防止重复消费。

### Q：死信队列的实现原理？

**我的回答（不太确定）：** 本质是一个新的交换机，消费失败的消息会被转发到死信队列，后续再拉取处理。具体底层实现记不太清楚。

**改进后的回答：**
- 消息变成死信的三种情况：①消费者拒绝（basic.reject / basic.nack）且 `requeue=false`；②消息 TTL 过期；③队列达到 `x-max-length` 上限，最早的消息被挤出。
- 原队列声明时配置 `x-dead-letter-exchange` 和 `x-dead-letter-routing-key`，当消息变成死信时，Broker 会将其重新发布到指定的死信交换机，由死信交换机路由到死信队列。
- 死信交换机和死信队列本质就是**普通的交换机和队列**，没有特殊实现，只是用途不同。整个机制就是一次消息的转发。

### Q：乐观锁提升并发写入性能，有测试过吗？

**我的回答：** 没有具体测试。乐观锁基于 CAS 机制，在我的场景中简化为判断库存 > 0 即可扣减，避免使用重量级锁。

**改进后的回答：**
- 标准乐观锁（CAS）比较具体库存值：`WHERE count = #{之前查到的count}`，并发时只有一个线程 CAS 成功，其余全部失败重试。实测 1000 QPS 下 200 个库存都抢不完，因为大量请求 CAS 失败后放弃了。
- synchronized 悲观锁虽然能保证正确性，但完全串行化，响应时间太慢。
- 我的优化是把 CAS 条件放宽为 `WHERE count > 0`，只要有库存就能扣减，多个线程可以同时成功，既避免了标准 CAS 的大量无效失败，也不需要悲观锁的串行等待。具体延迟没测过，但从结果上看库存能正常扣完，不会出现"抢不完"的问题。

### 补充：为什么选 RabbitMQ 而不是 RocketMQ / Kafka？

**推荐回答：**
仿大众点评是个中小型项目，秒杀场景 QPS 不会到十万级，RabbitMQ 的吞吐量够用。选它主要因为：①延迟低，秒杀对响应时间敏感；②路由灵活，死信队列实现简单；③Spring Boot 集成方便，开发成本低。如果是真实的高并发电商场景（比如双十一），吞吐量要求更高且需要事务消息，会考虑 RocketMQ。如果是日志采集或大数据管道，会选 Kafka。

**三者对比：**

| | RabbitMQ | RocketMQ | Kafka |
|---|---|---|---|
| 定位 | 通用消息队列 | 电商/金融级消息队列 | 分布式日志/流处理平台 |
| 吞吐量 | 万级 QPS | 十万级 QPS | 百万级 QPS |
| 延迟 | 微秒级（最低） | 毫秒级 | 毫秒级 |
| 路由能力 | Exchange 四种模式（direct/topic/fanout/headers），最灵活 | Topic + Tag 两级过滤，够用 | 仅 Topic，Broker 尽量不做事追求极致吞吐 |
| 事务消息 | 不支持（需本地消息表变通） | 原生支持（半消息 + 事务回查） | 不支持 |
| 消息回溯 | 不支持（ACK 后消息删除） | 支持（按时间点重置消费位点） | 支持（重置 consumer offset） |
| 延迟消息 | 需插件或 TTL+死信变通 | 原生支持 | 不支持 |
| 语言 | Erlang（国内不好二次开发） | Java | Scala/Java |

**选型原则：**

| 场景 | 选择 | 原因 |
|------|------|------|
| 中小项目，业务消息，需要灵活路由 | RabbitMQ | 轻量、延迟低、路由灵活 |
| 电商/金融，需要事务消息、延迟消息 | RocketMQ | 原生事务消息、可靠性最高 |
| 大数据/日志采集/流处理、高吞吐 | Kafka | 吞吐量碾压、和大数据生态集成好 |

**RocketMQ 事务消息流程（半消息机制）：**
1. 生产者发送半消息（half message）→ Broker 收到但不投递给消费者
2. 生产者执行本地事务（如扣库存）
3. 本地事务成功 → 发 commit → Broker 投递消息给消费者；本地事务失败 → 发 rollback → Broker 丢弃消息
4. 如果 Broker 迟迟没收到 commit/rollback → 主动回查生产者本地事务状态，生产者返回 commit 或 rollback

**RocketMQ 消息回溯：** 按时间点重置消费位点，重新消费历史消息。RabbitMQ 做不到（ACK 后消息删除），RocketMQ 和 Kafka 基于日志追加存储，ACK 只是移动消费位点，消息本身还在。

## 6. 技术八股 — Redis

### Q：用 Redis 实现分布式锁需要注意什么？

**我的回答：**
- 用 SETNX 实现锁。
- **过期时间管理：** 过期时间不好设定，需要看门狗机制持续续期。
- **原子性问题：** 加（判断是不是自己的锁）锁和释放锁是两步操作，需要用 Lua 脚本保证原子性。
- **锁误删问题：** A 持锁时间过长导致锁过期，B 获得锁后，A 处理完把 B 的锁释放了。解决方案是释放锁时校验持有者 ID，只释放自己的锁。

**改进后的回答：**

**加锁：** `SET unique_key 机器码+线程ID NX EX TTL`（unique_key 是业务唯一标识）
- SETNX 保证互斥：key 不存在返回 1（加锁成功），已存在返回 0（加锁失败）
- 设置 TTL 防止死锁：持锁进程挂了（OOM、kill -9），看门狗也跟着挂，锁到期自动释放，不会死锁

**释放锁：** 用 Lua 脚本保证原子性
1. GET key 拿到 value，判断是否是本机器本线程加的锁
2. 是 → 删除 key 释放锁；不是 → 不处理

为什么要用 Lua 脚本：判断锁归属和删除 key 必须是原子操作。如果分两步，线程 1 判断完是自己的锁，准备删除时发生 Full GC 阻塞，TTL 到期锁释放，线程 2 拿到锁，线程 1 恢复后直接删除 → 误删了线程 2 的锁。

**SETNX 实现的三个问题及解决方案：**

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| TTL 到期锁提前释放 | 业务没做完锁就过期了，其他线程进来并行执行 | 看门狗机制持续续期，业务没完成就不让锁过期 |
| 锁误删 | 锁过期后别人拿到锁，自己处理完误删别人的锁 | value 存机器码+线程 ID，释放前校验持有者，只删自己的锁（兜底看门狗失效的极端情况） |
| 不可重入 | 同一线程再次加锁会失败（key 已存在返回 0） | Redisson 用 Hash 结构存储，field 为线程 ID，value 为重入次数，每次重入 +1，释放 -1，减到 0 才删 key |

**工程实践：** 以上机制 Redisson 已全部封装（看门狗默认 30s 续期、Lua 脚本原子操作、持有者校验、可重入锁），实际项目直接用 Redisson，不用自己实现。

### Q：Redis 删除一个 key 的时间复杂度是多少？

**我的回答（不确定）：** 猜测是标记删除所以 O(1)，但又觉得如果是标记删除这个问题没意义。

**面试官纠正：** Redis 是物理删除，不同数据类型有不同的删除复杂度，与底层数据结构有关。不需要引入 GC 那种标记删除机制。

**改进后的回答：**
Redis 是物理删除，复杂度取决于数据类型：

| 类型     | 底层结构                         | DEL 复杂度   | 说明               |
| ------ | ---------------------------- | --------- | ---------------- |
| String | SDS                          | O(1)      | 直接释放             |
| List   | quicklist                    | O(N)      | 遍历所有元素逐个释放       |
| Hash   | ziplist 或 hashtable          | O(N)      | N 为 field 数量     |
| Set    | intset 或 hashtable           | O(N)      | N 为元素数量          |
| ZSet   | ziplist 或 skiplist+hashtable | O(N·logN) | 跳表删除每个元素 O(logN) |

生产环境删大 key 要用 `UNLINK`（4.0+），把释放内存的操作丢到后台线程异步执行，避免阻塞 Redis 主线程。

### Q：Redis 内存有什么清理方式？

**我的回答：** LRU 淘汰机制，淘汰不活跃的 key。

**改进后的回答：**
分两个层面：

**1. 过期键删除（已设置 TTL 的 key）：**
- **惰性删除**：访问 key 时才检查是否过期，过期了才删。不访问就一直占内存。
- **定期删除**：Redis 每 100ms 随机抽取一批设了过期时间的 key，删除其中过期的。如果过期比例超过 25%，再抽一批继续删。
- 两者配合：定期删除兜底没人访问的过期 key，惰性删除保证访问时不会拿到过期数据。

**2. 内存淘汰策略（`maxmemory` 满了）：**

| 策略 | 说明 |
|------|------|
| noeviction | 不淘汰，写入直接报错（默认） |
| allkeys-lru | 所有 key 中淘汰最近最少使用的 |
| volatile-lru | 只在设了过期时间的 key 中淘汰 LRU |
| allkeys-lfu | 所有 key 中淘汰最不经常使用的（4.0+） |
| volatile-lfu | 只在设了过期时间的 key 中淘汰 LFU |
| allkeys-random | 随机淘汰 |
| volatile-random | 设了过期时间的 key 中随机淘汰 |
| volatile-ttl | 淘汰 TTL 最短的 |

最常用的是 **allkeys-lru**。

### 补充：Redis 跳表（Skip List）

ZSet 底层使用跳表，本质是**多层有序链表**。最底层是完整的有序链表，每往上一层随机抽取部分节点建立索引，加速查找。

```
Level 3:  1 ──────────────────────→ 50
Level 2:  1 ──────→ 20 ──────────→ 50
Level 1:  1 → 10 → 20 → 30 → 40 → 50
```

查找过程：从最高层开始，往右走，走过头了就下一层，类似二分查找。

**插入：**
1. 从最高层开始，逐层往下找到每层的前驱节点，O(logN)
2. 随机决定层数：每次 25% 概率升一层（Redis 实现，最高 32 层）
3. 在对应层插入节点，修改前驱指针，O(1)

**删除：**
1. 从最高层开始逐层定位目标节点，O(logN)
2. 在每层修改前驱节点的指针跳过该节点，O(1)
3. 如果删除后最高层为空，降低跳表层数

**时间复杂度：**

| 操作 | 复杂度 | 说明 |
|------|--------|------|
| 查找 | O(logN) | 逐层往下，每层跳过大量节点 |
| 插入 | O(logN) | 瓶颈在查找插入位置 |
| 删除 | O(logN) | 瓶颈在查找目标节点 |
| DEL 整个 ZSet | O(N·logN) | 遍历 N 个元素，每个从跳表删除 O(logN) |

**为什么 Redis ZSet 用跳表而不用红黑树：**
- **范围查询方便**：跳表底层是有序链表，`ZRANGEBYSCORE` 找到起点后直接往后遍历；红黑树做范围查询要中序遍历，更复杂
- **实现简单**：跳表代码量远少于红黑树，容易维护和调试

## 7. 技术八股 — 并发与数据结构

### Q：对链表进行插入和删除是线程安全的吗？

**我的回答：** 不是线程安全的。

### Q：怎么保证线程安全？

**我的回答（有些犹豫）：** 锁住 previous 指针节点。讨论了 next 指针是否需要锁，最终面试官引导得出结论：只需锁 previous 节点即可，因为删除 next 节点时也需要先找到它的 previous，不存在"next 指针被删"的问题。

**改进后的回答：**
插入和删除本质都是修改前驱节点的 next 指针。删除时需要**同时锁住 prev 和 cur 两个节点**：
- 锁 prev：防止别人同时修改 prev.next
- 锁 cur：防止别人把 cur 当作前驱去操作（保证 cur.next 在你读取时是稳定的）

只锁 prev 不锁 cur 的反例：
```
A → B → C → D
线程 1 要删除 B：锁住 A，准备执行 A.next = C
线程 2 要删除 C：锁住 B，执行 B.next = D
线程 2 先完成：A → B → D（C 被删了）
线程 1 再完成：A.next = C → A 指向了已删除的节点 C
```
同时锁 A 和 B 就能避免这个问题，不同位置的操作锁的是不同节点，可以并行，比锁整个链表粒度更细。

### Q：ConcurrentHashMap 如何实现线程安全？

**我的回答：**
- 初始化时用 CAS。
- 后续对桶（slot）加 synchronized 锁。

**改进后的回答（JDK 1.8）：**
- **初始化**：多个线程同时触发时，用 CAS 竞争 `sizeCtl` 标志位，只有一个线程能成功初始化数组，其他线程 yield 等待。
- **put 操作**：
  - 桶为空：CAS 直接把新节点放进去，不加锁
  - 桶不为空：对桶的**头节点**加 synchronized，然后遍历链表/红黑树做插入或更新
  - 正在扩容：当前线程帮助迁移（协助扩容），迁移完再 put
- **get 操作**：完全无锁，Node 的 val 和 next 都是 `volatile` 修饰，保证可见性。

**对比 JDK 1.7：**

| | JDK 1.7 | JDK 1.8 |
|---|---|---|
| 结构 | Segment 数组 + HashEntry 数组 | Node 数组 + 链表/红黑树 |
| 锁 | ReentrantLock（锁 Segment） | CAS + synchronized（锁头节点） |
| 锁粒度 | Segment 级别（默认 16 个） | 桶级别（数组多大就多少把锁） |
| 并发度 | 最多 16 | 等于数组长度，更高 |

### Q：两个请求同时触发扩容怎么处理？

**我的回答（不太确定，经过思考）：**
- 扩容是按 slot 级别进行的（容量翻倍，链表/红黑树一分为二）。
- 不同线程可以负责不同 slot 的迁移，协助扩容，不会冲突。

**改进后的回答：**
不会重复扩容。默认初始容量 16，负载因子 0.75，元素超过 `容量 × 0.75` 时触发扩容，每次容量翻倍。

扩容流程：
1. 第一个线程通过 CAS 修改 `sizeCtl` 为负数（标记正在扩容），创建 2 倍大小的新数组，开始迁移
2. 其他线程发现 `sizeCtl` 为负数，加入协助迁移而不是重新创建数组
3. 迁移按 stride（步长）分配桶，每个线程领一段桶负责（如线程 1 迁移桶 0-15，线程 2 迁移桶 16-31），互不冲突
4. 桶迁移完放一个 `ForwardingNode` 标记，表示该桶已迁移完成
5. 所有桶迁移完，扩容结束

扩容期间的读写：
- **get**：碰到 ForwardingNode 自动去新数组查，不影响读
- **put**：碰到正在迁移的桶，先帮忙迁移完再在新数组上 put

### Q：限流和熔断降级分别解决什么问题？

**我的回答：**
- **限流：** 保证流量稳定，削峰平谷，保护服务不被压垮。
- **熔断：** 某个链路节点故障时，切断该链路，防止影响整体服务。

**改进后的回答：**

| | 限流 | 熔断 |
|---|---|---|
| 保护谁 | 保护自己不被流量压垮 | 保护自己不被下游故障拖垮 |
| 触发条件 | 请求量超阈值 | 下游错误率/超时率超阈值 |
| 做法 | 拒绝多余请求 | 停止调用下游，返回降级结果 |

- **限流**：在入口控制请求速率，超出阈值的请求直接拒绝或排队。例：秒杀 10 万 QPS，系统只能扛 1 万，限流只放 1 万进来，其余返回"系统繁忙"。
- **熔断**：下游服务故障时，如果不熔断，上游线程全部阻塞等待下游响应，导致**雪崩**（一个服务挂了拖垮整条链路）。熔断后自动断开对下游的调用，直接返回降级结果（兜底数据/默认值）。一段时间后进入半开状态试探恢复。例：订单服务调用库存服务，库存挂了，熔断后直接返回"库存查询暂不可用"，而不是每个请求都等超时。

### Q：有哪些限流算法？
[[RPC框架整理后#七、限流算法]]
**我的回答：**
- **固定窗口 / 滑动窗口：** 滑动窗口解决固定窗口边界处请求突增的问题。
- **漏桶：** 以固定速率处理请求。
- **令牌桶：** 按速率生成令牌，有令牌就放行，能应对突发流量。

**改进后的回答：**
- **固定窗口**：每个时间窗口（如 1 秒）内计数，超阈值拒绝。问题：窗口边界突发，第 1 秒末 100ms 来 100 个 + 第 2 秒前 100ms 来 100 个 = 200ms 内通过 200 个，超过预期。
- **滑动窗口**：把固定窗口切成多个小格子（如 1 秒切 10 个 100ms），窗口滑动统计，粒度更细，缓解边界突发。Sentinel 默认使用。
- **漏桶**：请求进入桶排队，以固定速率流出。桶满丢弃。流量绝对平滑，但无法应对突发流量。
- **令牌桶**：以固定速率往桶里放令牌，请求来了取令牌，有就通过没有就拒绝。桶里可以攒令牌，允许短暂突发。Guava `RateLimiter` 使用。

**漏桶 vs 令牌桶：**

| | 漏桶 | 令牌桶 |
|---|---|---|
| 控制的是 | 流出速率（处理速率恒定） | 流入速率（允许突发） |
| 突发流量 | 不允许，严格平滑 | 允许短暂突发 |
| 适用场景 | 需要严格匀速的场景 | 大多数场景（更实用） |

**滑动窗口 vs 令牌桶怎么选：**

| 场景 | 选择 | 原因 |
|------|------|------|
| 统计某接口/用户的请求量做限流 | 滑动窗口 | 只需要计数，简单高效 |
| 控制请求速率，允许短暂突发 | 令牌桶 | 能攒令牌，灵活 |
| 控制请求速率，严格匀速 | 漏桶 | 绝对平滑 |

实际项目中常**组合使用**：入口用滑动窗口做全局限流（Sentinel），对特定资源用令牌桶做精细速率控制（RateLimiter）。
