## 2. 西门子实习 — RAG 系统

### Q：项目整体介绍

**我的回答：**
- 项目背景：将 CT 操作文档提取出来，为操作人员在学城提供 AI 问答和培训。
- 接手时已有基于 LangChain 的基础 RAG 流程（问题 → 检索文档 → 喂给 LLM 回答）。
- 我的工作：
  1. 基于 LangSmith 搭建测试流水线，测出初始正确率 76%。
  2. 优化文档转换流程（原开源框架不再维护），重建 ETL pipeline，正确率提升到 80+%。
  3. 更换 embedding 模型（BGE-M3 → 千问三），向量维度从 1024 变为 4096，换用 Milvus 支持高维向量。
  4. 引入 Query Refine（多语言版本生成）解决中英文术语 gap。
  5. 混合检索：语义向量 + BM25 关键词检索 + BGE-M3 稀疏向量，三路混合。
  6. Reranker 重排：测试了 RRF 和基于 LLM 的 rerank，最终采用 LLM rerank（效果更好）。
  7. 最终正确率从 76% 提升到 92%。

### Q：如何评测回答正确率？

**我的回答：**
- 测试中心提供两三百个中英文问题 + 标准答案。

- RAG 系统本身使用 Llama、千问、GPT 三个模型（需本地部署，数据安全要求）。

### Q：使用来评估的大语言模型具体是什么？

**我的回答：** 评判用 DeepSeek V3，RAG 系统用 Llama + 千问 + GPT，均为本地部署（数据安全需求）。

## 3. AI Coding 使用经验

### Q：你使用 AI coding 的经验和体验如何？

**我的回答：**
- 使用过 Copilot（公司额度）、Cursor，最近切换到 Claude Code。
- 核心方法论：**不用 agent 模式直接改代码**，而是：
  1. 让 AI 先对代码仓库做预读，建立对项目的理解。
  2. 把需求发给 AI，让它写 plan：要改哪些文件、具体怎么改、在哪改。
  3. 审阅 plan，不理解的地方追问，不需要的地方删掉。
  4. 确认 plan 无误后才让 AI 执行修改。
  5. 改完跑 benchmark 脚本验证。
- 原因：简单 prompt 下 agent 模式会改出不符合预期的代码，尤其在内核开发这种复杂场景。

### Q：你会让 AI 生成测试用例吗？

**当时的回答：** 平时写代码会让 AI 生成测试用例。ARM 实习场景主要用专门的 benchmark 脚本跑性能数据验证。

**改进后的回答：**

我的做法是分层验证：

1. **测试与实现分离生成：** 让 AI 先根据需求写测试用例，我审阅测试用例的合理性和边界覆盖后，再让 AI 写实现代码。测试和实现是两次独立生成，能互相校验，避免"自己出题自己答"的盲区。
2. **Agent Loop 自动修复：** 在 Claude Code / Cursor 的 agent 模式下，AI 写完代码后自动执行测试 → 看报错 → 修复 → 再跑，循环直到通过。关键是提供一个能跑的测试环境和明确的验收标准。
3. **交叉验证：** 让一个 AI 写代码，另一个 AI review 或写测试。两次生成的思路不同，更容易发现问题，类似代码审查中"写的人和审的人不是同一个人"。
4. **Property-based Testing：** 用 Python hypothesis 等库定义不变量（如排序后数组有序、长度不变），自动生成大量随机输入验证，覆盖面比手写 case 大得多。
5. **对比测试（Differential Testing）：** 如果有已知正确的旧实现，用相同输入跑两个版本对比输出。我在 ARM 的 benchmark 脚本本质上就是这个思路。

### Q：对 Cursor 原理的了解？

**我的回答：**
- 我理解 Cursor 是一个 agent，核心是 plan → 执行（reasoning and acting）。
- 底层工具比较简单抽象：读文档、搜索文档、写代码，几个基础工具组合 + 上下文管理实现强大功能。
- 关键在于如何把上下文做好抽象。

**面试官补充：** Cursor 打开新项目时做 index，实际上是在做代码的语义化 embedding，上传到服务器建向量库（有数据安全风险讨论）。

### Q：对私域知识库在 AI coding 中的应用了解吗？

**当时的回答：**
- ARM 公司的 Copilot 可以接入 Confluence 文档，但体感不明显。
- 方案思路：公司内部部署 AI 系统，对公司文档库和代码库做 embedding 检索辅助编码。
- 另一种方式：编写代码规范文档（如 CLAUDE.md / .cursorrules），让 AI 遵循命名规则、编码规范。

**面试官补充：** 还有 MCP 服务，可以让 AI 通过 MCP 调取内部文档（如贴 PRD 链接，AI 通过 MCP 获取文档内容）。

### Q：你了解 MCP 吗？

**当时的回答：** 了解，MCP 是一个工具箱协议，返回工具信息给 LLM 进行工具调用。但不太理解如何用于公司内部知识共享。

**两题合并改进后的回答：**

私域知识接入 AI coding 主要有三种方式，由轻到重：

1. **Rules 文件：** 比如 `.cursorrules` 或 `CLAUDE.md`，把编码规范、命名约定、架构决策写进去，AI 每次对话都会读取。适合相对稳定的规范类知识，零成本。
2. **MCP（Model Context Protocol）：** MCP 本质是给 LLM 提供一套标准化的工具接口，让 AI 能实时访问外部数据源。比如公司搭一个 MCP server 对接 Confluence、Notion、内部 Git 仓库，AI coding 工具在需要时通过 MCP 调用拿到文档内容，不需要提前做 embedding，按需实时读取。这就是面试官提到的"贴个 PRD 链接 AI 就能拿到内容"的实现方式。
3. **RAG — 对代码库和文档做 embedding 建向量库：** Cursor 打开项目做 index 时本质就是在做这件事。适合代码量大、需要语义检索的场景，但维护成本更高，也有数据安全顾虑。

我在 ARM 用的 Copilot 接 Confluence 应该是第二或第三种思路，但体感不明显，可能是接入的文档质量或 context 管理还不够好。

## 4. Java 八股

### Q：简单介绍一下 ThreadLocal？

**我的回答：**
- ThreadLocal 是 Java 的一个类，用来保存线程上下文信息，避免每次函数调用都传递常用参数。
- 本质：每个 Thread 维护一个 ThreadLocalMap，key 是 ThreadLocal 对象，value 是存储的值。
- get() 操作：以 ThreadLocal 对象自身作为 key，在当前线程的 ThreadLocalMap 中查找对应值。

**追问：ThreadLocal 是线程安全的吗？**

**我的回答：** 是，每个线程维护自己的 map，不是多个线程共享一个 map。



### Q：TCP 和 UDP 的区别？

**我的回答：**
- TCP：面向连接，可靠传输（滑动窗口、Sequence/ACK number），保证按序到达和尽最大可能可靠交付。
- UDP：无连接，不保证可靠传输，不保证按序到达。



**推荐回答：**

**TCP** 是**面向连接**的，在通信前需要先建立连接，比如三次握手。它的特点是**可靠传输**，也就是说它会通过**序列号、确认应答 ACK、超时重传、滑动窗口、流量控制、拥塞控制**这些机制，尽量保证数据**不丢、不重、不乱序**，并且按序到达对端。所以 TCP 更适合对数据准确性要求高的场景，比如网页访问、文件传输、数据库通信。



**UDP** 是**无连接**的，发送数据之前不需要先建立连接，直接发就可以。它的特点是**开销小、传输效率高、时延低**，但是它**不保证可靠性**，也不保证对方一定收到，也不保证按序到达。所以 UDP 更适合对实时性要求高、能容忍少量丢包的场景，比如音视频通话、直播、在线游戏、DNS 查询。

### Q：JDK 动态代理讲一下？

**我的回答：**
- 类似包装器模式，通过实现代理接口实现动态代理。
- 实现 InvocationHandler，在函数调用前做 AOP 拦截。
- 在 RPC 框架中：调用方法时并不真正执行本地函数，而是发消息到服务器、接收结果、返回，实现无感远程调用。



**推荐回答：**

JDK 动态代理本质上就是在运行时生成一个代理对象，代理对象会实现目标类的接口。调用代理对象的方法时，不会直接进目标方法，而是统一进入 InvocationHandler 的 invoke 方法，所以我们可以在这里做统一增强，比如日志、事务、权限校验，或者 RPC 的远程调用封装。



它的核心类是 Proxy 和 InvocationHandler。通常先写一个 InvocationHandler，里面持有目标对象，再通过 Proxy.newProxyInstance() 创建代理对象。之后所有接口方法调用都会被拦截到 invoke 里。



它最大的限制是只能代理接口，如果一个类没有实现接口，就不能用 JDK 动态代理，一般改用 CGLIB。

### Q：ConcurrentHashMap 和 HashMap 的区别？

**我的回答：**
- HashMap 本质是数组，通过取模映射 index，用链表 + 红黑树解决哈希冲突。
- ConcurrentHashMap 在 HashMap 基础上保证线程安全：
  - **初始化阶段：** 用 CAS 保证单例初始化。
  - **slot 初始化：** 也用 CAS 方式。
  - **写入阶段：** 对 slot 头节点加 synchronized 锁，保证操作安全。

### Q：讲一下 CAS？

**我的回答：**

- Compare And Swap，原子操作。
- 有一个期望值和当前值，如果一致则 swap 新值进去，返回 true。
- 是一种无锁编程方式，通过 while 循环 + CAS 保证只有一个线程进入 critical section。
- 优点：性能比 synchronized 等重量级锁高。
- 缺点：竞争大时 while 循环空转，浪费 CPU 资源。



**推荐回答：**
CAS 是一种硬件级别的原子操作，它比较内存中的某个值是否为预期值，如果是就更新为新值，不是就不动。整个比较和交换的过程是一条 CPU 指令完成的，不可能被打断。





**CAS 的优缺点**

优点很明显，无锁并发，线程不会阻塞，不会有死锁问题，吞吐量高。Java 里的原子类、ConcurrentHashMap、AQS底层都大量使用 CAS。

缺点也有几个：

1）ABA 问题，值从A变成B又变回 A，CAS 检测不出来，以为没变过

2）自旋开销，高并发时大量线程反复重试，CPU 空转浪费资源

3）只能保证单个变量的原子性，多个变量的复合操作搞不定





### Q：Java 中锁的分类？

**我的回答：**
- synchronized、读写锁、可重入锁。
- 可重入锁：记录线程进入锁的次数，出一次减一，归零解锁。
- 读写锁：多线程可同时读，但有写锁时其他线程不能读或写。

**追问：synchronized 是可重入锁吗？**

**我的回答：** 一开始说不是，后来纠正为是。synchronized 底层通过 monitorenter/monitorexit 指令在对象头做标记，只要锁的是同一个对象就可重入。



**推荐回答：**

Java 中的锁可以按多个维度来分类。

1. 首先，按使用方式分，有 synchronized 和 Lock 体系。

synchronized 是关键字，由 JVM 实现，进入和退出同步块时自动加解锁；Lock 是 JDK 提供的接口，比如 ReentrantLock，需要手动加锁和释放锁。

Java 并发包里常见用法里，**基本接触到的主流锁大多都是可重入的**。

“不可重入锁”更多是概念上的对比，或者是你自己实现的简单互斥锁。





2. 其次，按是否可重入分，有可重入锁和不可重入锁。

可重入锁指同一个线程在持有锁的情况下，可以再次获取这把锁而不会死锁。Java 中 synchronized 和 ReentrantLock 都属于可重入锁。



3. 再按读写特性分，有独占锁和读写锁。

独占锁同一时刻只允许一个线程访问，比如 synchronized 和 ReentrantLock；

读写锁则允许多个线程同时读，但写的时候会和读写都互斥，比如 ReentrantReadWriteLock。



4. 另外按公平性还可以分成公平锁和非公平锁。公平锁会按请求顺序获取锁，非公平锁允许插队，吞吐量通常更高。

- new ReentrantLock(true) —— 公平锁
- synchronized：通常认为是**非公平锁**
- new ReentrantLock()：默认是**非公平锁**
- new ReentrantLock(false)：非公平锁
