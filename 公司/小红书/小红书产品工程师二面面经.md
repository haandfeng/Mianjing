## 2. ARM 实习 — Profiling 工具移植与 off-CPU 分析

### Q：你移植的这个工具原本跑在什么环境上？你主要做了什么？

**我的回答：**

这个工具最早来自 2015 年的一篇系统方向顶会论文，本质上是一个能够预测“如果优化某段代码，整体吞吐量会提升多少”的 profiling 工具。原方案已经在别的平台上做过实现，我这边主要工作不是从零发明，而是把它迁移到 Arm 平台，并把它做得更工程化、更可用。  
 [oai_citation:2‡小红书实习面试—谭远峰.txt](sediment://file_00000000a8187206851f887283c84ff2)

### Q：这个工具本身有什么问题？你为什么要改它？

**我的回答：**

我当时主要看到三类问题：

1. **功能不完整：** 比如分析 MySQL 时需要先 warm up，原工具对这类流程支持不好。
2. **上下文不够：** 它只能告诉你“某一行代码如果加速会怎样”，但不能很好展示更完整的调用栈上下文。
3. **不能分析 off-CPU 问题：** 它基于 Linux perf 做采样，而 perf 只能看到 on-CPU 的执行，看不到线程 sleep、锁等待、IO 等待这些 off-CPU 时间。  
    [oai_citation:3‡小红书实习面试—谭远峰.txt](sediment://file_00000000a8187206851f887283c84ff2)

### Q：off-CPU profiling 的原理是什么？

**我的回答：**

思路是记录线程被调出 CPU、再被调回 CPU 之间经过了多久。因为这个工具本质上是靠采样估算“某段代码值得被虚拟加速多少”，所以如果线程在 off-CPU 期间本来应该被采样但没有采到，我就需要根据这段时间去反推出“少采了多少样本”，再把这些样本补回来。这样就能把 off-CPU 的等待时间也纳入分析体系。  
它本质上不是直接把代码真的优化掉，而是通过给别的路径插入 delay，做一种“虚拟加速”实验，从而估计真正优化后的全局收益。  
 [oai_citation:4‡小红书实习面试—谭远峰.txt](sediment://file_00000000a8187206851f887283c84ff2)

### Q：这个项目对你来说最大的挑战是什么？

**我的回答：**

最大挑战是要真正改 Linux kernel 和 perf 子系统。因为为了支持 off-CPU 分析，我不是只改用户态代码，而是要深入 Linux 内核底层去看 perf 的调用链、时间戳记录点、等待类型分类这些逻辑。这里面难点主要有两个：

- **源码理解成本高：** Linux 内核源码本身比较复杂，不熟悉时阅读门槛很高。
- **迁移难度大：** 现有思路主要是在 5.15 附近版本实现的，但我要往 6.8 版本迁移，需要分析新旧版本之间函数、调用路径和数据结构的变化。  
   [oai_citation:5‡小红书实习面试—谭远峰.txt](sediment://file_00000000a8187206851f887283c84ff2)

### Q：你是怎么克服内核不熟的问题的？

**我的回答：**

我主要做了三件事：

1. **补底层基础：** 在去 Arm 前后，我系统看了 CSAPP 这类课程和资料，补系统底层知识。
2. **读官方文档：** 因为核心是改 perf 和 kernel，所以我会重点看 Linux 的相关开发文档，先建立对模块边界和改动位置的理解。
3. **借助 AI 辅助迁移：** 不是让 AI 替我乱改，而是用它帮我分析旧版本方案改了哪些点、为什么这样改，再结合源码自己验证，最后决定在 6.8 里应该把逻辑挂到哪些函数上。

## 3. ARM 实习 — 锁竞争监控与追问暴露的问题

### Q：锁竞争怎么纳入你的 trace 体系？

**我当时的回答：**

我提到的思路是：因为很多锁相关操作最终会走到 Linux/POSIX 相关 API，我可以通过链接阶段的符号拦截，把自己的函数符号放在更前面，让程序先进入我实现的包装函数，在里面做记录和插桩，再去调用真实实现。这个思路类似“函数劫持 + 轻量插桩”。  


### 面试官追问：你这不是系统级监控工具吗？为什么还需要和被测程序一起编译/打包？

**我当时的回答：**

我解释说，当前这套工具并不是一个完全零侵入、纯系统外置的监控工具，而是要和目标程序一起链接、一起运行；同时我还需要在源码里加入一些宏和标记，用来统计吞吐量和做轻量插桩。  


### 这一段的实际问题

这一段其实暴露出一个比较明显的问题：  
面试官在不断把问题往“**如果不是和应用一起打包，而是做成一个真正系统级、第三方监控工具，你会怎么实现**”这个方向推，但我后面的回答就有点跟不上了。

比如面试官问到：

- 如果是系统层面的监控，而不是一起打包，你会怎么抓锁竞争？
- 你有没有了解过 inline hook？
- 你知不知道 Linux 内核里锁是怎么处理的？
- 你能不能知道某把锁当前被谁持有？
- 你的监控线程怎么保证在整体程序足够早的时候启动？  

这些问题里，我能回答的部分主要还是围绕“插桩、拦截、main 前截取”这些已有思路展开，但对于**纯系统级监控、内核锁实现细节、inline hook** 这些知识点，回答得不够扎实，属于明显的短板。  


### 这部分的复盘

这块后续要重点补三类内容：

1. **用户态 hook / inline hook / PLT-GOT / LD_PRELOAD 这类机制的原理和适用边界**
2. **Linux 锁实现基础：mutex、自旋锁、futex、阻塞与唤醒链路**
3. **真正系统级观测方案：eBPF / uprobes / kprobes / perf tracepoint / 动态二进制插桩**  

这轮面试里，面试官其实不是在质疑你“有没有做项目”，而是在试图判断你对“工程方案边界”和“系统观测手段”的理解深不深。

## 4. 西门子实习 — RAG 系统整体优化

### Q：先介绍一下西门子这个项目

**我的回答：**

这是一个 CT 机操作问答系统，可以理解成一个面向内部场景的智能问答/智能客服系统。我进去的时候，系统基础链路已经搭好了，能够做文档提取和问答，但效果不够理想。所以我最开始做的是先搭建评测流水线，把问题系统化测起来；之后围绕 embedding、向量库、RAG 链路逐步做优化。  


### Q：评测系统是怎么做的？

**我的回答：**

测试中心会提供大约 300 个问题，中英文都有，同时给标准答案。我们把问题喂给 RAG 系统拿到输出，再从三个角度评测：

1. **答案正确性：** 回答与标准答案是否一致，有没有事实错误或关键缺漏。
2. **相关性：** 回答是否真正解决了问题。
3. **忠实度：** 回答是否忠于检索到的文档，有没有幻觉。  

这些判断本身主要是用大模型当 judge 来完成，但我每轮也会做人工 double check，去验证 judge 的判断是否靠谱，并结合失败 case 反查具体是哪一段链路出了问题。  

### Q：你们最后的准确率 76% 到 92%，这个 92% 是怎么确认的？

**我的回答：**

不是只跑一下大模型 judge 就直接认结果。每轮评测我都会人工核查，一方面验证 judge 的判定质量，另一方面也要从失败样本里找规律，看究竟是检索、重排、生成还是多语言术语匹配出了问题，再针对性优化。  

### Q：你们有没有做多轮纠错、反思、再生成？

**我的回答：**

当时没有。我们那时做的还是比较典型的一条工作流链路，没有做 agent 式的 loop，也没有做“第一次回答不好 → 再改写问题 → 再检索 → 再生成”的迭代闭环。  
这也是我后来的复盘点：现在很多更成熟的方案，都会在检索不到信息时自动做 query rewrite 和多轮重试，而我们当时没有往那个方向继续做。

## 5. 西门子实习 — 为什么要做混合检索

### Q：为什么会采用关键词检索、稀疏向量、混合检索这些技术？

**我的回答：**

最核心的出发点是：有些问题回答不上来，并不是因为语义不相关，而是因为问题里只有两三个很关键的专业术语，而这些术语在长文档里出现得非常稀疏。  
如果只做稠密语义检索，这几个关键词会被整篇文档的语义“稀释”掉，导致真正相关的文档召不回来。所以我当时就引入了关键词检索。  
同时，因为我们当时用的 embedding 模型支持稠密向量和稀疏向量能力，所以最终做成了多路召回：**稠密语义 + 稀疏/关键词 + 重排序**。  

### Q：为什么没有直接用多向量或更重的方案？

**我的回答：**

我也提到过，多向量方案理论上可以更细粒度地表达 token 级语义，但算力和工程成本都更高。考虑到当时项目实际资源和交付节奏，我们最终选择的是更务实的混合检索增强，而不是进一步上更重的方案。

## 6. 西门子实习 — Embedding 落库效率优化

### Q：你提到过 embedding 效率提升，这块具体怎么做的？

**我的回答：**

难点主要在于：文档量大，而且文档会动态增删。如果每次有文档调整都把所有文档重新 embedding 一遍，成本太高，完整跑一遍可能要一两个小时。  
所以我主要做了两类优化：

1. **吞吐优化：**  
   - 原来是单文档串行调用 embedding 服务；  
   - 后来改成 batch + 异步，把多篇文档一起发给服务端，服务端一次性返回结果，我再批量落库。  
   这样既减少了同步等待，也减少了 HTTP 频繁建连/断连的额外开销。

2. **增量更新：**  
   - 给每篇文档分配唯一 ID；  
   - 相同 ID 做覆盖，不同 ID 做插入；  
   - 支持针对某篇文档单独删除或更新；  
   - 避免每次都全量重建向量库。

## 7. 西门子实习 — 多语言术语问题

### Q：你说有中英文术语 gap，这个问题怎么解决的？

**我的回答：**

问题在于：文档里的专业术语很多是中文，但用户提问时更习惯用英文，而 embedding 模型对这类专业术语翻译和对齐能力有限；同时数据量也不够做专项微调。  
所以我用了一个折中的工程方案：先让一个小模型对问题做改写，生成 5 个左右语义相近、且包含中英文不同表达方式的问题，再把这些问题一起送去检索。  

这个方案的代价是大概增加了 1 秒左右时延，但对解决专业术语的中英文歧义问题很有效，属于性价比不错的优化。

## 11. 整体评价与复盘

### 整体表现

这轮面试整体上更像是**围绕项目持续深挖**，尤其是 ARM profiling 工具和西门子 RAG 项目。  
优点是：

- 项目经历本身比较扎实，很多内容能展开；
- 西门子 RAG 优化链路讲得比较顺，评测、混合检索、多语言改写、增量更新这几块都能自洽；
- 对 AI / agent / infra 的兴趣表达比较自然，也和岗位方向有一定契合。  

### 暴露的问题

最大的短板集中在 ARM 这段被持续追问时暴露出来：

1. **系统级监控 vs 源码侵入式插桩** 的边界理解还不够清晰；
2. **inline hook、系统级观测、内核锁机制** 这些知识点储备不够；
3. 回答有时停留在“我现在的实现怎么做”，但面试官问的是“如果换一种工程约束，你会怎么设计”。  

### 后续重点准备

建议重点补下面几块：

- Linux 锁、futex、阻塞/唤醒机制
- 用户态/系统态 hook 手段
- eBPF / uprobes / kprobes / tracepoints
- 系统级 profiling 与侵入式 profiling 的差异
- AI Coding 在大规模工程中的落地边界和评估方法
