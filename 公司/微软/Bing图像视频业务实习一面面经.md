## 2. 西门子实习 — RAG 系统优化

### Q：你在这个项目里真正做了什么？

**我的回答：**
- 我加入时，系统已经能完成“用户提问—检索文档—生成回答”的基础闭环。
- 我先搭建了一套测试 pipeline，用来系统评估回答效果。
- 测试后发现效果不够好，于是重构了文档解析和 embedding 落库流水线。
- 把原来的旧方案替换为支持新模型与 Milvus 的新方案。
- 之后继续做 query refine、混合检索、重排等优化，把效果从 **76 提升到 94**。

### 追问：为什么要重构 embedding / 落库流水线？

**我的回答：**
- 旧 pipeline 基于一个已不再维护的开源方案；
- 旧方案不支持更新的模型，也不适配 Milvus 这类高维向量库；
- 原来使用 Elasticsearch，但它不太适合当前的向量检索需求，所以需要重构整条流水线。

### 追问：你具体做了哪些优化？

**我的回答：**
- **Query refine：** 因为文档主体是中文，但用户会混用英文专业术语提问，所以先改写 query，增强术语对齐能力；
- **混合检索：** 结合语义检索和关键词检索，而不是只依赖 embedding；
- **重排：** 在召回后增加 rerank，提高最终送入 LLM 的上下文质量。

### 追问：76 到 94 是什么指标？

**我的回答：**
- 这里的提升更多是 correctness / 评测分数层面的提升；
- 此外我们还用 **F1 score** 去评估“LLM 评审”和“人工评审”的一致性；
- 具体做法是把模型打分和人工打分进行对比，按真阳/假阳/真阴/假阴计算 F1；
- 最后 F1 大概在 **0.8 左右**，说明模型评审和人工评审有较高一致性，因此这套自动评测流程是可落地的。

### 追问：当时用的模型有哪些？

**我的回答：**
测过千问、Llama、ChatGPT 等不同模型，开源和闭源方案都有做对比。

### 可优化回答

这段其实是这场面试最核心的技术亮点，后面可以答得更工程化一点：

**1. 先讲职责边界。**
你不是“从 0 搭了一个完整 RAG”，而是接手一个已经能跑的系统，重点负责：
- 评测体系搭建
- embedding / indexing pipeline 重构
- retrieval quality 优化

这样会更真实，也更容易让面试官信服。

**2. 把 76 到 94 说清楚。**
建议后面统一表述成：
- “这是我们内部评测集上的回答正确性分数，从 76 提升到 94。”
- 如果不是严格意义上的 accuracy / F1，就不要混着说，避免指标口径不一致。

**3. 把优化链路讲成固定框架。**
后面可以统一总结成：
**评测发现问题 → 重构索引流水线 → query 改写 → 混合召回 → rerank → 指标复测**。

这会比“想到什么说什么”更有条理。

## 3. LangChain / LangGraph 理解

### Q：你对 LangChain / LangGraph 的理解是什么？

**我的回答：**
- 它们更像是帮助管理 LLM 应用 workflow 的框架；
- LangGraph 对我来说最核心的价值是做**状态管理**和**流程编排**；
- 真正底层的数据库调用、SDK 调用，很多时候我不会完全依赖它封装的接口，而是自己直接调 SDK；
- 因为它的一些文档不完善，某些封装能力也不够好。

### Q：这个项目花了多少人、多少时间？

**我的回答：**
- 我理解前期大概是 **4 个人、2~3 周**，先把基础 RAG 系统搭起来；
- 后续质量优化和继续维护主要是我在跟进。

### Q：他们做 Agent 也是基于 LangGraph 吗？

**我的回答：**
- 我的理解是是的；
- 当时他们还在 planning 阶段，希望让大语言模型去操作 CTG；
- 因为可调用函数很多，所以他们在考虑如何做 tool selection 和 planning；
- 也希望借助 LangGraph 的 human-in-the-loop 能力，让人能够审核或修改模型给出的计划。

### 可优化回答

这里你答得其实不错，尤其是这句很加分：
> LangGraph 更多帮我做状态管理和 workflow 编排，底层实现我很多时候直接调 SDK。

这句话说明你不是“会调框架 API”，而是知道框架解决了什么、不解决什么。

后续可以再补一句：
- **LangChain / LangGraph 适合快速搭原型，但复杂生产系统往往还是要自己控制底层实现。**

这样你的判断会更成熟。

## 4. LangSmith / Trace / 评测理解

### Q：你怎么理解 LangSmith 的追踪能力？

**我的回答：**
- 它能把整个链路里每个节点的调用过程、输入输出都记录下来；
- 我更多是把它用来观察整条链路哪一步耗时高，以及回答出错时问题出在哪一环；
- 它不仅能做 trace，也能配合评测数据做自动打分。

### Q：你们把企业数据暴露给外部平台，会不会有风险？

**我的回答：**
- 理论上正式产品形态肯定不能这么做；
- 当时测试环境中只用了部分问题和部分数据，所以管理上没那么严格；
- 但最终上线肯定要走内部部署方案；
- 团队后面也考虑过用字节的 Coze / 扣子类方案做本地化 trace。

### 可优化回答

这块后面最好收束成两个点：

**1. LangSmith 的价值：**
- trace 链路
- 定位瓶颈
- 辅助评测

**2. LangSmith 的问题：**
- 生产环境有数据出境风险
- 企业级上线更适合自部署 / 内网可控方案

你现在原始回答里提到“字节扣子基本抄了 LangSmith”，这个口语化表达太直白，正式面试里建议弱化，改成：
- “字节也提供了类似的 tracing / evaluation 能力，而且更适合内部部署场景。”

## 5. 多轮对话与上下文压缩

### Q：你们有处理长上下文和多轮对话吗？

**我的回答：**
- 有做过一个简单的压缩机制；
- 如果对话达到 **10 轮**，系统会把前面的对话做一次 summary；
- 然后把这个 summary 作为新的 prompt 上下文继续传给模型，避免上下文窗口过长。

### Q：量级大概是多少？

**我的回答：**
- 单轮大概 5000 token；
- 十轮大概会到 5 万 token；
- 压缩后具体回落到多少我没有细看，因为我当时重点关注的是单轮回答准确率，而不是多轮对话质量。

### 可优化回答

这块你回答得很诚实，这反而是好事。后面可以更稳一点：

- 明确说**“我们当时的优化目标主要是单轮问答质量，不是多轮会话能力。”**
- 然后补一句：
  - 如果后续要做 Agent 或长对话系统，多轮 memory / summary / context window 管理会是重点。

这样既不会显得你没做过，又不会被追着问细节时失控。

## 6. AI Coding 的使用方式

### Q：你们开发时会用 AI coding 吗？

**我的回答：**
- 会用，西门子内部支持 Copilot；
- 我主要把它作为辅助编程工具；
- 方式不是让它完全接管，而是我先决定要加什么模块、要做什么设计，再让 AI 帮我实现具体代码。

### Q：你会用 Agent 模式直接改代码吗？

**我的回答：**
- 一开始试过，但感觉不太靠谱；
- 当时用 Claude 3.7 做 agent coding，生成的代码很冗余，而且改动过大，不容易读懂；
- 后面更多改成 ask 模式或补全模式，而不是完全放权给 agent。

### Q：你现在怎么更科学地用 AI？

**我的回答：**
- 先让 AI 帮我制定 plan；
- 我自己审阅、修改、打磨这个 plan；
- 等我完全清楚它会改哪些地方、为什么这么改，再让它执行；
- 改完以后我也会看每一行代码，确保自己真正掌握变更。

### 可优化回答

这段是全场另一个加分点，因为它非常贴近现在很多团队真实关心的问题。

后面建议把这段固定总结成一个三步法：

**1. 先让 AI 做方案，不直接改代码。**
先产出 plan，而不是一上来 agent 模式全量改。

**2. 人先 review 方案，再授权执行。**
自己先看懂改动边界、风险点、涉及模块。

**3. 改完必须 review，每一行关键代码都要能解释。**
这样才能既享受效率，又控制代码质量。

这套说法很适合后面继续复用。

## 7. ARM 实习 — 多线程 / 底层调试经验

### Q：为什么你觉得底层代码更难完全交给 AI？

**我的回答：**
- 因为底层 C/C++、多线程、系统调用相关问题，需要很强的背景知识；
- AI 对单文件、简单业务逻辑很强，但对复杂并发逻辑和系统行为理解有限；
- 我在 ARM 的工作里就遇到过多线程 bug，导致 MySQL TPS 直接掉到 0；
- 最终不是靠 AI 自动修好，而是我自己把整个项目逻辑理清，再和 AI 一起定位问题；
- 如果我自己不读懂代码，AI 很难真正解决这种问题。

### 可优化回答

这段经验很宝贵，后面建议讲成一个完整 STAR：

- **S：** merge 新代码后引入 bug，多线程问题导致性能掉到 0；
- **T：** 要定位并修复这个并发问题；
- **A：** 先手动梳理核心线程模型和代码路径，再把自己的理解喂给 AI 辅助分析；
- **R：** 最终定位并修复问题，也反过来验证了“底层复杂问题不能只靠 AI 自动修复”。

这比泛泛谈“AI 不太行”更有说服力。

## 8. C++ / Java / 底层能力讨论

### Q：你现在主要写什么语言？

**我的回答：**
- 刷题会用 Python；
- ARM 的工具和内核相关工作主要是 C / C++；
- 会涉及 Linux kernel 阅读和修改。

### Q：你理解 `shared_ptr` 吗？

**我的回答：**
- 它是智能指针的一种，通过引用计数管理对象生命周期；
- 当引用计数归零时对象会自动释放；
- 好处是减少手动内存管理带来的风险。

### Q：为什么觉得 C/C++ 比 Python 更依赖底层理解？

**我的回答：**
- 因为 C/C++ 里内存、指针、系统调用、并发这些问题都更直接暴露出来；
- Python 封装程度高，很多细节都被 runtime 屏蔽了；
- 所以写底层 C/C++ 时，更需要真的理解系统原理，而不能只停留在 API 层。

### 可优化回答

这部分主要是闲聊式延展，没有太强压力。后面可以准备更标准一点的表述：

- `shared_ptr`：共享所有权，引用计数，适合多处共同持有对象；
- `unique_ptr`：独占所有权，开销更低；
- 多线程下 `shared_ptr` 适合解决生命周期问题，但不等于对象本身线程安全。

## 9. 算法题 — 岛屿数量

### 题目

给定一个二维网格，由 `'1'` 和 `'0'` 组成，求岛屿数量。

### 我的思路

- 把这个问题抽象成图的**连通域**问题；
- 网格中的每个格子都可以看成一个节点；
- 如果某个位置是 `'1'`，并且和上下左右的 `'1'` 相连，就属于同一个连通块；
- 整体做法是遍历整个矩阵，遇到没访问过的 `'1'` 就启动一次 BFS，把与它相连的整块区域都标记掉；
- 每启动一次 BFS，岛屿数量加一。

### 我现场的回答特点

- 先说了这是一个图论里的连通域问题；
- 对比了 BFS 和 DFS；
- 最后选择用 **BFS** 实现，理由是自己觉得矩阵场景下 BFS 更直观。

### 复杂度

- **时间复杂度：** `O(m * n)`，每个格子最多被访问一次；
- **空间复杂度：** `O(m * n)`，最坏情况下队列或访问标记会达到这个量级。

### 面试官追问

#### Q：如果换成 DFS，和 BFS 的时间空间复杂度有区别吗？

**我的回答：**
- 时间复杂度本质没有区别，都是 `O(m * n)`；
- 空间复杂度量级上也差不多；
- 只是 DFS 走递归，BFS 用队列；
- 工程上真实系统里递归用得少一些，所以 BFS 也更稳妥。

### 可优化回答

你这题整体没问题，但有两个地方可以更稳：

**1. BFS / DFS 的选择理由别说成“矩阵更适合 BFS”。**
更稳的说法是：
- 二者都可以；
- 我这里选 BFS，因为我更习惯显式队列写法，也能避免递归栈问题。

**2. 空间复杂度别说 BFS 一定更低。**
严格来说两者最坏都可能到 `O(m * n)`，只是实现形式不同。

可关联：[[专栏/后端/图论基础：BFS、DFS、连通域问题]]

## 12. 整体评价与后续准备建议

### 这场面的整体表现

这场面试整体其实是 **“项目深挖 + AI coding 观念 + 一道中等算法题 + 岗位双向了解”** 的组合。

你的表现里比较亮眼的点有：
- 西门子 RAG 项目讲得比较真，不浮夸；
- 对 LangGraph 的理解不是停留在 API 层，而是理解它做 workflow / 状态管理；
- 对 AI coding 的使用方式很成熟，体现了你有自己的方法论；
- ARM 的底层调试经历，也让你和纯应用型候选人拉开了一点差异。

### 主要可改进点

**1. 指标口径要统一。**
76 到 94、correctness、F1 这几个概念容易混，要提前统一成标准表达。

**2. 回答可以更框架化。**
尤其是讲项目优化时，尽量按“问题—动作—结果”结构来答。

**3. 算法题的复杂度表述要更严谨。**
不要随口说 BFS 空间一定更低，这种地方容易被面试官抓住。

### 后续建议重点准备

1. 把西门子 RAG 项目整理成一版 **2 分钟回答** 和一版 **5 分钟深挖回答**。
2. 单独准备一套 **AI coding 方法论** 说法，这个很适合你当前简历。
3. 补一补以下八股，避免后续被顺着追问：
   - LangGraph / Agent 的 planning 与 memory
   - RAG 的评测指标体系
   - 向量库 vs Elasticsearch 的区别
   - C++ 智能指针与并发场景
4. 算法题继续保持，但复杂度表述要更谨慎。
