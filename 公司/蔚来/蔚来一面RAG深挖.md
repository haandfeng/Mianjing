# 蔚来面试问题：RAG 系统相关问题

## Q4：你做的 RAG 现在实际使用情况如何？使用时遇到的最大挑战是什么；是否指文本分段时前后句子分不同段，按段落分段能否解决该问题，解决此问题通常有哪些办法；RAG 解析的文档规模多大，为何不用大窗口大模型直接检索；用户仅文本输入输出时，多模态内容怎么转文本输出、具体怎么输出

**A：** 这是一个综合性的 RAG 系统问题，我分几个方面来回答：

---

### 1. RAG 实际使用情况

**使用场景**：
- 在实习项目中，RAG 系统已经上线到生产环境，用于 CT 机技术文档的问答系统
- 用户可以通过自然语言提问，系统从技术文档中检索相关信息并生成答案
- 主要服务于 CT 操作人员的培训和问题解答

**使用效果**：
- 准确率从初始的 60-70% 提升到 92%
- 通过多路召回（稠密向量 + 稀疏向量 + BM25）和重排序优化，显著提升了文档检索的准确性
- 支持中英文混合查询，能够处理专业术语

---

### 2. 使用时遇到的最大挑战

**主要挑战**：

1. **文档检索不准确**：
   - 问题：检索到的文档与问题相关性不高，或者关键信息没有被检索到
   - 原因：通用 embedding 模型对领域专业术语理解不够，中英文语义不一致
   - 解决：采用多路召回、领域微调 embedding、query 改写等策略

2. **上下文信息丢失**：
   - 问题：检索到的文档片段缺少必要的上下文，导致生成答案不完整
   - 原因：固定长度分块可能切断语义连贯性
   - 解决：使用语义分块、滑动窗口、层次化分块等策略

3. **多文档信息整合困难**：
   - 问题：检索到多个文档时，模型难以有效整合信息，可能出现信息冲突
   - 原因：模型对长上下文的处理能力有限，注意力分散
   - 解决：使用上下文压缩、LongContextReorder、CoT 推理等技术

4. **评估体系不完善**：
   - 问题：缺乏客观的评估指标，难以量化系统改进效果
   - 原因：只用大模型评估大模型，存在主观性和波动
   - 解决：引入召回率、精确率等客观指标，建立标准测试集

---

### 3. 文本分段问题及解决方案

**问题描述**：
- 是的，文本分段时前后句子分不同段是一个常见问题
- 例如：一个完整的概念或操作步骤可能被切分到两个不同的 chunk 中
- 这会导致检索时只找到部分信息，或者信息不完整

**按段落分段能否解决**：
- **部分解决**：按段落分段比固定长度分段好，因为段落通常有语义边界
- **但不够**：段落可能很长（超过模型窗口），也可能很短（信息不足）
- **更好的方案**：需要结合多种策略

**解决方案**：

1. **语义分块（Semantic Chunking）**：
   - 原理：根据语义相似度切分，不是固定长度
   - 实现：计算相邻句子/段落的 embedding 相似度，相似度低于阈值时切分
   - 优点：保证每个 chunk 语义完整，不会切断概念
   - 适用：技术文档、长文章

2. **滑动窗口（Sliding Window）**：
   - 原理：相邻 chunk 有 10-20% 的重叠部分
   - 优点：避免边界信息丢失，提高召回率
   - 缺点：产生更多 chunk，存储成本增加
   - 适用：所有类型文档，特别是需要高召回率的场景

3. **层次化分块（Hierarchical Chunking）**：
   - 原理：按文档结构（章节、段落）进行多粒度分块
   - 策略：粗粒度（章节）→ 中粒度（小节）→ 细粒度（段落）
   - 优点：既保证精确性又保证上下文完整性
   - 适用：结构化文档（技术手册、API 文档）

4. **递归分块（Recursive Chunking）**：
   - 原理：先按大块切，如果太大再递归切小块
   - 优点：自适应不同长度的内容
   - 适用：文档长度差异大的场景

5. **基于文档类型的分块**：
   - 技术文档：按章节 + 语义分块
   - FAQ：每个问答对作为一个 chunk
   - 代码文档：按函数/类分块
   - 操作手册：按步骤分块，保持步骤完整性

**推荐组合**：
- 对于技术文档：**语义分块 + 滑动窗口 + 层次化**的组合
- 先用层次化识别文档结构，在每层内部用语义分块，再用滑动窗口避免边界丢失

### 4. RAG 解析的文档规模及为何不用大窗口大模型

**文档规模**：
- 一个机型大约 5000 个文档
- 每个文档平均 500 tokens 左右
- 总规模：约 250 万 tokens（一个机型）
- 多个机型，文档会持续增加

**为何不用大窗口大模型直接检索**：

1. **成本问题**：
   - 大窗口模型（如 GPT-4 Turbo 128K、Claude 3 200K）每次调用成本很高
   - 如果每次查询都把所有文档塞进上下文，成本会非常昂贵
   - RAG 方式：只需要检索相关文档（通常 5-10 个），成本低得多

2. **效率问题**：
   - 大窗口模型处理长上下文需要更多时间
   - 把所有文档都放进去，响应时间会很长（可能几十秒）
   - RAG 方式：只处理相关文档，响应时间通常在几秒内

3. **准确性问题**：
   - 即使是大窗口模型，面对 250 万 tokens 的文档，也很难有效利用所有信息
   - 模型对长上下文的注意力会分散，可能"淹没"关键信息
   - RAG 方式：通过检索筛选出最相关的文档，模型可以更专注地处理

4. **可扩展性问题**：
   - 文档会持续增加，大窗口模型的上限也会被突破
   - RAG 方式：文档增加不影响查询效率，只需要更新向量库

5. **灵活性**：
   - RAG 可以结合多种检索策略（向量检索、关键词检索、混合检索）
   - 可以根据问题类型动态调整检索策略
   - 大窗口模型是"一刀切"，不够灵活

**总结**：
- 大窗口模型适合：文档规模小（<100K tokens）、对成本不敏感、需要全局理解的场景
- RAG 适合：文档规模大、需要高效检索、成本敏感、需要灵活检索策略的场景

### 5. 多模态内容转文本输出

**问题场景**：
- 用户只通过文本输入输出
- 但文档中可能包含图片、表格、公式等多模态内容
- 需要将这些内容转换为文本形式输出

**解决方案**：

1. **图片处理**：
   - **OCR 识别**：使用 OCR 工具（如 Tesseract、PaddleOCR）提取图片中的文字
   - **图像描述生成**：使用视觉-语言模型（如 GPT-4V、Claude 3）生成图片的文字描述
   - **图表理解**：对于图表，可以提取数据并转换为结构化文本描述
   - **输出格式**：
     ```
     [图片描述：CT 机操作界面示意图，显示了参数设置区域的布局...]
     ```

2. **表格处理**：
   - **表格识别**：使用表格识别工具（如 Table Transformer）提取表格结构
   - **转换为 Markdown**：将表格转换为 Markdown 格式的文本
   - **转换为自然语言描述**：用 LLM 将表格内容转换为自然语言描述
   - **输出格式**：
     ```
     参数设置如下：
     | 参数名 | 默认值 | 范围 |
     | --- | --- | --- |
     | 参数X | 100 | 50-200 |
     ```

3. **公式处理**：
   - **LaTeX 识别**：如果文档中有 LaTeX 格式的公式，直接保留 LaTeX 代码
   - **公式转文字**：使用 LLM 将数学公式转换为自然语言描述
   - **输出格式**：
     ```
     计算公式为：E = mc²（能量等于质量乘以光速的平方）
     ```

4. **流程图/示意图处理**：
   - **结构识别**：识别流程图中的节点和连接关系
   - **文字描述**：用 LLM 生成流程的文字描述
   - **输出格式**：
     ```
     操作流程如下：
     1. 首先进行系统初始化
     2. 然后设置参数X
     3. 最后启动扫描
     ```

**实现流程**：

```python
def process_multimodal_content(document):
    """处理多模态文档内容"""
    result = []
    
    # 1. 提取文本内容
    text_content = extract_text(document)
    result.append(text_content)
    
    # 2. 处理图片
    images = extract_images(document)
    for img in images:
        # OCR 提取文字
        ocr_text = ocr_extract(img)
        if ocr_text:
            result.append(f"[图片文字：{ocr_text}]")
        else:
            # 生成图片描述
            description = vision_model.describe(img)
            result.append(f"[图片描述：{description}]")
    
    # 3. 处理表格
    tables = extract_tables(document)
    for table in tables:
        # 转换为 Markdown
        markdown_table = table_to_markdown(table)
        result.append(f"\n表格内容：\n{markdown_table}")
    
    # 4. 处理公式
    formulas = extract_formulas(document)
    for formula in formulas:
        # 转换为文字描述
        description = formula_to_text(formula)
        result.append(f"[公式：{description}]")
    
    # 5. 整合所有内容
    final_text = "\n".join(result)
    return final_text
```

**在 RAG 中的应用**：
- **索引阶段**：将多模态内容转换为文本后，再进行 embedding 和索引
- **检索阶段**：用户查询时，检索到的 chunk 可能包含多模态内容的文本描述
- **生成阶段**：模型基于文本描述生成答案，可以引用图片、表格等信息

**注意事项**：
- 信息损失：多模态转文本会有信息损失，特别是复杂的图表
- 描述准确性：图片描述的质量取决于视觉模型的能力
- 成本：使用视觉模型会增加处理成本
- 权衡：需要在信息完整性和成本之间权衡

---

### 总结

RAG 系统在实际使用中面临的主要挑战是检索准确性和上下文完整性。通过合理的分块策略（语义分块、滑动窗口、层次化）、多路召回、上下文压缩等技术可以有效解决。对于大规模文档，RAG 比大窗口模型更经济高效。多模态内容可以通过 OCR、图像描述、表格识别等技术转换为文本，但需要注意信息损失和成本权衡。
