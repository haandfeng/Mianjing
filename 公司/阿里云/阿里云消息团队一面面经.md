# 面经记录

**岗位方向：** 阿里云消息商业化团队（RocketMQ 开源维护 + 云上消息服务研发，技术栈 Java）

## 1. 团队介绍

面试官先介绍了团队背景：

- **核心业务：** 阿里云上的消息商业化团队，面向外部客户提供消息服务。将 Kafka、RocketMQ、RabbitMQ 等开源项目做托管，提供免运维、自动扩容等云原生能力。
- **商业化特色：** 做开源没有的能力，如 EventHub（消息内容可结构化、可查询，类似 SQL 查询消息字段）。
- **AI 方向：** AI 场景下的异步处理支持（大消息体、会话保持 / sticky session）、利用 AI 做诊断和提效。
- **开源维护：** 团队是 RocketMQ 开源项目的维护者（最早由团队贡献出去）。

## 2. 西门子实习 — RAG 平台

### Q：异步任务队列提升两倍嵌入效率，出于什么目的？

**我的回答：** 嵌入过大语言模型，文档多时一次嵌入要一两个小时。原来的 ETL pipeline 基于智源开源的 RAG 系统，后来不维护了需要重构。原版是同步串行的，没有异步和批处理。改为批量发送文档给服务器做 embedding，异步将返回数据落库到 Milvus，通过批处理 + 异步方式提升效率。

### Q：是增量文档还是全量文档做处理？

**我的回答：** 都有。不同机型有不同的数据表，重新构建数据表时做全量嵌入；特定机型有新文档（如政策更新）时做增量嵌入。使用场景更偏向一次性批量嵌入，不会高频新增。

### Q：如果是持续高频涌入大量文档（如钉钉内部知识库场景），要求近实时、不丢不漏，架构上怎么改进？

**当时的回答：** 不丢靠持久化和事务 commit；快靠更多 embedding 服务器做负载均衡 + 协程异步落库；整条流水线每步都要能动态扩容。

**改进后的回答：**

核心思路：用消息队列把 ETL 的每个阶段解耦成独立的消费者组，每个阶段可以独立扩容。

**整体架构：**

```
文档变更 → Queue 1 (待解析) → 解析 workers → Queue 2 (待嵌入) → Embedding workers → Queue 3 (待落库) → 落库 workers → Milvus
```

三个 Queue 把三个阶段解耦，每个 Queue 后面挂多个 worker（competing consumers），哪一步慢就加哪一步的 worker 数量。文档之间互相独立、不需要严格顺序，天然适合 Work Queue 模式——谁空闲谁拿下一条，自动负载均衡。

**Queue 1 → 解析 workers（CPU 密集）：** 文档被创建或更新时，往 Queue 1 发一条轻量消息 `{doc_id, action, timestamp}`，不放文档本身。解析 worker 拿到 doc_id 后自己去 S3 拉文档 → 转 Markdown（处理嵌套表格等复杂结构）→ 切片（chunking）→ 把 chunks 发到 Queue 2，处理完手动 ACK。一台机器跑 2-3 个 worker，扛不住就加机器加 worker。

**Queue 2 → Embedding workers（最大瓶颈）：** worker 用 `prefetch_count` 一次拿一小批（如 32 条），攒够一批后一次性发给 embedding 服务器做批量推理（GPU 批处理比逐条快得多）。同时设超时（如 200ms 攒不够也发出去），保证时效性。拿到向量后发到 Queue 3，逐条 ACK。多部署几个 embedding 服务器实例，worker 用异步请求打满它们。

**Queue 3 → 落库 workers：** 从 Queue 3 拿到向量，批量写入 Milvus。delete 操作按 doc_id 删除所有 chunks；update 操作先删后插。手动 ACK。

**不丢不漏保证：** 生产者开启 confirm 机制，发送失败重试。Queue 和消息都设 `durable=true`，Broker 重启不丢。消费者 `autoAck=false`，手动 ACK，worker 挂了消息自动回到队列。幂等用 doc_id + chunk_index 做唯一标识，重复消费时先删旧再插新，不产生重复向量。多次失败的消息进死信队列，兜底服务处理或人工介入。

**近实时保证：** 监控每个 Queue 的消息积压量。正常情况下积压接近 0。如果 Queue 2 积压持续增长，说明 embedding 消费能力不足，加 worker 或加 embedding 服务器。端到端延迟目标：文档变更到向量可查，秒级到十几秒级。

**面试官补充：** 不仅是 embedding 服务器需要多机器，从 S3 读文档和转 Markdown 这一步也是入口瓶颈，同样需要多机器扩容。

### Q：自己部署大语言模型的原因？用的什么模型？

**我的回答：** 安全性考虑，数据是医院数据，需要本地部署。模型用过 LLaMA 70B、GPT 的 20B 和 120B 版本，也测试了千问 36B。

## 3. 用友实习 — 数据迁移

### Q：介绍一下数据迁移的场景？

**我的回答：** 帮中国电信国际升级人力系统，旧系统和新系统都是 PostgreSQL，但表结构 Schema 不同（大版本升级不兼容），需要将旧系统数据迁移到新系统。是一次性的离线迁移。

### Q：gRPC 在这里的角色是什么？

**我的回答：** Python 作为中间层：从旧系统通过 gRPC 接口读数据（旧系统提供的查询接口），Python 做数据处理（如手机号格式转换），再通过新系统接口写入（新系统会做数据校验）。源端和目的端都是调接口而非直连数据库。

### Q：遇到了什么问题导致错误？

**我的回答：** 主要是数据格式问题。最麻烦的是手机号格式——多国员工填写格式不统一（如 +86、(86)、+86- 等），需要用正则表达式统一格式。新系统会做格式校验，不通过就返回错误。需要修正正则后重新同步，而每次同步耗时很长。

### Q：如果要求不停机迁移，你怎么解？

**当时的回答：** 通过 `updated_at` 时间戳做增量同步的标记（比自增 ID 好，能捕获 update 操作），删除场景引入逻辑删除标志，延迟小于阈值后切流到新系统。

**改进后的回答 — 方案一：双写 + 灰度切流（业界主流，十亿级数据迁移标准方案）**

分五个阶段：

**阶段一：全量数据搬迁（分块 + 禁用索引）。**

十亿行数据不能一次性导出——数据库会占用巨大缓冲区，可能溢出到磁盘甚至崩溃。正确做法：

- **按主键范围分块：** 将表拆成多个区间（如 ID 1–5M、5M–10M），每次只处理一个块，内存占用可控。
- **加载期间禁用二级索引和外键约束：** 正常插入时每行都要更新索引（B+ 树分裂）和校验外键（跨表查询），十亿行就是十亿次。禁用后插入变成纯顺序追加写，速度快得多。数据全部灌完后一次性重建索引（批量建索引可以排序后顺序写入，比逐行更新高效得多）。
- **并行多 worker：** 各块主键范围不重叠、互不依赖，多个 worker 同时跑不同块。
- **每块跑校验和（checksum）：** 分块 + 并行 + 禁用约束都是在"放松安全检查换速度"，必须事后验证。源库和目标库对同一块做 checksum 比对，一致才算搬对。

新库此时不承担任何业务流量，可以慢慢跑。同时用 DTS 或主从复制开启增量同步链路，持续追平全量搬迁期间旧库新产生的数据。

**阶段二：开启双写，接住实时流量。**

全量搬迁期间每秒仍有大量新写入涌入旧库，同步链路追平后，需要切换到双写模式：

- **打开双写开关**。
- **先写旧库（主），成功后再写新库（副）。** 旧库 WAL 落盘成功，数据就安全了。新库写失败不影响主链路——失败事件推到 Kafka 重试队列，消费者反复重试直到成功。旧库 WAL 兜底 = 低成本分布式事务（最终一致），不需要跨库两阶段提交。
- **幂等 ID 保证不重复：** 共存期间同一条数据可能被同步链路写一次、双写又写一次；写新库超时但其实成功了，重试又写一次；Kafka 消费者重启重复消费。给每次写入打唯一 ID，用 `INSERT ... ON CONFLICT (unique_id) DO NOTHING` 保证写多次和写一次效果一样。
- 双写开关可配置，随时关闭回退到只写旧库。

**阶段三：影子读（Shadow Read），线上暗测新库。**

双写跑稳后，新库数据和旧库同步了，但**能不能信新库的查询结果**？测试环境永远覆盖不了真实流量的复杂度。

- 客户仍然从旧库读取数据（感知不到任何变化）。
- 后台每个查询默默地对新库重复执行一遍，比对两边结果。
- 典型会暴露的问题：时区处理差异（`TIMESTAMP WITHOUT TZ` vs `WITH TZ`）、NULL 值变成默认值、排序规则不同（UTF-8 vs Latin1）导致排序结果不一致。

影子读给了几周时间修这些坑，客户毫无感知。这是在客户发现之前捕获查询差异的唯一方法。

**阶段四：正式切换（切读流量到新库）。**

影子读跑稳、差异清零后，选业务低峰期（如凌晨）切流：

- **切换前预热新库：** 新库缓冲池（Buffer Pool）是冷的，直接切流会导致大量磁盘读、延迟飙升。提前跑合成查询，把热点数据和索引加载到缓存。
- **灰度切读：** 通过 Feature Flag 逐步切（1% → 5% → 20% → 100%），每个阶段观察延迟、错误率、业务指标（如每分钟订单量）。
- **保持双写不关：** 切换后继续双写跑一周以上，出问题随时关开关回滚到旧库。
- **回滚预案：** 双写阶段 → 关双写开关；灰度切读阶段 → 读流量切回旧库；全量切换后 → 旧库数据靠双写一直是最新的，可直接切回。

**阶段五：可观测性（贯穿全程的生命线）。**

整个迁移过程中，监控比代码更重要。核心监控指标：

- **同步/复制延迟：** 新库比旧库慢多少秒。
- **影子读不匹配计数：** 新旧库查询结果差异数。
- **新库死锁和慢查询：** 新库的执行计划可能因统计信息不准而选错。
- **缓存命中率：** 切换后必须保持在 95% 以上，否则磁盘 IO 会爆。
- **业务 KPI：** 每分钟订单量、收入流水——技术指标全绿不代表业务没问题。

没有这些仪表盘，迁移就是盲人摸象。迁移本质上是伪装成数据问题的监控问题。

**改进后的回答 — 方案二：CDC 监听日志做增量同步**

PG 没有 binlog，但有 WAL（Write-Ahead Log）。通过 Debezium 监听 PG 的逻辑解码（logical decoding），将每条 INSERT / UPDATE / DELETE 变更捕获出来，发到消息队列，消费端按顺序写入新库。

**保证同一用户操作有序的关键：** 积分/余额这类场景，同一用户的操作顺序不能乱（先扣 30 再加 50，反过来就错了）。需要保证同一用户的变更消息被同一个消费者按 FIFO 顺序处理。

如果用 Kafka：用用户 ID 做 partition key，同一用户的消息自动路由到同一个 partition，partition 内天然有序。

如果用 RabbitMQ：RabbitMQ 没有 partition 概念，但可以用 Consistent Hash Exchange 实现等效效果——建多个功能相同的 Queue（如 queue_0、queue_1、queue_2），Exchange 根据 routing key（设为 userId）的 hash 值自动分配到对应 Queue。每个 Queue 绑定一个消费者，同一用户的消息一定在同一个 Queue 里，被同一个消费者按 FIFO 处理。不同用户在不同 Queue 里并行消费，不影响吞吐量。

流程：全量快照导入新库 → 开启 CDC 增量追平 → 监控消费延迟降到秒级 → 短暂拦截写请求（几秒）→ 确认完全追平后切流 → 旧库保留不下线，出问题随时切回。



- **幂等性：** 每个写入操作都要可重复执行，因为重试一定会发生。
- **分阶段而非大爆炸：** 大爆炸快但无法回滚，分阶段慢但可逆——选可逆。
- **先加载后建索引：** 批量导入期间禁用索引和约束，导入完成后一次性重建。
- **影子读验证：** 测试环境永远抓不到真实流量暴露的问题（时区、编码、排序规则差异），只有线上暗测才行。
- **预热再切换：** 新库缓冲池是冷的，切换前必须跑合成查询预热缓存和索引。
- **监控 > 代码：** 监控同步延迟、影子读差异、缓存命中率、死锁、业务 KPI，迁移本质上是监控问题。
- **全程可回滚：** 任何阶段出问题都要能退回上一步，自信来自于知道可以撤销。

### Q（延伸）：电商场景下积分/点券的增量同步，对数据准确性和顺序要求极高，怎么做数据校验/对账？

**当时的回答：** 初步想法是把旧系统和新系统的操作包在一个事务里，任何一边出错就回滚，保证强一致。但不确定实际场景中性能是否可行。

**改进后的回答：**

面试官提到"这不一定是纯技术问题"，核心思路是**同步系统和对账系统分离**，对账从业务维度校验。

**对账系统设计（独立于同步系统）：**

**抽样数据对账：** 全量校验几亿条数据不现实。按主键取模（如 `ID % 100 = 0`）或按时间范围抽样，定时从新旧两库各捞一批数据，比对关键字段。不一致的写差异表 + 告警。

**业务维度对账：** 积分场景不能只比对余额快照——两边余额恰好相等不代表中间流水没错。正确做法是用流水表：把某用户某段时间内所有流水的净变化额加总，和余额的实际变化做比对。这能发现"流水写了但余额没变"或"顺序错乱导致中间状态异常"的问题。

**积分幂等的特殊处理：** 积分的加减操作不能直接用 UPSERT 覆盖余额（覆盖会丢中间变化）。改为用流水表记录每一笔变更，以流水的唯一 ID 做幂等。即使重复写入，同一笔流水只生效一次。

**对账节奏：** 只有对账系统持续跑一段时间（几天到一周）、差异率稳定为零，才进入下一阶段切流。切完流后继续对账，确认稳定后才下线旧库。

**面试官补充：**
- 金融场景会专门搭建一套对账系统，独立于同步系统，去校验金额变化。
- PG 没有 binlog（MySQL 有），但有其他方式做 CDC（如 WAL 逻辑解码）。
- 增量同步、切流、校验是一个持续的过程，不是瞬间完成的。

### Q（进一步延伸）：支付环节涉及对账/对接银行，如何保证系统与外部系统的数据一致性？

**问题本质：** 我们的系统和银行是两个独立数据库，没法用一个事务同时操作两边。扣了用户余额、调银行接口转账，银行可能超时——不知道钱到底转没转。这是分布式一致性问题。

**改进后的回答：** 实际系统中不是只选一种方案，而是分层组合：

**1. 本地消息表（核心保障）：** 把"扣余额"和"记一条待支付消息"放在同一个本地事务里，保证要么都成功要么都失败。然后异步调银行接口完成支付。银行返回成功 → 更新消息状态为"已完成"。如果调银行超时或失败，定时任务扫描消息表里"待支付"的记录，调银行查询接口确认结果再更新状态。关键：用唯一支付流水号保证幂等，重试多少次都不会重复扣款。

**2. 独立对账系统（兜底防线）：** 不管上面做得多好都可能有边界情况。每天（或每小时）从银行拉账单文件，和系统支付流水逐条比对（支付流水号、金额、时间）。三种差异：
- **我们有银行没有：** 我们记录成功但银行没这笔，可能状态更新错了，需退款或重新发起。
- **银行有我们没有：** 银行扣款成功但我们没收到回调（网络丢了），需补支付成功记录、更新订单状态。
- **金额不一致：** 最严重，必须人工介入。

差异记录写入差异表，小额自动补偿，大额告警人工处理。

**3. TCC（多方资源协调场景）：** 同时涉及余额、优惠券、积分时使用。三阶段：Try 冻结资源（余额标记冻结不真扣）→ Confirm 所有方都冻结成功后一起确认 → Cancel 任何一方 Try 失败则全部释放。避免"钱扣了但优惠券没扣"的半成功状态，但实现复杂（每个参与方都要实现三个接口）。

**4. 最大努力通知（银行侧回调）：** 银行支付完成后主动回调通知我们，未响应则按策略重试（1s、5s、30s、5min）。我们收到通知后更新订单状态并返回成功。最简单但不够可靠，必须配合对账兜底。

**一句话总结：** 实时靠本地消息表 + 幂等重试，回调靠最大努力通知，多方操作靠 TCC，兜底靠独立对账系统。核心原则是"宁可多查一次也不能漏一笔"。

## 4. 仿大众点评 — 秒杀方案

### Q：为什么用 RabbitMQ 实现秒杀？耗时从 400ms 降到 100ms 的原因？

**我的回答：** 用哪个消息队列区别不大（QPS 不高），主要实现异步削峰。原来扣减库存（Redis）和订单落库（数据库）在同一个同步函数里。改为：Redis 扣库存后，将订单落库操作发到 RabbitMQ 异步处理，提升整体响应速度。

### Q：扣库存成功但发消息失败，或消费失败怎么处理？

**当时的回答：** 通过订单 ID 做幂等去重；生产者端开启 confirm ACK 机制，发送失败重试；消费者端手动 ACK，成功消费才确认，否则消息留在队列。

**改进后的回答（从生产者、消息队列、消费者三个环节保证不重、不漏、一定成功消费）：**

**一、生产者端 → 消息队列：保证消息一定发出去**

**Confirm 机制（Publisher Confirm）：** 生产者开启 confirm 模式，每条消息发出去后，RabbitMQ 收到并持久化后回一个 ACK 给生产者。收到 ACK 才算发送成功，收到 NACK 或超时没收到响应就重发。

边界情况：消息已到达 RabbitMQ 并持久化，但 ACK 回来时网络断了，生产者以为失败又发了一次。所以 confirm 保证的是"至少一次投递"（at least once），重复的问题交给消费者端幂等解决。

**mandatory 参数：** 消息发到 Exchange 但路由不到任何 Queue（如 routing key 配错），消息会被默默丢弃。开启 `mandatory` 后，路由失败时 RabbitMQ 把消息退回给生产者（触发 ReturnCallback），生产者可记录日志或走补偿。

**二、消息队列本身：保证 Broker 重启不丢消息**

三个层面都要做持久化：

- **Exchange 持久化：** 声明时设 `durable=true`，Broker 重启后 Exchange 还在。
- **Queue 持久化：** 声明时设 `durable=true`，Broker 重启后 Queue 还在。
- **消息持久化：** 发送时设 `deliveryMode=2`，消息写入磁盘。不设的话消息只在内存里，Broker 一重启就没了。

三个都设了消息才真正安全。如果对可靠性要求更高，可以用 Quorum Queue（消息在多节点有副本，单节点挂了也不丢）。

**三、消费者端：保证消息一定被成功处理**

**手动 ACK（Manual Acknowledge）：** 关闭自动 ACK（`autoAck=false`）。消费者收到消息后先处理业务逻辑（落库），落库成功后再手动调 `basicAck`。如果消费者处理到一半挂了没发 ACK，RabbitMQ 检测到连接断开，把消息重新放回队列投递给其他消费者。

同样会导致重复消费——落库成功但 ACK 没发出去消费者就挂了，重启后这条消息又被消费一次。所以消费者业务逻辑必须幂等：用订单 ID 做唯一键，`INSERT ... ON CONFLICT DO NOTHING`，重复消费不产生重复订单。

**Reject 的区分：** `basicNack` 或 `basicReject` 可以把消息重新入队（`requeue=true`）或丢弃。数据库临时连不上 → `requeue=true` 重试；数据格式错误等不可恢复的失败 → `requeue=false` 让消息进死信队列，不要无限重试。

**全链路串起来：** 生产者 Confirm + mandatory 保证消息一定到达队列（至少一次）→ 队列三层持久化（Exchange + Queue + 消息 durable）保证 Broker 重启不丢 → 消费者手动 ACK 保证消息一定被成功处理（至少一次）→ 全链路都是"至少一次"所以一定会有重复，靠消费者端幂等（订单 ID 唯一键）保证重复消费无副作用。

**一句话总结：** RabbitMQ 的可靠性设计核心是 **at least once + 幂等 = effectively once**。

### Q：如果发送消息多次都失败了怎么办？

**当时的回答：** 没有考虑过发送失败的场景。临场想到的思路：人工处理（发邮件告警）；或者回退到同步下单 + 限流，不走消息队列。

**改进后的回答：** 标准做法是**本地消息表**。不直接发消息队列，而是在 Redis 扣库存的同时，把订单信息写到本地数据库的一张消息表里（同一个事务，要么都成功要么都失败）。然后异步地从消息表里捞记录发到 RabbitMQ。

发送成功 → 更新消息表状态为"已发送"。发送失败 → 状态还是"待发送"，定时任务每隔几秒扫一遍，捞出"待发送"的记录重新发。这样即使 RabbitMQ 整个挂了，订单数据也不会丢——安全地在本地数据库里，等 RabbitMQ 恢复后继续发。当时说的"回退到同步下单"方向是对的，本地消息表本质就是这个思路——先落本地再异步投递，只是不需要真的改回同步模式。

### Q：发送成功但消费多次失败怎么处理？

**当时的回答：** 消息积压在队列中。RabbitMQ 支持多次消费失败后自动转入死信队列，由专门的兜底服务从死信队列读取并落库。

**改进后的回答：** 死信队列的思路是对的，补充完整处理链路：

消费失败后 RabbitMQ 自动重试（配重试次数，如 3 次）。3 次都失败 → 消息自动转入死信队列。死信队列有专门的兜底消费者，做两件事：把失败消息写入数据库的失败记录表（持久化，防止死信队列也出问题），同时发告警通知。人工排查修复后从失败记录表重新触发落库。

补充一个细节：要区分"可重试的失败"和"不可重试的失败"。数据库临时连不上 → 重试几次可能就好了；数据格式错误 → 重试一万次也不会成功，应该直接进死信队列，不浪费重试次数。

**两个问题的共同原则：** 都是靠"先持久化到一个可靠的地方，再异步重试"兜底。发送失败持久化到本地消息表，消费失败持久化到死信队列 + 失败记录表。核心是：任何一步失败都不能导致数据丢失，总有一个地方记着这笔订单还没处理完。

**面试官点评：** 死信队列确实是解决这种场景的标准做法——方便监控问题，也能二次拉取消费。
