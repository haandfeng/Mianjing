## 2. 用友实习 — 数据迁移方案

### Q：数据迁移中如何保证数据一致性、完整性？如果异常如何回滚？

**当时的回答（较粗糙）：** 通过 ID 实现幂等，增量同步，分批同步实现事务控制。

**改进后的回答：**

按"迁移前 → 迁移中 → 迁移后"三阶段组织：

**迁移前：** 对目标库做快照备份（pg_dump），定义好 checkpoint 机制和幂等写入策略（upsert）。

**迁移中：**
- **写入层（一致性）：** 每个批次在一个数据库事务内完成，要么整批 commit，要么整批 rollback，不会出现半批脏数据。
- **传输层（一致性，不需要）：** gRPC 基于 HTTP/2 有内置校验，关键数据上额外做 checksum / 记录数对比。
- **断点续传（完整性）：** 每批 commit 成功后，将 checkpoint（最后一条记录的 ID 或 updated_at）和数据写入放在同一个事务里持久化。失败时事务回滚，checkpoint 不推进，下次从该批次重试。配合 upsert 天然幂等。

**迁移后：** 全量数据校验——count 对比 + 关键字段 hash 聚合对比（按表按主键区间做 MD5），确认源和目标完全一致。

### Q：迁移是停机迁移还是不停机？

**当时的回答：** 停机迁移。说停机

**改进后的回答：** 不停机，分三阶段：

1. **全量同步：** 搬存量数据，同时记录同步开始的位点（如最大 updated_at）。
2. **增量追平：** 从位点开始，通过 `WHERE updated_at > checkpoint` 持续拉取变更，upsert 写入新系统。监控同步延迟（当前时间 - 最新同步记录的 updated_at），延迟降到秒级即追平。
3. **流量切换：** 短暂拦截写请求（几秒），等增量完全追平后切流量到新系统。旧系统保留不下线，出问题随时切回。

**追问：为什么不用自增 ID 做增量标记？** 自增 ID 只能捕获新插入的行，无法感知已有行的 update。用 updated_at 时间戳字段可以同时捕获 insert 和 update。删除场景用软删除（deleted_at 字段）配合。更重的方案是 CDC（监听 WAL/binlog），但人力系统数据量可控，时间戳方案足够。

### Q：分批次回滚时，每个批次是在一个事务内吗？具体怎么回滚？

**当时的回答：** 用类似 page index 记录批次，失败后重试。

**改进后的回答：**

**批次内部 — 数据库事务保证原子性。** 一批 500 条记录包在一个事务里，任何一条出错整个事务 rollback，不需要手动清理。

**批次之间 — checkpoint 实现断点续传。** checkpoint 更新和数据写入在同一个事务内，保证两者的原子性。不会出现"数据写了但 checkpoint 没更新"或反过来的情况。失败时 checkpoint 不推进，下次从该批次重试，配合 upsert 幂等。

```sql
checkpoint = load_checkpoint()
for batch in fetch_batches(start=checkpoint):
    BEGIN TRANSACTION
        upsert batch into target_db
        update checkpoint to batch.last_id
    COMMIT  # 成功：推进 | 失败：自动回滚，checkpoint 不变
```

### Q：如果遇到没考虑到的问题，需要完全回滚整个升级，怎么办？

**当时的回答：** 通过 ID 覆盖重刷一遍。

**改进后的回答：** 迁移前已对目标库做了快照备份，出现整体性问题直接恢复快照即可，干净彻底不引入新变量。覆盖式重刷本质上是又一次写操作，可能引入新风险，不如快照恢复可靠。

### Q：Redis 缓存了哪些内容？（补充准备）

招聘业务中缓存的是**读多写少、变更频率低**的数据：

- **组织架构与部门信息：** 部门层级、部门负责人。展示岗位、审批流转时频繁读取，但几乎不变。
- **岗位字典/职位模板：** 岗位名称、职级体系、薪资范围区间、工作地点列表。发布和筛选职位时高频读取，HR 可能几个月才改一次。
- **招聘流程配置：** 面试轮次设置（如"技术一面→二面→HR 面→总监面"）、审批链。配置后很少变，但每个候选人流转时都要读。
- **用户权限与角色信息：** 哪个 HR 能看哪些岗位的候选人、谁有审批权。每次请求都要做权限校验，但变更频率很低。

这些数据在秋招高峰期（大批量简历涌入）读 QPS 很高，每次都打 PostgreSQL 容易成为瓶颈。放到 Redis 后 90% 请求直接从缓存返回。

**缓存一致性策略：** Cache Aside 模式——写入时删除缓存，下次读取 miss 后从数据库加载并回填，同时设置 TTL 兜底。这些数据变更频率低，简单策略足够。

## 3. ARM 实习 — Profiling 工具开发

### Q：off-CPU profiling 的原理和背景是什么？

**我的回答：**
- 工具基于 perf 命令做采样，但 perf 采样不到线程 sleep 状态的数据。
- 要实现 off-CPU 功能，需要修改 Linux 内核和 perf 工具，加入线程 sleep 时间、wake up 时间、被调度进 CPU 的时间等记录。
- 本质目的是识别多线程程序的 critical path，IO wait 也是 critical path 的一部分，但目前无法对 sleep 状态的线程做虚拟加速来量化优化效果。

### Q：目前进展如何？

**我的回答：**
- 思路来源于 2015 年 SOSP best paper，AMD 已将类似方案应用到 LCM 中。2024 年也有作者提出了优化方案。
- 该作者已改好内核和 perf，但基于 Linux 5.x 内核，而我们需要迁移到 6.8 内核。
- 目前主要工作是将旧版本代码迁移到新内核，并将 COZ 项目（2015 年至今一直维护）的代码与 2024 年论文的代码合并。

## 4. AI 辅助开发的使用方式

### Q：你现在用 AI 一般怎么提效？

**我的回答：**
- 让 AI 写脚本、做 benchmark 测试。
- 让 AI 做代码 research，分析修改过的 Linux kernel 与原版的区别。
- 让 AI 生成迁移 plan（Markdown 格式），然后自己审阅、理解、修改，反复迭代。
- 迭代到清楚每一步改什么代码后，再让 AI 帮忙执行修改。

### Q：怎么判断 AI 生成的 C++ 代码性能是否足够好？

**当时的回答：** 依靠个人知识储备审阅代码，工作之余学习 CSAPP、链接器等补充基础。

**改进后的回答：**

**1. 代码审查 — 看有没有明显的性能反模式。** C++ 里 AI 经常犯的低级性能问题，肉眼就能看出来：
- **值传递 vs const 引用：** 对 string、vector 等大对象，值传递会完整拷贝一份，const 引用只传一个指针（8 字节），零拷贝。
- **循环内频繁分配内存：** 每次循环 new 一个 vector 再销毁，不如把 vector 提到循环外，用 clear() 复用已分配的内存。
- **该 move 的地方用了 copy：** move 只是把指针"偷"过来（几个赋值），copy 是真的逐个元素拷贝。对象不再需要时应该用 std::move。
- **该用 unique_ptr 的地方用了 shared_ptr：** shared_ptr 的引用计数是原子操作，比普通整数加减慢得多。对象只有一个拥有者时 unique_ptr 就够了，零额外开销。

本质都是同一件事：C++ 不像 Java 有 GC 兜底，每一次内存分配、拷贝、引用计数操作都是实打实的开销。

**2. Benchmark 实测 — 数据说话。** 审查只能发现明显问题，真正的性能差异必须跑出来。用 Google Benchmark 或计时脚本对比 AI 生成版本和手写优化版本的吞吐量和延迟。这也是在 ARM 实习的工作习惯，每次改完代码都跑 MySQL TPS benchmark 验证。

### 面试官点评：
面试官提到，对于不熟悉的语言，AI 生成的代码一般能跑但性能不一定好，和专家写的差距可能很大。

## 5. 西门子实习 — RAG 平台

### Q：两倍嵌入效率的提升是怎么实现的？

**我的回答：**
- 原来是串行的：一个文档发给服务器 → 等返回 → 再发下一个，同步阻塞效率低。
- 改为异步 + batch 方式：批量发送给服务器做 embedding，异步拿结果后分批落库。
- 大语言模型部署框架（如 Ollama）本身支持异步 API。

## 6. Java 相关 — 依赖包冲突

### Q：遇到 Java 包版本冲突怎么排查和解决？

**当时的回答：** 报错一般是某个函数或类不存在，查源码确认缺失，上网搜 issue，下载对应版本替换。

**改进后的回答：**

**排查：** 用 `mvn dependency:tree` 打印完整依赖树，看到每个依赖传递引入了哪些子依赖和版本。典型报错是 `NoSuchMethodError` 或 `ClassNotFoundException`，说明运行时加载的 jar 版本和编译时期望的不一致。用 `mvn dependency:tree -Dincludes=groupId:artifactId` 过滤出具体依赖的所有引入路径，定位冲突来源。

**解决（从简单到复杂）：**

1. **`<exclusion>` 排除传递依赖：** A 引入了 C 的 1.0，B 引入了 C 的 2.0，想用 2.0 就在 A 的依赖里 exclude 掉 C。最常见的做法。
2. **`<dependencyManagement>` 锁定版本：** 在父 POM 里统一声明某个依赖的版本号，所有子模块和传递依赖都被强制对齐。大型项目管理依赖的标准做法。
3. **Maven Shade Plugin relocate（极端情况）：** 两个版本 API 确实不兼容、谁都不能让步时，用 shade plugin 把其中一个版本的包名改掉（如 `com.google.guava` → `shaded.com.google.guava`），让两个版本共存，类加载器不会冲突。

### Q：如果调版本解决不了的复杂冲突怎么办？

**当时的回答：** 自己改源码，找两个 commit 的区别把代码加进去。

**改进后的回答：** 优先用上述第三种方案（shade relocate）让两个版本共存，而不是手动改源码。改源码意味着你 fork 了一个自维护版本，后续上游更新你都要手动合并，维护成本很高，不是工程化的做法。

## 7. 算法题 — DAG 拓扑排序

### 题目：
设计数据结构存储 DAG（有向无环图），实现遍历算法，要求所有父节点被遍历后才能遍历子节点，输出一种合法顺序。

### 我的思路：
- **数据结构：** 邻接表法，数组 + 链表存储父子关系。
- **算法：** 拓扑排序。从入度为 0 的节点开始遍历，遍历后将子节点入度减 1，入度变为 0 的节点加入队列继续处理。

### 结果：
思路正确，代码实现过程中遇到一些语法问题，调试后输出正确。
