## 4. 用友实习 — 系统扩展讨论

### Q：如果用户量从 8K 扩展到 8 万甚至 80 万，系统需要怎么改造？
[[阿里云消息团队一面面经#Q：如果要求不停机迁移，你怎么解？]]
**我的回答：**
- 原来是停机同步，大用户量需要做增量同步 + 双写策略 + 灰度切流。
- **增量同步**：先做离线全量同步，然后持续增量追平。
- **双写**：数据差异足够小时，新旧系统双写（包在一个事务里），保证数据一致。
- **灰度验证**：后台做灰度读测试，在新旧系统同时读数据做对比，验证新系统数据完整性。
- **逐步切流**：10% → 逐步提高 → 100%，完全确认没问题后才全量切换。
- **旧系统保留**：切完后旧系统保留几个月，出问题随时回滚。

### Q：Redis 缓存了什么？P95 延迟降低 40% 怎么做到的？

**我的回答：**
- 缓存的是读多写少的数据：部门信息、员工信息、权限信息等。
- 原来查询响应时间大概一两百毫秒，配置缓存后大部分请求直接从 Redis 返回。
[[快手一面面经#Q：Redis 缓存了哪些内容？（补充准备）]]

**建议的回答：**

缓存的是招聘流程中高频读、低频写的数据：部门信息、员工信息、权限配置。这些数据一天可能才改一次，但每次请求都要查。
策略是 Cache-Aside（旁路缓存）：先查 Redis，命中直接返回；没命中查 PG 再写回 Redis，设置 TTL过期。数据更新时先更新数据库再删缓存。
效果：缓存命中率约 90%，也就是 90% 的读请求不再打到 PG。峰值流量下 PG 减少了约 1K QPS 的负载，P95 延迟从大概150ms 降到 90ms 左右（下降约 40%）。主要收益就是把高频的数据库查询变成了 Redis 的内存读取，Redis 响应在亚毫秒级。

## 5. 仿大众点评项目

### Q：为什么选择 NACOS 做服务注册？
[[RPC框架整理后#5.6 Zookeeper vs etcd vs Nacos]]
**我的回答：**
- 主要因为用的 Spring Cloud Alibaba 全家桶，生态配套方便。
- 了解其他方案：Zookeeper、Eureka、etcd（K8S 用的 key-value 存储）。

**改进后的回答：**
- 用 Spring Cloud Alibaba 全家桶，Nacos 跟 Sentinel、Feign 天然集成。另外 Nacos 同时支持服务注册和配置中心，不用额外搭一套配置管理。

**追问准备 — 各注册中心对比：**

| | Nacos | Zookeeper | Eureka | etcd |
|---|---|---|---|---|
| CAP | AP/CP 可切换 | CP | AP | CP |
| 功能 | 服务注册 + 配置中心 | 分布式协调（服务注册需自己实现） | 仅服务注册 | KV 存储（K8S 用） |
| 健康检查 | 主动探测 + 客户端心跳 | 客户端心跳（Session） | 客户端心跳 | 租约（Lease） |
| 生态 | Spring Cloud Alibaba | Dubbo、Hadoop | Spring Cloud Netflix（已停止维护） | Kubernetes |

**AP vs CP 区别：**
- **AP（可用性优先）**：Nacos 默认、Eureka。节点挂了其他节点还能提供服务列表（可能不是最新的）。适合服务注册——宁可拿到稍旧的服务列表，也不要拿不到。
- **CP（一致性优先）**：Zookeeper、etcd。节点挂了选举期间不可用，但保证数据强一致。适合配置管理、分布式锁。
- **Nacos 的优势**：两种都支持，服务注册用 AP（临时实例），配置管理用 CP（持久化实例）。

**ZK "服务注册需自己实现"的含义：** ZK 本身是分布式协调服务（带 watch 机制的树形 KV 存储），不是专门的注册中心。用它做服务注册需要自己写代码：服务启动时创建临时节点，消费者读取子节点拿服务列表，通过 watch 监听上下线。Dubbo 用 ZK 做注册中心是 Dubbo 框架帮你封装了这些操作。而 Nacos/Eureka 开箱即用，有现成的 API 和管理控制台。

### Q：登录功能和 TOKEN 校验怎么实现的？

**我的回答：**
- 网关层做登录校验，用户 cookie 里带 TOKEN（session），存在 Redis 里。
- 校验成功后从 Redis 取出用户信息，通过网关传给下游服务。
- 下游服务通过拦截器把用户信息放到 ThreadLocal 里，方便业务代码使用。
- 请求结束后拦截器移除 ThreadLocal 数据，防止内存泄露。
- 没带 cookie 的请求重定向到登录页面。

**改进后的回答：**

完整流程：
```
请求 → 网关 Gateway Filter 1（所有请求：带了 TOKEN 就刷新 TTL）
     → 网关 Gateway Filter 2（需要登录的接口：校验 TOKEN，取用户信息放请求头）
     → 下游微服务 Interceptor preHandle（从请求头取用户信息放 ThreadLocal）
     → Controller 业务逻辑
     → Interceptor afterCompletion（清理 ThreadLocal）
```

- **网关层（Spring Cloud Gateway Filter）**：TOKEN 相关逻辑全在网关统一处理。Filter 1 对所有请求刷新 TOKEN TTL（用户一直活跃就不过期）；Filter 2 对需要登录的接口校验 TOKEN，从 Redis 取用户信息放到请求头传给下游。没带 TOKEN 的请求重定向登录页。
- **微服务层（Spring Boot Interceptor）**：下游微服务完全不感知 TOKEN 和 Redis，只知道请求头里有用户信息。拦截器做两件事：`preHandle` 从请求头取用户信息放 ThreadLocal，`afterCompletion` 清理 ThreadLocal 防止内存泄露。
- **好处**：TOKEN 逻辑集中在网关，新增微服务不需要重复写校验逻辑；Gateway Filter 和 Interceptor 职责清晰分离。

### 追问：拿到 TOKEN 是不是就能访问系统？

**我的回答：**
- 是的，如果拿到 session TOKEN 就能访问，因为它存在 cookie 里是可以获取到的。

**改进后的回答：**
- 是的，单纯的 session TOKEN 被盗确实可以冒充用户。生产环境需要多层防护：
  1. **HTTPS**：防止 TOKEN 在传输中被窃取
  2. **Cookie 安全标记**：`HttpOnly`（JS 无法读取，防 XSS）、`Secure`（只在 HTTPS 下发送）、`SameSite`（防 CSRF）
  3. **TOKEN 绑定信息**：在 TOKEN 中绑定 IP 或设备指纹，换了设备就失效
  4. **短 TTL + 刷新机制**：TOKEN 有效期短（如 30 分钟），活跃用户自动刷新，被盗的 TOKEN 窗口期也短

### Q：为什么用双重拦截器？（实际是网关双重 Filter）

**我的回答：**
- 第一重拦截器：刷新 TOKEN 有效期（所有请求都经过，不管是否需要登录）。
- 第二重拦截器：判断是否需要登录，不需要登录的页面直接放行，需要登录的做校验。

**改进后的回答：**
- 准确说是网关层的两个 Gateway Filter，不是微服务层的拦截器。
- **Filter 1（所有请求）**：只要带了 TOKEN 就刷新 TTL。这样用户浏览不需要登录的页面时 TOKEN 也不会过期。
- **Filter 2（需要登录的接口）**：校验 TOKEN，取用户信息放请求头。
- 如果只用一个 Filter，要么不登录的页面 TOKEN 不会被刷新（用户浏览过程中过期），要么不需要登录的页面也被拦截了。

### 补充：RabbitMQ 消息积压 100 万条怎么处理？

**推荐回答：**
首先排查消费者有没有故障（挂了、bug、下游超时），消费者有问题加再多实例也没用。

**处理方案：**
1. **增加消费者数量**：Work Queue 模式下一个 Queue 挂多个消费者并行消费，设置 `prefetchCount=1` 确保均匀分配。
2. **消费者内部优化**：批量处理（攒 100 条一次 batch insert），比逐条写快得多。
3. **紧急转移**：Queue 快撑满时，起临时消费者把消息分发到多个临时 Queue，每个 Queue 后面挂多个消费者并行处理。
4. **可丢弃的消息**（日志、统计类）：直接丢弃积压消息，从最新的开始消费。

**预防措施：**
- 监控积压量，超阈值立刻告警
- 消费者自动伸缩
- 不重要的消息设 TTL，过期自动丢弃
- 上游做限流控制生产速率

### Q：如果用户量到 1000 万，系统需要怎么改造？

**我的回答：**
- **Nginx**：反向代理 + 静态资源缓存 + 负载均衡 + 限流。
- **数据库**：主从分离（主库写、从库读）+ 分库分表（单表数据量不要过大）。
- **Redis**：也做主从架构。
- **服务实例**：微服务多实例部署，Nginx 做流量分发。

面试官总结：加缓存、水平扩容、限流。

**改进后的回答（按层 + 结合项目具体业务）：**

**1. 接入层：**
- Nginx 集群做负载均衡 + 静态资源缓存（图片、CSS/JS 直接由 Nginx 返回，不经过后端）
- 用户量更大时加 CDN，把静态资源分发到全国边缘节点，减少网络延迟

**2. 缓存层：**
- Redis 单机改 **Redis Cluster 分片**：数据按 key 哈希分散到多个节点，容量和 QPS 线性扩展。每个主节点配从节点做高可用。
- 加**本地缓存（Caffeine）做多级缓存**：极热数据（商铺信息）先查本地缓存（纳秒级，进程内内存读取），不命中再查 Redis（亚毫秒级，需要网络往返）。适合读多写少、短暂不一致可接受的场景。

**3. 数据库层：**
- **读写分离**：主库写、从库读，商铺详情/评论等读多写少场景，读请求打到从库。
- **分库分表**：订单表按用户 ID 分表（千万级用户订单量巨大，单表撑不住）。
- **搜索独立**：商铺搜索从 MySQL `LIKE` 迁移到 Elasticsearch，支持全文检索和分词。

**4. 服务层：**
- 微服务多实例部署，通过 Nacos 注册发现，Gateway 做负载均衡。
- Sentinel 限流保护秒杀接口，下游故障时熔断降级。

**5. 消息队列层：**
- RabbitMQ 集群化（Quorum Queue 保证高可用），或换 RocketMQ 提升吞吐。

### Q：Nginx 有哪些负载均衡策略？

**我的回答：**
- 轮询（Round Robin）
- 加权轮询（不同机器性能不同，权重不同）
- 随机
- 一致性哈希：哈希环上分布服务器节点，请求哈希后顺时针找最近节点。扩缩容只影响相邻节点，不需要全量重新哈希。
- 基于请求内容的路由转发

**改进后的回答：**
1. **轮询**：默认策略，依次分配
2. **加权轮询**：性能好的机器权重高，分更多请求
3. **IP Hash**：同一 IP 固定打到同一台机器，适合需要 session 粘滞的场景
4. **最少连接**：优先分给当前连接数最少的机器，适合请求处理时间差异大的场景
5. **URL Hash**：同一 URL 打到同一台机器，缓存命中率高
6. **一致性哈希**：哈希环分布节点，扩缩容只影响顺时针方向的下一个节点（加节点分担它的负载，移除节点它接管负载），其他节点完全不受影响。适合缓存场景

### Q：项目中遇到不理解的地方怎么解决？

**我的回答：**
- 主要难点是 Redis 与 MySQL 缓存一致性策略的选择。
- 通过看源码、搜网上博客/视频、让大语言模型举例说明不同策略为什么会出现不一致，辅助理解。

## 7. 算法题 — 丑数

### 题目：
求第 N 个丑数（质因数只包含 2、3、5 的正整数）。

### 我的思路：
- 三指针法：维护三个指针分别对应乘 2、乘 3、乘 5。
- 每次取三个候选值中的最小值作为下一个丑数，推进对应指针。
- 例：初始 [1]，1×2=2, 1×3=3, 1×5=5，最小是 2，放入数组，推进 2 的指针。
