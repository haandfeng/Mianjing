## 项目简介

卡码网盘是一个基于 **Spring Boot 3 + Spring Cloud Alibaba** 构建的分布式网盘系统。

项目设计围绕：
- 文件**上传 / 下载**（含分片上传、断点续传）
- **目录管理**
- 文件**搜索**
- 文件**分享**（链接 + 提取码）
- **回收站**（逻辑删除）
- **AI 文档问答**（pgvector + embedding 向量检索）

> 鸽了 3 年多终于发布。**进阶项目**——建议先做卡码笔记项目作为前置（见 [[卡码信息]]）。

## 表结构（来自官方 DDL）

SQL 文件：`/Users/ha/Downloads/卡码网盘sql.zip`（**未入库**，本地参考；包含 2 个文件 `DDL_network_disk.sql` 和 `DDL_network_disk_ai_pgvector.sql`）。

### MySQL 主库（8 张表）

| 表名 | 用途 |
|---|---|
| `user` | 用户信息（昵称 / 邮箱 / 密码哈希 / 已用空间 / 总空间 / 邀请码）|
| `file` | 物理文件信息（real_path / 文件大小 / 后缀 / identifier 唯一标识）|
| `user_file` | 用户文件视图（user_id × parent_id × real_file_id 三元组；支持文件夹）|
| `file_chunk` | 文件分片信息（**分片上传 + 断点续传**核心）|
| `share` | 分享记录（提取码、过期时间）|
| `share_file` | 分享内容关联表 |
| `user_search_history` | 用户搜索历史 |
| `error_log` | 错误日志 |

关键字段约定：
- `lock_version` —— 乐观锁版本号，所有业务表都有
- `deleted` —— 逻辑删除标记
- `gmt_create` / `gmt_modified` —— 标准审计时间
- `identifier` —— **同一文件物理去重**的关键（多个 user_file 可指向同一个 file.identifier）

### PostgreSQL AI 库（3 张表，用 pgvector）

| 表名 | 用途 |
|---|---|
| `ai_document_index` | 文档索引（chunk 数 / 解析器 / 解析结果元数据）|
| `ai_document_chunk_vector` | 切片向量（**embedding vector(768)** 字段）|
| `ai_document_result` | AI 处理结果（summary / tags JSON）|

向量维度 768——对应 `com.disk.ai.pgvector.dimension` 配置。

## 技术栈

| 层 | 技术 |
|---|---|
| 后端框架 | Spring Boot 3 + Spring Cloud Alibaba |
| 注册中心 / 配置 | 推测 Nacos（Alibaba 标配）|

| 文件存储 | 推测本地 / 对象存储（real_path 字段）|
| AI | LLM + embedding（向量 768 维）|

## 涉及的核心知识点（面试准备角度）

可能被深挖的考点（按 BU 分组）：

### 分布式 / 中间件
- Nacos 注册发现 + 配置热加载
- Sentinel 限流 / 熔断
- 分布式事务（Seata?）
- 跨服务文件上传一致性

### 文件分片上传
- `file_chunk` 表设计：identifier × chunkIndex 为主键？
- 客户端如何组合 chunk → 整文件
- 断点续传：服务端如何告知客户端已上传 chunk 列表
- 秒传：通过 file.identifier 哈希命中

### 文件物理去重
- `file` vs `user_file` 双表设计
- `real_file_id` 字段引用 file.id
- 删除策略：当某 file 被所有 user_file 取消引用时回收

### 分享系统
- 提取码生成 / 校验
- 分享链接过期处理
- 防爬：限频 / 验证码

### AI 文档问答
- 文档解析（PDF / Word / Markdown）→ chunk 切分
- Embedding 生成（哪个模型）
- pgvector 向量检索（IVFFlat / HNSW 索引）
- RAG: 召回 → rerank → LLM 生成

### 通用
- 乐观锁实现（lock_version 自增 + CAS）
- 逻辑删除 vs 物理删除
- 用户配额（use_space / total_space）控制
